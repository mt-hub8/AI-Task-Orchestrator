# V2.7.2 RAG Demo Data / Manual Ingestion Golden Path 面试文档

## 1. 版本摘要

V2.7.2 的版本名称是：

```text
V2.7.2 RAG Demo Data / Manual Ingestion Golden Path
```

中文可以表述为：

```text
RAG 演示数据与手工摄入黄金路径
```

这个版本的核心目标不是继续新增复杂后端能力，而是补齐一条稳定、可复现的 RAG Demo（检索增强生成演示）路径。

在 V2.7 中，项目已经完成了：

```text
POST /rag/answers
```

它可以把用户 query 转成 query embedding，然后调用 VectorStore 检索相关 chunk，构造 citations，最后调用 LLM 生成 answer。

在 V2.7.1 中，项目又新增了：

```text
/rag-demo.html
```

用户可以在浏览器中输入问题，并查看 answer、citations、retrieval metadata 和 generation metadata。

但是到 V2.7.1 为止，仍然存在一个实际演示问题：

```text
页面已经能打开，接口也已经能调用，但如果没有稳定的 demo 数据和可复现的数据摄入流程，演示时可能只能得到 no-context answer。
```

所以 V2.7.2 的任务是补一条完整的 manual golden path（手工黄金路径）：

```text
启动依赖
  -> 准备 demo source 文档
  -> 摄入 demo 文档
  -> 打开 /rag-demo.html
  -> 输入推荐问题
  -> 看到 answer
  -> 看到 citations
  -> 看到 retrieval metadata
  -> 看到 generation metadata
```

这个版本的重点是工程交付能力，而不是功能堆叠。

---

## 2. 为什么现在做这个版本

V2.7 和 V2.7.1 分别解决了两个问题：

```text
V2.7：后端 RAG Answer API 可用
V2.7.1：浏览器 Web Demo 可用
```

但一个可演示的 RAG 项目，仅有 API 和页面还不够。

原因是 RAG 系统天然依赖一条比较长的链路：

```text
文档数据
  -> 文档切分
  -> embedding 生成
  -> 写入向量库
  -> query embedding
  -> 向量检索
  -> citations 构造
  -> prompt 构造
  -> LLM 生成
  -> 页面展示
```

如果没有提前准备演示数据，页面虽然可以打开，但可能无法稳定展示 citations。

这会导致面试演示时出现以下问题：

- 不知道应该摄入哪份文档；
- 不知道应该问什么问题；
- 不知道如何验证 citations 是否正确；
- 不知道 Qdrant、Local Embedding Worker、Spring profile 应该如何配合启动；
- 出现 no-context answer 时不知道是正常无数据，还是链路故障；
- reviewer 很难从零复现项目核心能力。

所以 V2.7.2 选择在 Web UI 之后补一条演示黄金路径，是非常合理的版本顺序。

它解决的不是“代码功能有没有”，而是：

```text
项目能力能不能被稳定复现、稳定演示、稳定交接。
```

## 3. 本版本解决的问题

### 3.1 解决 Web UI 缺少稳定演示数据的问题

V2.7.1 已经提供了浏览器页面，但如果向量库里没有数据，RAG Answer API 会进入 no-context 场景。

no-context 是合理的系统行为，但不适合作为主展示路径。

V2.7.2 通过新增 demo source 文档，让项目具备一份专门服务于演示的稳定语料。

这份 demo source 文档的内容围绕项目本身展开，例如：

- AI Task Orchestrator 是什么；
- RAG Answer API 的核心流程；
- citations 的作用；
- Embedding Cache 的 cache key；
- Minimal Web UI 为什么不引入 React / Vue；
- Qdrant Benchmark 的意义。

这样演示时可以提出稳定问题，并且能够命中对应内容。

---

### 3.2 解决 RAG 手工演示步骤分散的问题

RAG demo 涉及多个组件：

- Spring Boot 后端；
- MySQL；
- Qdrant；
- Local Embedding Worker；
- EmbeddingProvider；
- VectorStore；
- RAG Answer API；
- Web UI；
- demo source 文档。

如果启动和验证步骤分散在多个文档里，演示时很容易漏掉某一步。

V2.7.2 通过 manual golden path 文档，把这些步骤串起来，形成一条清晰流程：

```text
先启动依赖
再启动 worker
再启动 Spring Boot
再摄入 demo 文档
再打开 Web UI
再输入推荐问题
最后观察 answer / citations / metadata
```

这降低了复现成本，也降低了后续维护成本。

---

### 3.3 解决“能跑”到“可复现”的差距

很多项目的问题不是功能完全没有，而是别人很难复现。

V2.7.2 解决的就是这个差距：

```text
能跑：作者自己知道怎么跑。
可复现：别人按照文档也能跑。
可交付：步骤、数据、预期结果、失败排查都清楚。
```

这个版本让项目从个人开发状态进一步向工程交付状态靠近。

## 4. 核心设计

### 4.1 Demo Source 文档设计

本版本新增一份 demo source 文档，推荐路径为：

```text
docs/demo/rag-demo-source.md
```

这份文档不是普通说明文档，而是一份“专门用于 RAG 演示的检索语料”。

它的内容设计有两个要求：

第一，内容要和项目强相关。

这样在演示时，用户问的问题可以围绕项目本身展开，而不是问一些随机知识。

第二，内容要适合被 chunk 检索。

也就是说，每个小节最好围绕一个清晰主题展开，避免多个主题混在一段里，影响检索命中和 citations 展示。

推荐主题包括：

```text
1. AI Task Orchestrator 是什么；
2. RAG Answer API 的核心流程；
3. citations 在 RAG 中的作用；
4. Embedding Cache 的设计与 cache key；
5. Minimal Web UI 的设计取舍；
6. Qdrant Benchmark / manual verification 的意义。
```

这样设计的好处是：

```text
demo 文档既能作为语料，又能作为项目能力说明；
推荐问题可以稳定命中文档内容；
citations 可以展示出项目自解释能力；
面试时可以围绕自己的项目进行 RAG 演示。
```

---

### 4.2 Manual Golden Path 文档设计

本版本新增手工黄金路径文档，推荐路径为：

```text
docs/manual/rag-demo-golden-path.md
```

这份文档的作用是告诉使用者如何完整复现 RAG demo。

它应该覆盖：

- 目标；
- 前置条件；
- 启动 Qdrant；
- 启动 Local Embedding Worker；
- 启动 Spring Boot；
- demo source 文档路径；
- 如何摄入 demo 文档；
- 如何打开 `/rag-demo.html`；
- 推荐问题；
- 预期结果；
- 常见失败排查。

它的关键不是“写得长”，而是要把演示链路中的不确定点写清楚。

尤其是这类问题：

```text
如果 Qdrant 没启动，会发生什么？
如果 worker 没启动，会发生什么？
如果 profile 用错，会发生什么？
如果 embedding dimension 不匹配，会发生什么？
如果页面能打开但 /rag/answers 请求失败，应该查哪里？
如果 citations 为空，是没有数据，还是检索失败？
```

这些内容可以显著提升项目可维护性。

---

### 4.3 推荐问题设计

本版本的 manual 文档中应该提供至少 5 个推荐问题。

推荐问题的设计原则是：

```text
问题必须能稳定命中 demo source 文档；
问题要覆盖不同主题；
问题要能展示 answer、citations 和 metadata；
问题不应该依赖外部知识；
问题不应该依赖模型自己编造。
```

示例问题：

```text
AI Task Orchestrator 的目标是什么？
RAG Answer API 的核心流程是什么？
citations 在 RAG 系统中有什么作用？
Embedding Cache 使用什么作为 cache key？
为什么 Minimal Web UI 没有引入 React 或 Vue？
Qdrant Benchmark 的目的是什么？
```

这些问题都围绕 demo source 文档设计，因此适合稳定演示。

---

### 4.4 预期结果设计

manual 文档中不只应该写操作步骤，还应该写“怎么看结果是否正确”。

预期现象应该包括：

```text
answer 区域能返回与 demo source 相关的回答；
citations 区域至少出现一个引用；
citation 中能看到 documentId、chunkId、score、contentSnippet；
retrieval metadata 中能看到 topK、retrievedCount 等检索信息；
generation metadata 中能看到生成阶段信息；
如果没有摄入数据，则可能返回 no-context answer。
```

这样使用者不只是机械执行命令，还能判断系统是否处于预期状态。

---

### 4.5 失败排查设计

RAG demo 的失败原因可能来自多个层次。

V2.7.2 要把常见问题写清楚，例如：

```text
Qdrant 未启动；
Local Embedding Worker 未启动；
Spring profile 使用错误；
embedding dimension 不匹配；
demo 数据没有成功摄入；
向量库 collection 为空；
页面可以打开，但 /rag/answers 请求失败；
citations 为空；
返回 no-context answer；
```

这部分体现的是工程排障意识。

一个成熟的工程项目不只是提供 happy path（成功路径），还要告诉别人失败时如何定位问题。

## 5. 关键取舍

### 5.1 为什么这个版本主要是文档和 demo 数据

V2.7.2 看起来主要是新增文档和 demo source，但它不是普通文档版本。

它解决的是 RAG 系统的可复现问题。

RAG 系统不像普通 CRUD 接口，只要数据库有一条记录就能测试。它依赖：

```text
文档内容
chunk 策略
embedding provider
embedding dimension
vector store
检索参数
LLM provider
Web UI
```

这些组件必须按正确顺序连接起来，才能稳定演示。

所以这个版本虽然以文档和 demo 数据为主，但它提升的是项目交付质量。

面试时可以这样表达：

```text
V2.7.2 不是单纯写文档，而是把一个多组件 RAG 系统的演示路径固化下来，让 reviewer 可以按照文档从零准备数据，并在浏览器中复现 answer、citations 和 metadata。
```

---

### 5.2 为什么不新增 seed API

本版本没有新增 demo seed API（一键导入接口）。

原因是 seed API 会引入新的后端设计问题：

- seed API 放在哪个 controller；
- 是否只允许 local profile 使用；
- 是否需要鉴权；
- 重复导入是否幂等；
- 如何避免污染真实数据；
- 如何清理旧 demo 数据；
- 如何处理 Qdrant collection 已存在；
- 如何处理 MySQL 和 Qdrant 的一致性；
- 是否需要新增 Flyway 表或状态记录。

这些问题已经超出 V2.7.2 的目标。

当前版本的目标是先让手工流程稳定，而不是过早产品化 seed 能力。

更合理的演进顺序是：

```text
先有 manual golden path
再抽象 opt-in seeder
最后考虑 Web UI ingestion
```

所以 seed API 可以留到后续独立版本，例如：

```text
V2.7.3 Demo Data Seeder / Opt-in Seed Command
```

---

### 5.3 为什么不做文件上传 UI

本版本没有新增文件上传 UI。

原因是文件上传属于 RAG ingestion（摄入）产品入口，而 V2.7.1 的页面属于 query（查询）产品入口。

这两条链路应该拆开演进：

```text
ingestion 链路：
上传文档 -> 解析 -> 切分 -> embedding -> 写入 vector store

query 链路：
输入问题 -> query embedding -> 检索 -> citations -> 生成回答
```

V2.7.2 只是为了让 query demo 有稳定数据，不是正式建设 ingestion UI。

如果把上传 UI 一起做进去，会导致版本边界变大：

- 要处理文件格式；
- 要处理上传接口；
- 要处理解析状态；
- 要处理 embedding 进度；
- 要处理失败重试；
- 要处理前端展示；
- 要处理数据清理。

这些都应该作为后续独立版本处理。

---

### 5.4 为什么不改默认自动测试依赖边界

本版本仍然坚持：

```powershell
.\mvnw.cmd test
```

默认测试不能依赖：

- Qdrant；
- Docker；
- Python worker；
- 外部网络；
- 真实模型下载；
- OpenAI API key。

原因是 manual golden path 本身就是 opt-in（可选手工执行）验证路径。

真实 Qdrant、真实 embedding worker、真实模型下载都属于较重依赖，不能污染默认测试。

这个边界非常重要。

它说明项目把测试分成两层：

```text
默认自动测试：低依赖、快速、稳定；
手工验证路径：真实组件、真实数据、真实链路。
```

这是成熟后端项目常见的测试分层思路。

## 6. 测试设计

### 6.1 默认自动测试

V2.7.2 完成后，仍然必须运行：

```powershell
.\mvnw.cmd test
```

并保证现有自动测试通过。

如果本版本只新增 docs/demo 和 docs/manual，可以不新增自动测试。

如果新增了轻量测试，也应该只做资源存在性测试，例如：

```text
确认 demo source 文档存在；
确认 manual golden path 文档存在；
确认文档路径没有拼写错误。
```

不能新增需要 Qdrant、Docker、worker 或网络的测试。

---

### 6.2 手工验证

本版本真正强调的是手工验证。

手工验证流程可以概括为：

```text
1. 启动 Qdrant；
2. 启动 Local Embedding Worker；
3. 启动 Spring Boot；
4. 按 manual 文档摄入 demo source；
5. 打开 /rag-demo.html；
6. 输入推荐问题；
7. 检查 answer；
8. 检查 citations；
9. 检查 retrieval metadata；
10. 检查 generation metadata。
```

这个验证方式和默认自动测试分离。

原因是：

```text
真实 RAG 链路依赖外部组件，不适合每次 mvnw test 都运行；
但它又必须有一条清晰的 opt-in 验证路径。
```

这就是 V2.7.2 的价值。

---

### 6.3 验收重点

验收时不应该只看文件是否存在，而应该看：

```text
demo source 是否适合被检索；
manual 是否能从零指导演示；
manual 是否引用了真实存在的命令、API、profile 和路径；
是否编造了不存在的 ingestion API；
是否说明了 no-context / citations 为空等失败情况；
是否保持默认测试低依赖；
是否没有越界修改业务代码。
```

尤其是“不编造不存在的 API”很重要。

因为工程文档一旦写了不存在的命令，会严重降低项目可信度。

## 7. 当前局限

V2.7.2 仍然有明确局限。

它不做：

```text
一键导入；
seed API；
文件上传 UI；

自动清理 demo 数据；
数据版本管理；

浏览器自动化测试；
streaming answer；
multi-turn conversation；
evaluation dashboard；

```

这些不是当前版本缺陷，而是刻意控制边界。

当前版本只解决一个问题：

```text
让 RAG Web Demo 有稳定、可复现的演示数据和手工操作路径。
```

## 10. 面试高强度追问与回答

### 10.1 这个版本是不是只是写文档？

回答：

```text
表面上它主要新增 demo 文档和 manual 文档，但它解决的是 RAG 系统的可复现和可交付问题。

RAG demo 不是单个接口能完成的，它依赖 Qdrant、Embedding Worker、Spring Boot profile、demo 文档、文档摄入、向量写入和 Web UI。如果没有一条 golden path，功能虽然存在，但别人很难稳定复现。

所以 V2.7.2 的价值不是普通文档，而是把一个多组件 RAG 系统的演示路径固化下来。
```

---

### 10.2 为什么不继续做新功能，而是先补 golden path？

回答：

```text
因为 V2.7 已经有 RAG Answer API，V2.7.1 已经有 Web UI。这个时候如果继续堆新功能，但没有稳定 demo 数据和复现路径，项目展示价值反而不稳定。

我认为工程项目不是只追求功能数量，还要追求可复现、可演示和可交接。V2.7.2 就是为了让已有能力能被稳定展示出来。
```

---

### 10.3 为什么不做一键导入 demo 数据？

回答：

```text
一键导入会引入新的后端能力和边界问题，比如 seed API 放在哪里、是否需要鉴权、是否幂等、重复导入如何处理、如何清理旧数据、是否污染真实数据、MySQL 和 Qdrant 如何保持一致。

当前版本的目标只是收敛手工演示路径，所以我先做 manual golden path。等手工流程稳定后，再把它抽象成 opt-in seeder 会更安全。
```

---

### 10.4 为什么不做文件上传 UI？

回答：

```text
文件上传 UI 属于 RAG ingestion 产品入口，而 V2.7.1 的 Web UI 是 query 产品入口。两个方向都重要，但不应该混在一个小版本里。

当前版本只是为了让 query demo 有稳定数据。如果把上传 UI 一起做，会涉及文件解析、切分、embedding 进度、失败重试、状态展示等问题，边界会明显扩大。
```

### 10.5 为什么 demo source 文档要写项目自身内容？

回答：

```text
因为这份文档的目标是服务项目演示。把内容写成项目自身说明，可以让推荐问题围绕项目能力展开，例如 RAG Answer API 流程、citations 作用、Embedding Cache 设计等。

这样演示时，answer、citations 和 metadata 都能围绕项目本身展开，更适合面试展示，也更容易稳定命中。
```

### 10.6 如果页面返回 no-context answer，怎么排查？

回答：

```text
我会先区分这是正常无数据还是链路故障。

第一步检查 demo 文档是否已经成功摄入。
第二步检查 Qdrant 是否启动，collection 是否存在数据。
第三步检查 Local Embedding Worker 是否启动，并确认 embedding dimension 和 VectorStore 配置一致。
第四步检查 Spring profile 是否正确。
第五步检查 query 是否和 demo source 内容相关。
第六步查看 retrieval metadata，如果 retrievedCount 是 0，说明问题在检索或数据侧；如果有 citations 但 answer 异常，再看 generation 侧。
```

### 10.7 为什么默认测试不跑真实 Qdrant 和 worker？

回答：

```text
因为真实 Qdrant、Python worker 和模型下载都属于重依赖。如果默认 `mvn test` 依赖这些组件，会导致测试慢、不稳定，也不适合 CI 或普通开发环境。

我的设计是把测试分层：默认自动测试保持低依赖和高稳定性；真实 Qdrant / worker / RAG demo 走 opt-in manual golden path。这样既保证日常开发稳定，也保留真实链路验证能力。
```

它体现的是复杂系统的交付意识、可复现意识和测试分层能力。

## 11. 后续演进路线

### 11.1 Demo Data Seeder / Opt-in Seed Command

在手工 golden path 稳定后，可以考虑做一键 seed 能力。

但建议必须是 opt-in（显式启用），不能污染默认启动流程。

可能设计为：

```text

local-only admin endpoint；
CommandLineRunner + profile 控制；
独立 CLI 脚本；
```

需要重点解决：

- 幂等；
- 重复导入；
- 数据清理；
- collection 命名；
- demo 数据与真实数据隔离；
- MySQL 与 Qdrant 一致性。

---

### 11.2 RAG Ingestion Web UI

后续可以做文档上传和摄入页面。

它可以支持：

```text

```

这会把 ingestion 链路也产品化。

---

### 11.3 RAG Evaluation Dashboard

后续可以做 RAG 评估面板。

指标可以包括：

- Recall@K（召回率@K）；
- Precision@K（精确率@K）；
- MRR（Mean Reciprocal Rank，平均倒数排名）；
- NDCG@K（归一化折损累计增益@K）；
- HitRate@K（命中率@K）；
- latency（延迟）；
- per-case result（逐样本结果）；
- summary metrics（汇总指标）。

这样可以从“能演示”进一步升级到“能评估质量”。
