# V2.4.1 Retrieval Metrics Completion 面试文档

## 0. 版本摘要

V2.4.1 在 V2.4 Retrieval Evaluation Harness 基础上补充了两个标准 retrieval ranking metrics：`Precision@K` 和 `NDCG@K`。

V2.4 已经支持：

- Recall@K；
- HitRate@K；
- MRR / RR@K；
- ContextPrecision@K。

V2.4.1 让 evaluation 从“能评估召回、首个相关结果排名和上下文纯度”，升级为“同时评估召回、精度、排序质量和 RAG context 质量”。

必须明确：本版本只评估 Retriever，不评估 Generator。

本版本不做：

- Faithfulness；
- Answer Relevance；
- Answer Correctness；
- Citation Accuracy；
- LLM-as-a-judge；
- evaluation result 持久化；
- retrieval_eval_case / run / result 表；
- 真实 Embedding；
- Vector DB；
- Rerank。

这版的定位是 Retrieval Metrics Completion，不是完整 RAG Evaluation Platform。

## 1. 本版本解决的问题

V2.4 已经让系统具备 Retrieval Evaluation Harness，能够通过 `POST /evaluations/retrieval` 接收 query-evidence cases，并计算：

- Recall@K：expected relevant chunks 有多少被找回；
- HitRate@K：TopK 是否至少命中一个 relevant chunk；
- MRR / RR@K：第一个 relevant chunk 是否排得靠前；
- ContextPrecision@K：实际返回 context 中 relevant chunk 的比例。

但这套指标还有两个缺口。

第一，缺少标准 `Precision@K`。`ContextPrecision@K` 更贴近 RAG context 质量，但它不是严格的标准 Precision。它使用实际返回数量作为分母，而标准 Precision@K 使用固定 K 作为分母。

第二，缺少 `NDCG@K`。MRR 只关心第一个 relevant chunk 的位置，不关心后续多个 relevant chunks 的整体排序质量。如果一个 query 有多个 expected chunks，NDCG 能更好衡量相关结果整体是否排在前面。

V2.4.1 解决：

- `Precision@K`：衡量 TopK 中 relevant chunk 的比例；
- `NDCG@K`：衡量多个 relevant chunks 的整体排序质量。

一句话表达：

```text
V2.4 让系统能评估 Retrieval；
V2.4.1 让 Retrieval Evaluation 指标体系更标准。
```

## 2. 为什么现在做 V2.4.1

V2.4.1 的时机很自然。

前面已经具备：

- V2.2 可以检索 TopK chunks；
- V2.3 可以基于检索结果做 RAG Answer；
- V2.4 已经有 evaluation API、metrics calculator 和 service 编排。

在这个时候补 `Precision@K` 和 `NDCG@K`，改动最小：不需要新增 API，不需要改 request schema，不需要新增数据库表，只需要扩展 response DTO、calculator 和 summary 聚合逻辑。

为什么不先做 Dataset Seed？

Dataset Seed 很重要，但指标体系应该先稳定。如果先做 seed dataset，之后再补 response 字段、metric calculator、测试和文档，会让后续 benchmark 格式反复调整。先补完整 retrieval metrics，后续 seed 可以直接面向完整指标体系设计。

为什么不后置？

后续 Real Embedding、Vector DB、Rerank 都需要更标准的指标对比。如果不补 NDCG，就很难评价 rerank 是否真的改善整体排序质量；如果不补 Precision@K，就缺少标准 retrieval precision 指标。

因此 V2.4.1 是 V2.5、V2.6、V2.8 之前的一次必要指标补齐。

## 3. 方案选择

### 方案 A：不补指标，后续再说

优点：

- 不改代码；
- 可以继续快速推进 V2.5 / V2.6。

缺点：

- evaluation 指标体系不完整；
- 面试中容易被追问 Precision@K / NDCG@K；
- 后续比较 embedding / rerank 时指标不够标准；
- 只能解释 Recall、MRR、ContextPrecision，缺少经典 ranking 指标。

结论：不推荐。

### 方案 B：只补 Precision@K

优点：

- 实现简单；
- 与 ContextPrecision@K 有相似基础；
- 测试成本低；
- 能补齐标准 precision 维度。

缺点：

- 仍然缺少整体排序质量指标；
- MRR 只看第一个 relevant，不看多个 relevant 的整体排序；
- 无法很好评价 rerank 对整体 ranking 的改善。

结论：不推荐作为 V2.4.1 最终方案。

### 方案 C：补 Precision@K + binary NDCG@K

优点：

- 兼容当前 `expectedChunkIds`；
- 不改 request schema；
- 不新增数据库表；
- 指标体系更标准；
- 可以同时评估精度和排序质量；
- 后续可以升级 graded relevance。

缺点：

- NDCG 只能区分 relevant / non-relevant；
- 不能表达强相关 / 弱相关；
- 对成熟 benchmark 来说还不够精细。

结论：这是当前推荐方案，也是 V2.4.1 实际采用的方案。

### 方案 D：一次性支持 graded relevance

方向示例：

```json
{
  "expectedChunks": [
    { "chunkId": 12, "relevance": 3 },
    { "chunkId": 13, "relevance": 2 }
  ]
}
```

优点：

- NDCG 更标准；
- 能区分强相关和弱相关；
- 更适合成熟 benchmark。

缺点：

- 需要修改 request schema；
- 需要兼容 `expectedChunkIds`；
- 需要重新设计 DTO；
- 当前没有真实 relevance grade 标注；
- 版本边界过大。

结论：暂时不选。等后续有稳定 benchmark 和标注规范后，再升级 graded relevance。

## 5. 指标设计

### 5.1 Recall@K

定义：

```text
Recall@K = TopK 中命中的 expected relevant chunks 数 / expected relevant chunks 总数
```

它衡量：应该找回的证据，有多少被找回。

Recall@K 更关注“找全”，适合评估 Retriever 是否遗漏关键证据。

### 5.2 Precision@K

定义：

```text
Precision@K = TopK 中 relevant chunk 数 / K
```

重点：Precision@K 的分母固定为 K。

示例：

```text
K = 5
实际返回 3 个 chunks
其中 2 个 relevant

Precision@5 = 2 / 5 = 0.4
```

Precision@K 衡量 TopK 检索结果中，有多少比例是 relevant。

这是标准 retrieval evaluation 指标。如果系统目标是返回 K 个候选结果，那么没有返回足够结果或返回噪声结果都会影响 Precision@K。

### 5.3 ContextPrecision@K

定义：

```text
ContextPrecision@K = TopK 中 relevant chunk 数 / actualReturnedCountWithinK
```

如果 `actualReturnedCountWithinK = 0`，则 `ContextPrecision@K = 0.0`。

示例：

```text
K = 5
实际返回 3 个 chunks
其中 2 个 relevant

ContextPrecision@5 = 2 / 3 = 0.6667
```

ContextPrecision@K 更贴近 RAG 场景，因为它衡量实际塞给 LLM 的 context 中有多少是真正相关内容。

### 5.4 Precision@K 与 ContextPrecision@K 的区别

两者公式不同：

```text
Precision@K = relevantInTopK / K
ContextPrecision@K = relevantInTopK / actualReturnedCountWithinK
```

区别：

- Precision@K 是标准检索指标；
- ContextPrecision@K 是 RAG context 质量指标；
- Precision@K 更严格；
- ContextPrecision@K 更关注实际传给 LLM 的上下文纯度；
- 返回结果不足 K 时，两者会不同。

举例：

```text
K = 5
实际返回 3 个 chunks
2 个 relevant

Precision@5 = 2 / 5 = 0.4
ContextPrecision@5 = 2 / 3 = 0.6667
```

这就是为什么两个指标都需要保留，不能互相替代。

### 5.5 HitRate@K

定义：

```text
TopK 中是否至少命中一个 relevant chunk
```

命中则 `1.0`，否则 `0.0`。

它衡量系统是否至少给 LLM 提供了一条有效证据。这个指标粗，但很直观。

### 5.6 MRR / RR@K

定义：

```text
第一个 relevant chunk 的排名倒数
```

示例：

```text
第一个 relevant chunk 在 rank 2
RR = 1 / 2 = 0.5
```

MRR 衡量第一个正确证据是否足够靠前。

局限是：MRR 只关注第一个 relevant result，不关注后续多个 relevant chunks 的整体排序。

### 5.7 NDCG@K

NDCG 是 Normalized Discounted Cumulative Gain，用于衡量整体排序质量。

当前版本使用 binary relevance：

- `chunkId` 在 `expectedChunkIds` 中：`relevance = 1`；
- 否则：`relevance = 0`。

DCG@K：

```text
DCG@K = sum(rel_i / log2(rank_i + 1))
```

其中 rank 从 1 开始。

IDCG@K 是理想排序下的 DCG。

对于 binary relevance：

```text
ideal relevant count = min(expectedRelevantCount, K)
```

理想情况下，relevant chunks 都排在最前面。

NDCG@K：

```text
NDCG@K = DCG@K / IDCG@K
```

如果 `IDCG@K = 0`，则 `NDCG@K = 0.0`。

NDCG@K 衡量相关 chunk 是否不仅被找回，而且排在更靠前的位置。它比 MRR 更全面，因为它考虑多个 relevant chunks 的整体排序，而不是只看第一个命中。

## 6. 为什么 NDCG 当前只做 binary relevance

当前 request 只有 `expectedChunkIds`，它只能表达：

- relevant；
- non-relevant。

它不能表达：

- 强相关；
- 中等相关；
- 弱相关。

因此第一版 NDCG 使用 binary relevance。

优点：

- 兼容当前 API；
- 不需要修改 request schema；
- 不需要新增 graded annotation；
- 可以先评估基础排序质量；
- 适合当前 request-driven evaluation baseline。

局限：

- 不能区分不同相关程度；
- 不能表达某个 chunk 比另一个 chunk 更重要；
- 不能覆盖更成熟 benchmark 的 graded relevance 场景。

后续如果 benchmark 支持 relevance grade，可以升级为 graded NDCG，例如把 expected evidence 从 `expectedChunkIds` 扩展为 `{chunkId, relevance}`。

## 7. 为什么不做 Faithfulness / Answer Relevance

Precision@K、NDCG@K 属于 Retrieval Evaluation。

Faithfulness、Answer Relevance 属于 Generation Evaluation。

Faithfulness 评估：

```text
answer 中的说法是否被 context 支撑
```

Answer Relevance 评估：

```text
answer 是否回答了 query
```

当前不做这些的原因：

- 当前仍然使用 `MockLlmClient`；
- 没有真实 generated answer；
- 没有 expectedAnswer；
- 没有 judge model；
- 没有 answer claim extraction；
- 没有 citation-level alignment；
- 没有 LLM-as-a-judge 设计；
- 现在做会变成形式化字段，不具备真实工程价值。

结论：Generation Evaluation 应该后置到真实 LLM / RAG Answer 更稳定 / Citation Evaluation 阶段。

## 8. 为什么不做 retrieval_eval_case 持久化

当前仍然是 request-driven evaluation。

本版本不新增：

- `retrieval_eval_case`；
- `retrieval_eval_run`；
- `retrieval_eval_result`；
- `rag_eval_case`；
- `rag_eval_result`。

原因：

- 当前还没有稳定 benchmark dataset；
- 还没有 evaluation dashboard；
- 还没有多次 run 对比需求；
- 还没有真实 embedding / rerank 对比；
- 当前目标只是补齐指标体系；
- 先保持版本边界小。

后续如果要做持久化，可以单独设计：

- `evaluation_dataset`；
- `retrieval_eval_case`；

这样比在 V2.4.1 里顺手加表更稳。

## 9. 核心流程

V2.4.1 没有改变 V2.4 的 API 和检索流程，只扩展 metrics。

流程：

```text
POST /evaluations/retrieval
-> 标准化 topKValues
-> 对每个 case 使用 maxK 调用现有 search service
-> 获取 retrieved chunks
-> 根据 expectedChunkIds 标记 relevant
-> 对每个 K 计算 Recall@K
-> 计算 Precision@K
-> 计算 HitRate@K
-> 计算 MRR / RR@K
-> 计算 NDCG@K
-> 计算 ContextPrecision@K
-> 聚合 per-case metrics
-> 聚合 summary metrics
-> 返回 response
```

重点：

- 不重新实现 search；
- 不重新实现 embedding；
- 不重新实现 cosine similarity；
- 不改变每个 case 只查 maxK 一次的逻辑；
- 不改变 documentId filter；
- 不改变 topKValues 标准化逻辑；
- 只扩展指标计算和 response 字段。

## 13. 工程权衡

### 13.1 为什么 Precision@K 和 ContextPrecision@K 都保留

两者服务于不同视角。

Precision@K 是标准 retrieval 指标，分母固定为 K。如果系统请求 Top5，却只返回 3 条，其中 2 条 relevant，Precision@5 是 0.4。它会惩罚返回不足 K 的情况。

ContextPrecision@K 更贴近 RAG context，分母是实际返回数量。同样的例子里，ContextPrecision@5 是 2/3。它衡量实际塞给 LLM 的上下文纯度。

所以一个用于标准检索评估，一个用于 RAG context 质量评估。不能用一个替代另一个。

### 13.2 为什么 NDCG 用 binary relevance

当前 ground truth 是 `expectedChunkIds`，只能表达 relevant / non-relevant。

binary NDCG 虽然不如 graded NDCG 精细，但能先衡量基础排序质量：相关 chunk 是否排得更靠前。

这与当前 API 和数据结构兼容，也避免过早引入复杂标注体系。

### 13.3 为什么不改 request schema

现有 API 已经能支持 binary metrics。

如果改成 `expectedChunks: [{chunkId, relevance}]`，会扩大版本边界：需要新 DTO、兼容旧字段、更新校验、更新测试、重新设计文档。

当前没有 relevance grade 标注，因此 schema 改了也缺少真实数据支撑。后续 benchmark 成熟后再做兼容升级更合理。

### 13.4 为什么不支持 graded relevance

graded relevance 需要标注每个 chunk 的相关程度，例如强相关、中等相关、弱相关。

当前项目还没有真实 benchmark dataset，也没有标注规范。如果现在做 graded relevance，只是在 schema 上提前复杂化。

V2.4.1 先用 binary relevance 建立标准排序评估，后续真实评测数据成熟后再升级。

### 13.5 为什么不做 Faithfulness / Answer Relevance

Faithfulness 和 Answer Relevance 属于 Generation Evaluation，不属于 Retrieval Metrics Completion。

当前版本只看 retrieved chunks 和 expectedChunkIds，不评估 answer 文本。更重要的是，项目当前仍然使用 Mock LLM，做生成质量评估意义不大。

后续真实 LLM、answer persistence、citation alignment、judge model 都准备好后，再做 Generation Evaluation 才有工程价值。

### 13.6 为什么不持久化 eval case

当前仍是 request-driven evaluation。

还没有稳定 dataset 管理、多次 run 对比、dashboard 和企业报告需求。此时持久化会引入较多表设计和产品设计问题。

所以 V2.4.1 只补指标，不加表。后续可以单独做 Evaluation Persistence。

### 13.7 为什么不接真实 Embedding / Vector DB / Rerank

V2.4.1 是评测指标补齐，不是 retrieval engine 升级。

没有标准指标前接 Rerank，很难判断排序是否真的改善。现在补上 Precision@K 和 NDCG@K，正是为了后续评估真实 Embedding、Vector DB、Rerank 的效果。

这体现了先补测量工具，再做优化的工程顺序。

## 14. 产品化 / VIP / 商业化思考

以下是后续产品化方向，不代表当前已经实现。

普通用户：

- 使用默认检索策略；
- 不感知复杂 evaluation 指标；
- 只看到搜索或 RAG Answer 结果。

- 可以查看 Recall@K、Precision@K、MRR、NDCG、ContextPrecision；
- 可以知道知识库检索质量是否可靠；
- 可以判断不同 topK 下上下文噪声和召回的平衡。

VIP / Enterprise：

- 支持更大规模 benchmark；
- 支持 embedding 对比；
- 支持 rerank 前后 NDCG 对比；
- 支持 chunking 策略对比；
- 支持知识库质量报告；
- 支持 retrieval regression test；
- 支持低质量 query 告警；

收费点：

- 按 evaluation case 数收费；

## 18. 面试高强度追问与回答

### Q1：为什么已经有 ContextPrecision@K，还要 Precision@K？

因为它们衡量的对象不同。Precision@K 是标准检索指标，分母固定为 K；ContextPrecision@K 更贴近 RAG context，分母是实际返回数量。

如果请求 Top5 只返回 3 条，其中 2 条 relevant，Precision@5 是 0.4，ContextPrecision@5 是 0.6667。前者会惩罚返回不足 K，后者衡量实际给 LLM 的上下文纯度。

两个指标都保留，可以同时满足标准 retrieval evaluation 和 RAG context quality evaluation。

### Q2：Precision@K 和 Recall@K 有什么区别？

Precision@K 看 TopK 里有多少是 relevant，Recall@K 看 expected relevant chunks 有多少被找回。

Precision 更关注结果纯度，Recall 更关注证据完整性。RAG 中两者都重要：Recall 低可能漏关键证据，Precision 低说明上下文噪声多。

只看一个指标容易误判。高 Recall 但低 Precision 说明找到了证据但混入噪声；高 Precision 但低 Recall 说明返回内容干净但可能漏证据。

### Q3：Precision@K 和 ContextPrecision@K 有什么区别？

Precision@K 的分母固定为 K，是标准检索指标。ContextPrecision@K 的分母是 actualReturnedCountWithinK，更贴近实际传给 LLM 的 context。

在返回结果不足 K 时，两者会明显不同。Precision@K 更严格，ContextPrecision@K 更关注当前实际返回内容的纯度。

项目同时保留两者，是为了既能和标准检索评测对齐，也能解释 RAG context 的质量。

### Q4：为什么 Precision@K 分母固定为 K？

这是标准定义。系统请求 TopK，本质上是在评估 K 个候选位置的质量。如果只返回 3 个结果但 K 是 5，未填满的位置也代表召回能力不足，因此 Precision@K 应该受到影响。

这和 ContextPrecision@K 不同。ContextPrecision 更关心实际返回的 context 质量，因此用 actualReturnedCount 做分母。

### Q5：NDCG@K 是什么？

NDCG@K 是 Normalized Discounted Cumulative Gain，用来衡量排序质量。

它不仅看 relevant chunk 是否出现，还看它排在第几位。排得越靠前，贡献越大；排得越靠后，贡献按 `log2(rank + 1)` 折损。

NDCG 会和理想排序 IDCG 比较，因此结果通常在 0 到 1 之间。越接近 1，说明排序越接近理想状态。

### Q6：为什么 NDCG 比 MRR 更全面？

MRR 只看第一个 relevant chunk 的排名。如果第一个 relevant 排第 1，MRR 就已经很高，但它不关心后续 relevant chunks 排得好不好。

NDCG 会考虑 TopK 中多个 relevant chunks 的位置，因此更适合评估整体 ranking quality。

在有多个 expected chunks 的 RAG 场景中，NDCG 比 MRR 更能反映整体证据排序情况。

### Q7：为什么当前 NDCG 用 binary relevance？

因为当前 request 只有 `expectedChunkIds`，只能表示某个 chunk 是否 relevant。

没有 relevance grade，就不能区分强相关和弱相关。为了不改 schema，也为了保持版本边界，V2.4.1 先实现 binary NDCG。

后续如果 benchmark 支持 `{chunkId, relevance}`，可以升级为 graded NDCG。

### Q8：graded relevance 怎么升级？

可以把 request 从 `expectedChunkIds` 扩展为类似：

```json
"expectedChunks": [
  { "chunkId": 12, "relevance": 3 },
  { "chunkId": 13, "relevance": 2 }
]
```

然后 DCG 使用 relevance grade，而不是 0 / 1。IDCG 也按 relevance 从高到低排序计算。

升级时需要兼容旧字段，明确 relevance 标注规范，并补充测试和文档。

### Q9：为什么不改 request schema？

因为当前二元相关性已经足够支持 Precision@K 和 binary NDCG@K。

改 request schema 会引入兼容问题和更多 DTO 设计，但当前没有真实 relevance grade 标注。为了避免过度设计，V2.4.1 只扩展 response metrics，不改 request。

### Q10：为什么不做 Faithfulness？

Faithfulness 是生成评估，衡量 answer 中的说法是否被 context 支撑。

V2.4.1 是检索评估，只看 retrieved chunks 和 expected evidence。当前仍然使用 Mock LLM，没有真实 answer quality，所以做 Faithfulness 意义不大。

后续真实 LLM、answer persistence、citation alignment 和 judge model 准备好后，再做 Faithfulness 更合理。

### Q11：Faithfulness 和 Retrieval Evaluation 有什么区别？

Retrieval Evaluation 评估“证据有没有被找回来、排序好不好”。Faithfulness 评估“生成的答案是否忠实于证据”。

一个在 generation 之前，一个在 generation 之后。

如果检索不到正确证据，Faithfulness 很难好；但即使检索到了正确证据，模型也可能回答不忠实。所以两者都重要，但应该分阶段实现。

### Q12：为什么不做 Answer Relevance？

Answer Relevance 衡量 answer 是否回答了 query，也属于 generation evaluation。

当前项目 V2.4.1 只补 retrieval metrics，且仍使用 Mock LLM。此时做 Answer Relevance 会变成形式化字段，无法反映真实回答质量。

后续接真实 LLM 和 judge model 后，可以再做 Answer Relevance。

### Q13：为什么不持久化 retrieval_eval_case？

当前仍是 request-driven evaluation，没有稳定 dataset 管理和多次 run 对比需求。

如果现在持久化，需要设计 dataset、case、expected evidence、run、result 等表，还要考虑权限、版本和历史报告。这个边界明显超过 V2.4.1。

因此先不持久化，后续做 Evaluation Dataset Seed 和 Evaluation Persistence 时再设计。

### Q14：后续接 Rerank 时怎么用 NDCG 评估？

可以用同一批 query-evidence benchmark，分别跑 rerank 前和 rerank 后的检索结果。

如果 rerank 有效，相关 chunks 应该整体排得更靠前，NDCG@K 应该提升。MRR 也可能提升，但 NDCG 对多个 relevant chunks 的整体排序更敏感。

所以 V2.4.1 补 NDCG，是为后续 Rerank 评估打基础。

### Q15：后续接真实 Embedding 后怎么比较效果？

用同一套 benchmark cases，分别对 Mock、BGE、Qwen、OpenAI、Jina 等 embedding 模型跑检索。

比较 Recall@K、Precision@K、NDCG@K、MRR、ContextPrecision@K，同时记录 latency、成本、维度、输入长度和部署复杂度。

这样模型选型才有依据，而不是凭主观感觉。

### Q16：如果企业用户要看知识库质量，怎么产品化？

可以让企业维护自己的 query-evidence benchmark，定期评估知识库检索质量。

系统可以输出哪些 query 召回差、哪些文档缺少证据、哪些 chunking 策略表现差、哪些 topK 设置更合适。

后续再配合 dashboard、报告导出、低质量 query 告警，就能变成企业知识库质量管理能力。

### Q17：如果数据量扩大 10 倍，evaluation 的瓶颈在哪里？

瓶颈首先是底层 search。当前仍然是 Mock Embedding + MySQL TEXT vector + Java in-memory exact scan，不适合大规模检索。

evaluation 会放大这个成本，因为每个 case 都要跑一次 search。case 数越多，检索成本越高。

后续需要 Vector DB、ANN、批量评测、异步 evaluation job、结果缓存和持久化 run。

### Q18：如果 Precision 高但 Recall 低，说明什么？

说明返回结果很干净，但漏掉了不少 expected evidence。

对于只需要一个证据的问题，这可能还能回答；但对于需要多个证据综合的问题，答案可能不完整。

优化方向可能是提高 topK、改进 embedding、改进 chunking 或引入多查询检索。

### Q19：如果 Recall 高但 NDCG 低，说明什么？

说明相关证据被找回了，但排序不够好，可能排得比较靠后。

这对 RAG 很危险，因为模型更容易关注前面的 context，靠后的证据可能被忽略或被截断。

这种情况适合引入 rerank，或者优化 embedding / query rewrite，让相关证据排到更前面。

### Q20：如果 ContextPrecision 高但 Precision@K 低，可能是什么情况？

最典型情况是实际返回结果少于 K。

比如 K=5，只返回 2 个 chunk，两个都 relevant。那么 ContextPrecision@5 是 1.0，但 Precision@5 是 0.4。

这说明实际传给 LLM 的上下文很干净，但从标准 TopK retrieval 角度看，系统没有填满候选位。两个指标共同解释，才不会误判。
