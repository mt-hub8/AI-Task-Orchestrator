# V2.6.3 Project Stabilization / Evidence Hardening

## 1. 版本摘要

V2.6.3 不是继续堆新功能的版本，而是一次 **Project Stabilization / Evidence Hardening（项目收敛与证据强化）**。

在这个版本之前，AI Task Orchestrator 已经具备较多模块：

- Reliable Async Task Execution（可靠异步任务执行）
- Transactional Outbox（事务型消息外盒）
- Atomic Claim（原子任务领取）
- Task Attempt（任务执行尝试）
- Mock LLM / Prompt Template / Model Router
- Document Upload / Chunking
- EmbeddingProvider 抽象
- Local Embedding Worker 原型
- Retrieval Evaluation Harness
- VectorStore 抽象
- ExactCosineVectorStore
- QdrantVectorStore 原型
- VectorStore Benchmark Harness

但项目进入这个阶段后，最重要的问题已经不是“继续加功能”，而是：

> 让代码状态、项目文档、可演示链路和任务可靠性闭环变得可信、可复现、可解释。

因此 V2.6.3 合并处理了三类收敛任务：

1. **README / docs 状态同步**：让项目文档准确反映当前实现，不再把已实现能力写成未来能力。
2. **E2E Demo Golden Path**：补齐可复现演示路径，让面试官或 reviewer 能快速看到系统真实工作。
3. **Atomic Finalization**：补强异步任务终态写入，避免成功、失败、超时、取消之间互相覆盖。

这个版本的核心价值是：

> 把项目从“功能很多”推进到“状态清楚、能跑能看、可靠性链路更闭环”。

---

## 2. 为什么暂停继续做 V2.7

原计划下一步是 V2.7 Retrieval Policy & VIP Search，但在进入 V2.7 前，项目暴露出更高优先级的问题：

1. README 和代码状态不一致；
2. 项目功能很多，但缺少一条最小可演示闭环；
3. 任务执行终态更新存在竞态风险；
4. Qdrant、VectorStore、Retrieval Evaluation 等能力已经有代码，但项目对外说明没有同步；
5. 继续做 VIP Search 会让项目更大，但不一定让项目更可信。

所以 V2.6.3 选择先收敛，而不是继续扩张。

这是一个工程化判断：

> 当项目能力增长到一定规模后，继续加功能的边际收益下降，补文档、补 demo、补可靠性证据的收益更高。

## 3. 本版本解决的问题

### 3.1 README 和代码状态不一致

之前 README 还可能把以下已经实现或已原型化的能力写成未实现：

- Retrieval Evaluation
- EmbeddingProvider
- VectorStore
- QdrantVectorStore
- VectorStore Benchmark
- Local Embedding Worker

这会让面试官产生不好的判断：

> 项目推进很快，但 owner 对项目当前状态掌控不稳定。

V2.6.3 将 README 和基础 docs 改成三层状态：

- Implemented（已实现）
- Prototype / Experimental（原型或实验性实现）
- Not Implemented Yet（尚未实现）

这个结构能清楚表达：

- 哪些能力已经在代码里；
- 哪些能力只是原型或手工验证；
- 哪些能力尚未实现，不能夸大。

---

### 3.2 项目缺少可复现演示路径

项目模块很多，但面试官会问：

> 我拉下代码后，怎么在几分钟内看到它真的能跑？

V2.6.3 新增了 E2E Demo Golden Path，目标是让外部 reviewer 能顺着文档和 `.http` 文件运行核心链路：

- 启动基础设施；
- 启动 Spring Boot；
- 创建任务；
- 查询任务状态；
- 查询 Task Attempt；
- 查询 Output Chunks；
- 上传文档；
- 触发 chunking / embedding / search；
- 运行 Retrieval Evaluation；
- 可选手工验证 Qdrant / Local Worker。

这个动作把项目从“代码存在”变成“可复现”。

---

### 3.3 任务终态写入还不够硬

此前系统已经实现 Atomic Claim，它解决的是：

> 谁能把任务从 PENDING 领取成 RUNNING。

但任务结束时仍然可能出现竞态：

```text
线程 A：任务执行成功，准备写 SUCCESS
线程 B：超时扫描器把任务改成 TIMED_OUT / FAILED
线程 A：继续写 SUCCESS，覆盖掉超时终态
```

如果终态写入逻辑是：

```text
find task
check status == RUNNING
save SUCCESS
```

那么检查和保存之间仍然有时间窗口。

V2.6.3 引入 Atomic Finalization：

```sql
UPDATE task
SET status = 'SUCCESS', result_content = ?
WHERE id = ?
  AND status = 'RUNNING'
```

通过 affected rows 判断是否写入成功：

- `updated == 1`：本线程成功写入终态；
- `updated == 0`：任务已经被其他终态处理，不允许覆盖。

这让任务可靠性链路从：

```text
Atomic Claim
```

升级为：

```text
Atomic Claim + Atomic Finalization
```

- 代码已接入；
- 默认测试不依赖外部服务；

## 6. Atomic Finalization 设计

### 6.1 Atomic Claim 与 Atomic Finalization 的区别

Atomic Claim 解决任务开始阶段：

```sql
UPDATE task
SET status = 'RUNNING'
WHERE id = ?
  AND status = 'PENDING'
```

它保证只有一个执行者能领取任务。

Atomic Finalization 解决任务结束阶段：

```sql
UPDATE task
SET status = 'SUCCESS'
WHERE id = ?
  AND status = 'RUNNING'
```

它保证只有当前仍然是 RUNNING 的任务才能写入终态。

两者组合后，任务生命周期变成：

```text
谁能开始执行：Atomic Claim 决定
谁能写入终态：Atomic Finalization 决定
```

---

### 6.2 为什么多次检查 `isTaskRunning` 不够

多次检查比不检查好，但仍然不是最强解法。

问题在于：

```text
check status == RUNNING
        ↓
其他线程修改状态
        ↓
save SUCCESS
```

检查和保存之间不是原子操作。

条件更新将检查和更新合成一条数据库语句：

```sql
UPDATE task
SET status = ?
WHERE id = ?
  AND status = 'RUNNING'
```

数据库保证这条语句的原子性。

---

### 6.3 成功路径

成功路径必须满足：

- 只有 RUNNING 任务才能变成 SUCCESS；
- 如果任务已经 FAILED / TIMED_OUT / CANCELLED / SUCCESS，不允许再被写成 SUCCESS；
- finalization rejected 时，不应该保存成功 output chunks；
- finalization rejected 时，不应该把 attempt 标记为 SUCCESS；
- task finalization、output chunks、attempt status 要保持事务一致性。

核心原则：

> 不能出现 task 终态写入失败，但成功输出和成功 attempt 仍然落库的情况。

---

### 6.4 失败 / 超时路径

失败路径：

```sql
UPDATE task
SET status = 'FAILED', error_message = ?
WHERE id = ?
  AND status = 'RUNNING'
```

超时路径：

```sql
UPDATE task
SET status = 'TIMED_OUT', error_message = ?
WHERE id = ?
  AND status = 'RUNNING'
```

如果 affected rows 为 0，说明任务已经进入其他终态，不允许覆盖。

## 7. 方案选择与权衡

### 7.1 为什么不使用 `@Version`

JPA `@Version` 可以解决通用并发覆盖问题，但本版本的问题更明确：

> 只有 RUNNING 任务才能进入终态。

条件更新能直接表达业务状态转移，不需要新增字段，也不需要数据库迁移。

---

### 7.2 为什么不使用悲观锁

悲观锁可以保证强一致，但会增加锁等待和事务复杂度。

当前只需要 CAS（compare-and-set）式状态转移：

```text
if status == RUNNING then update to final status
```

条件更新更轻量。

---

### 7.3 为什么不重写完整状态机

完整状态机能统一管理所有状态转移，但现在项目已有状态流。V2.6.3 的目标是补齐最关键的终态竞态，不是重写整个任务生命周期。

## 8. 测试方案

### 8.1 Repository 条件更新测试

应覆盖：

- RUNNING -> SUCCESS 成功；
- FAILED -> SUCCESS 不覆盖；
- TIMED_OUT -> SUCCESS 不覆盖；
- SUCCESS -> SUCCESS 不覆盖；
- RUNNING -> FAILED 成功；
- SUCCESS -> FAILED 不覆盖；
- RUNNING -> TIMED_OUT 成功；
- SUCCESS -> TIMED_OUT 不覆盖；
- 如果存在 CANCELLED，则覆盖 CANCELLED -> SUCCESS / FAILED 不覆盖。

---

### 8.2 Service 测试

应覆盖：

- success finalization 成功时返回 true / success result；
- task 已经 FAILED 时，success finalization 返回 false；
- task 已经 TIMED_OUT 时，success finalization 返回 false；
- failed finalization 不覆盖 SUCCESS；
- timeout finalization 不覆盖 SUCCESS。

---

### 8.3 Execution 流程测试

如果当前结构允许，应覆盖：

- 执行成功时，task 从 RUNNING 变 SUCCESS；
- 执行成功前任务已被 timeout，则不会覆盖；
- finalization rejected 时，不把 attempt 标记为 SUCCESS；
- finalization rejected 时，不保存成功 output chunks，或事务回滚。

---

### 8.4 回归测试

需要保证：

- Atomic Claim 不被破坏；
- Outbox dispatch 不被破坏；
- Task Attempt 不被破坏；

默认测试仍然不依赖：

## 13. 面试高强度追问与回答

### Q1：为什么暂停做新功能，先做项目收敛？

A：因为项目已经有较多模块，如果继续加功能但文档、demo 和可靠性证据跟不上，项目可信度会下降。V2.6.3 的目标是让当前系统状态清楚、演示路径清楚、可靠性链路更完整，再继续做 VIP Search / Rerank / Agent Runtime。

---

### Q2：为什么 README 要分 Implemented / Prototype / Not Implemented Yet？

A：因为项目里有一些能力已经有代码，但不是 production-ready，例如 QdrantVectorStore 和 Local Embedding Worker。用三层状态可以避免低估已实现能力，也避免夸大原型能力。

---

### Q3：E2E Demo 的价值是什么？

A：它让项目从“看起来有很多模块”变成“可以按步骤跑通的系统”。面试官可以通过 `.http` 文件快速验证任务创建、异步执行、attempt、output chunks、文档上传、retrieval search 和 evaluation 等链路。

---

### Q4：为什么默认 demo 不依赖 Qdrant / OpenAI / Python worker？

A：默认 demo 应该低依赖、稳定、易复现。Qdrant、OpenAI、Python worker 都属于外部依赖或可选增强能力，应该放到手工验证路径，而不是默认 demo 路径。

---

### Q5：Atomic Claim 和 Atomic Finalization 的区别是什么？

A：Atomic Claim 解决任务开始阶段的竞争，确保只有一个 worker 能领取任务。Atomic Finalization 解决任务结束阶段的竞争，确保只有 RUNNING 任务才能写入 SUCCESS / FAILED / TIMED_OUT，防止终态覆盖。

---

### Q6：为什么多次检查 `isTaskRunning` 仍然不够？

A：因为检查和保存之间仍然有竞态窗口。线程 A 检查时任务是 RUNNING，线程 B 可能马上改成 TIMED_OUT，线程 A 随后 save SUCCESS 就会覆盖超时结果。条件更新把检查和更新合成一条 SQL，数据库保证原子性。

---

### Q7：为什么不用乐观锁？

A：乐观锁是通用并发控制，但当前问题是明确的状态转移约束：只有 RUNNING 才能进入终态。条件更新更直接表达业务规则，也不需要新增 version 字段和 Flyway migration。

---

### Q8：success 和 timeout 同时发生，谁赢？

A：谁先成功执行条件更新谁赢。后执行的一方会因为 `WHERE status = RUNNING` 不满足而 affected rows 为 0，从而不会覆盖已有终态。

---

### Q9：finalization rejected 时怎么处理？

A：不能继续保存成功 output chunks，也不能把 attempt 标记为 SUCCESS。应该记录日志并停止成功路径，保证不会产生“任务终态失败但成功结果落库”的假象。

---

### Q10：这个版本为什么比 V2.7 VIP Search 更重要？

A：VIP Search 是产品策略，但它依赖一个可信的基础系统。如果 README 不准、demo 不完整、任务终态可能被覆盖，那么继续加 VIP 策略会让项目更复杂但不更可信。V2.6.3 先补项目证据和可靠性底座。

---
