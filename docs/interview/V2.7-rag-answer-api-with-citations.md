# V2.7 RAG Answer API with Citations

## 1. 版本摘要

V2.7 的目标是为 AI Task Orchestrator 增加一个真正可体验的 **RAG Answer API with Citations（带引用来源的 RAG 问答接口）**。

在这个版本之前，项目已经具备大量 RAG 基础设施能力：

- Document Upload（文档上传）；
- Chunking（文档切分）；
- EmbeddingProvider 抽象；
- Embedding Cache / Dedup；
- Embedding Cache Metrics；
- VectorStore 抽象；
- ExactCosineVectorStore；
- QdrantVectorStore；
- Qdrant Manual Verification；
- VectorStore Benchmark；
- Local Embedding Worker Packaging；
- API Error Response Standardization。

但是这些能力大多是底层能力。它们能证明工程深度，但用户还不能直接感知：

> 我上传了一份文档，然后问一个问题，系统基于文档回答我，并告诉我答案依据来自哪里。

V2.7 解决的就是这个问题。

本版本新增：

```http
POST /rag/answers
```

用户可以提交 query，系统会：

```text
query
  -> query embedding
  -> VectorStore search
  -> topK chunks
  -> citations
  -> RAG prompt
  -> LLM generation
  -> answer
```

最终返回：

- answer（最终回答）；
- citations（引用来源）；
- retrieval metadata（检索元数据）；
- generation metadata（生成元数据）。

这个版本是项目从“RAG Infra”走向“可用 AI 应用”的关键转折点。

---

## 2. 为什么 V2.7 很重要

### 2.1 之前更多是基础设施，V2.7 开始有用户体验

前面的版本解决了很多工程问题：

```text
任务可靠性
错误响应标准化
Qdrant 接入
benchmark 结果采集
embedding cache
cache metrics
```

这些都很重要，但用户并不会直接感知。

用户真正关心的是：

```text
我问的问题能不能被回答？
回答是不是基于我的文档？
系统能不能告诉我答案依据？
```

V2.7 让项目第一次具备完整的 RAG 问答体验。

---

### 2.2 把分散模块串成闭环

在 V2.7 之前，很多模块是分散存在的：

```text
EmbeddingProvider
VectorStore
LLM Client
Document Chunk
Benchmark
Cache
Metrics
```

V2.7 把这些模块串成一个业务链路：

```text
用户问题
  -> embedding
  -> 检索相关文档片段
  -> 构造 prompt
  -> 调用 LLM
  -> 返回答案与引用
```

这意味着项目不只是“功能很多”，而是开始形成真实产品能力。

---

### 2.3 回答必须可追溯

RAG 系统如果只返回 answer，会有一个问题：

> 这个答案是模型根据文档生成的，还是模型自己编的？

因此 V2.7 必须返回 citations。

Citation 的价值是：

- 告诉用户答案依据来自哪些 chunk；
- 让 reviewer 可以检查回答是否有来源；
- 降低 hallucination（幻觉）风险；
- 为后续 answer evaluation 打基础；
- 让系统更像真实知识库产品。

## 3. API 设计

### 3.1 请求接口

新增同步接口：

```http
POST /rag/answers
```

请求体第一版保持克制：

```json
{
  "query": "V2.6.8 Embedding Cache 的 cache key 为什么不能只用 chunkHash？",
  "topK": 5
}
```

字段说明：

| 字段 | 含义 | 规则 |
|---|---|---|
| `query` | 用户问题 | 必填，不能为空 |
| `topK` | 检索返回数量 | 可选，默认 5，范围 1 到 10 |

第一版不引入：

- userPlan；
- temperature；
- model override；
- provider override；
- documentIds filter；
- conversationId；
- sessionId；
- streaming；
- rerank；
- hybrid search。

原因是 V2.7 的目标是先跑通单轮 RAG 问答闭环，不是一次性做完整聊天系统。

---

### 3.2 响应结构

典型响应：

```json
{
  "answer": "Embedding Cache 的 cache key 不能只用 chunkHash，因为 chunkHash 只能说明文本内容相同，不能说明 embedding space 相同。同一段文本如果使用不同 provider、model 或 dimension 生成 embedding，向量不能混用。因此 cache key 必须包含 chunkHash + provider + model + dimension。",
  "citations": [
    {
      "sourceIndex": 1,
      "documentId": 10,
      "chunkId": 101,
      "score": 0.91,
      "contentSnippet": "chunkHash 只能说明文本内容一样，但不能说明向量空间一样。provider、model、dimension 才能说明 embedding space 一样。"
    }
  ],
  "retrieval": {
    "topK": 5,
    "returned": 1,
    "provider": "local-worker",
    "model": "sentence-transformers/all-MiniLM-L6-v2",
    "dimension": 384,
    "vectorStore": "ExactCosineVectorStore"
  },
  "generation": {
    "provider": "mock",
    "model": "mock-llm"
  }
}
```

核心字段：

| 字段 | 含义 |
|---|---|
| `answer` | LLM 生成的最终回答 |
| `citations` | 回答引用的检索来源 |
| `retrieval` | 检索链路元数据 |
| `generation` | 生成链路元数据 |

## 4. Citation 设计

### 4.1 citation 字段

每条 citation 至少包含：

```text
sourceIndex
documentId
chunkId
score
contentSnippet
```

字段解释：

| 字段 | 含义 |
|---|---|
| `sourceIndex` | 引用编号，从 1 开始 |
| `documentId` | 来源文档 ID |
| `chunkId` | 来源 chunk ID |
| `score` | VectorStore 返回的相关性分数 |
| `contentSnippet` | chunk 内容摘要或截断片段 |

---

### 4.2 sourceIndex 的作用

`sourceIndex` 用于把 answer 中的引用标记和 citations 对应起来。

例如 answer 中出现：

```text
Embedding Cache 的 key 不能只用 chunkHash，因为 chunkHash 只能说明文本一致，不能说明 embedding space 一致。[1]
```

其中 `[1]` 对应：

```json
{
  "sourceIndex": 1,
  "documentId": 10,
  "chunkId": 101
}
```

这样回答就不是一个黑盒文本，而是可追溯到具体文档片段。

---

### 4.3 contentSnippet 为什么要截断

不建议把完整 chunk 原样返回得过长。

原因：

- API 响应会变大；
- 前端展示不方便；
- chunk 可能包含大量无关内容；
- citation 的目标是辅助定位，不是复制完整文档。

第一版可以把 snippet 控制在 300 到 500 字符左右。

---

## 5. 核心 Pipeline

V2.7 的核心链路是：

```text
1. 接收 query；
2. 校验 query / topK；
3. 使用 EmbeddingProvider 生成 query embedding；
4. 使用 VectorStore 检索 topK chunks；
5. 将检索结果转换为 citations；
6. 如果没有 citations，返回 no-context answer；
7. 如果有 citations，构造 RAG prompt；
8. 调用 LLM provider；
9. 返回 answer + citations + metadata。
```

这条链路的意义是：

> 把检索和生成连起来，把底层 RAG infra 变成用户可调用的问答能力。

---

## 6. Query Embedding 为什么不进入 Embedding Cache

V2.6.8 的 Embedding Cache 主要用于文档 chunk embedding。

V2.7 中用户 query 也需要 embedding，但第一版不建议写入 `embedding_cache`。

原因：

1. query 生命周期短，和文档 chunk 不同；
2. query 可能包含隐私信息；
3. query 重复率不一定高；
4. query cache 需要 TTL；
5. query cache 需要用户隔离；
6. query cache 需要清理策略；
7. 当前版本只做单轮 RAG answer。

所以 V2.7 的边界是：

```text
document chunk embedding：继续使用 Embedding Cache
query embedding：直接调用 EmbeddingProvider，不写 embedding_cache
```

这是一个重要的正确性和隐私边界。

## 7. Prompt 设计

### 7.1 PromptBuilder 的目标

`RagPromptBuilder` 的目标是把检索到的 chunks 编号后组织成上下文，让 LLM 基于这些上下文回答问题。

Prompt 的核心原则：

```text
只根据提供的上下文回答；
如果上下文不足，明确说无法确定；
尽量使用 [1]、[2] 这样的引用标记；
不要编造文档中没有的信息。
```

---

### 7.2 Prompt 模板示例

可以使用类似模板：

```text
你是一个基于文档回答问题的助手。

请只根据下面的上下文回答用户问题。
如果上下文中没有足够信息，请回答：“根据当前检索到的文档内容，无法确定。”
回答中尽量使用 [1]、[2] 这样的引用标记指向对应来源。

上下文：
[1]
documentId: 10
chunkId: 101
content: ...

[2]
documentId: 10
chunkId: 102
content: ...

用户问题：
{query}

请给出简洁、准确、可追溯的回答。
```

这个 prompt 不是为了追求复杂，而是为了让第一版 RAG answer 行为稳定、可解释、可测试。

---

### 7.3 为什么要求“不知道就说不知道”

RAG 系统最常见的问题是 hallucination（幻觉）。

如果检索到的上下文不足，LLM 可能凭常识或训练数据编造答案。

V2.7 通过 prompt 约束：

```text
如果上下文中没有足够信息，请回答：“根据当前检索到的文档内容，无法确定。”
```

这能降低幻觉风险。

注意：这不能完全消除幻觉，但它是第一版 RAG 系统必须具备的基本防线。

## 8. Mock LLM 设计

### 8.1 为什么默认测试必须使用 Mock LLM

V2.7 不能让默认测试依赖真实 LLM。

如果默认测试访问 OpenAI 或其他外部 LLM，会带来：

- API key 依赖；
- 网络依赖；
- 成本问题；
- 结果不稳定；
- CI 不稳定；
- 模型输出不可控。

因此默认测试必须使用 deterministic mock LLM。

---

### 8.2 Mock LLM 的职责

Mock LLM 不需要生成高质量回答。

它的职责是验证链路：

- prompt 是否构造；
- citations 是否传入；
- LLM provider 是否被调用；
- answer 是否返回；
- 默认测试是否不依赖外部服务。

例如 mock answer 可以是：

```text
根据检索到的上下文，问题与以下来源相关：[1] [2]
```

这样足够验证 RAG Answer API 的流程。

---

## 9. 无检索结果时的行为

如果 VectorStore search 返回空结果，V2.7 不应该返回 404 或 500。

应该返回 HTTP 200：

```json
{
  "answer": "根据当前检索到的文档内容，无法确定。",
  "citations": [],
  "retrieval": {
    "topK": 5,
    "returned": 0
  },
  "generation": {
    "skipped": true,
    "reason": "NO_RETRIEVED_CONTEXT"
  }
}
```

原因：

> 没有检索到相关上下文是正常业务结果，不是系统异常。

同时，这种情况下不需要调用 LLM。

因为没有上下文时调用 LLM 只会增加 hallucination 风险。

## 10. 错误处理

V2.7 继续使用 V2.6.4 的统一错误响应。

### 10.1 参数校验错误

例如：

- query 为空；
- query 超长；
- topK < 1；
- topK > 10。

应该返回：

```text
VALIDATION_ERROR
```

---

### 10.2 EmbeddingProvider 失败

如果 query embedding 失败，系统无法检索。

应该返回：

```text
EMBEDDING_PROVIDER_ERROR
```

不能吞掉异常，也不能返回空 answer。

---

### 10.3 VectorStore 失败

如果 VectorStore search 失败，系统无法找到上下文。

应该返回：

```text
VECTOR_STORE_ERROR
```

不能伪造 answer。

---

### 10.4 LLM provider 失败

如果 LLM provider 调用失败，应该返回：

```text
LLM_PROVIDER_ERROR
```

第一版不返回 partial answer。

原因是：

- retrieval 成功但 generation 失败，整体问答仍然失败；
- partial response 会让 API 语义复杂化；
- 后续可以单独做 partial response 或 degraded response。

---

## 11. 为什么选择同步 API

V2.7 选择：

```http
POST /rag/answers
```

同步返回结果。

没有接入已有 Task Orchestrator 异步任务系统。

原因：

- 第一版体验更直接；
- 便于 Web UI 后续调用；
- 便于测试；
- 不需要新增任务类型；
- 不扩大状态机范围；
- 不引入轮询；
- 当前目标是先让用户能问答。

后续如果需要长耗时任务，可以做：

```text
Async RAG Answer Task
```

再接入 Task Orchestrator。

---

## 12. 为什么不做 Streaming

V2.7 不做 streaming：

- 不做 SSE；
- 不做 WebSocket；
- 不做 token streaming；
- 不做 chunked response。

原因：

- streaming 会改变 response 结构；
- citations 何时返回需要设计；
- 前端也要配合；
- 测试复杂度上升；
- 当前最重要的是先完成非流式单轮问答闭环。

---

## 13. 为什么不做多轮对话

V2.7 不做：

- conversationId；
- sessionId；
- chat history；
- memory；
- follow-up question rewriting。

原因：

多轮对话会引入：

- 历史消息存储；
- query rewriting；
- 上下文压缩；
- 引用归因；
- 会话过期；
- 用户隔离；
- 隐私与清理策略。

这些都应该后续单独设计。

V2.7 只做：

```text
one query -> one answer
```

边界清楚，容易验收。

---

## 14. 为什么不做 documentIds filter

V2.7 第一版不强制做 document filter。

原因：

- documentIds filter 需要 VectorStore 支持 metadata filter；
- ExactCosineVectorStore 和 QdrantVectorStore 都需要一致语义；
- 会扩大版本范围；
- 当前目标是先跑通全局 RAG answer。

后续 Web UI 或知识库模式中可以再做：

```json
{
  "query": "...",
  "documentIds": [1, 2, 3]
}
```

但 V2.7 不必做。

---

## 15. 为什么不保存问答历史

V2.7 不新增：

```text
rag_answer_history
chat_message
conversation
answer_log
```

原因：

- 会引入用户隐私；
- 需要 user/session；
- 需要历史查询 API；
- 需要清理策略；
- 需要权限隔离；
- 会扩大范围。

当前只做无状态 API。

## 21. 面试高强度追问与回答

### Q1：V2.7 和之前的 retrieval evaluation 有什么区别？

A：Retrieval evaluation 是工程验证能力，用来评估检索质量；V2.7 是用户功能入口，用户提交 query 后系统会检索上下文、调用 LLM 并返回 answer 和 citations。前者是评估工具，后者是产品 API。

---

### Q2：为什么 answer 要带 citations？

A：因为 RAG 的核心不是只生成答案，而是让答案可追溯。Citations 可以告诉用户回答依据来自哪个 documentId / chunkId，降低黑盒生成和幻觉风险，也方便后续评估 answer 是否 faithful。

---

### Q3：为什么 query embedding 不进入 embedding_cache？

A：embedding_cache 主要用于文档 chunk embedding 去重。query 生命周期短、重复率不一定高，而且可能包含用户隐私。如果缓存 query，还需要 TTL、用户隔离和清理策略。V2.7 先不做 query cache。

---

### Q4：为什么没有检索结果时返回 200，而不是 404？

A：没有检索到相关上下文是正常业务结果，不是系统异常。系统应该明确回答“根据当前检索到的文档内容，无法确定”，并返回空 citations，而不是把它当成错误。

---

### Q5：为什么不调用 LLM 处理 no-context 情况？

A：没有上下文时调用 LLM 容易让模型凭常识或训练数据编造答案。为了降低 hallucination 风险，V2.7 在 no-context 情况下直接返回固定 no-context answer。

---

### Q6：为什么先做同步 API，而不是走 Task Orchestrator？

A：同步 API 更适合第一版体验，测试简单，也方便后续 Web UI 调用。Task 异步模式适合长任务、重试、取消和状态追踪，可以后续单独接入。

---

### Q7：为什么不做 streaming？

A：streaming 会改变响应结构，而且 citations 何时返回需要额外设计。V2.7 先做非流式完整回答，保证 RAG answer 闭环稳定。

---

### Q8：为什么默认用 mock LLM？

A：默认测试必须稳定、低依赖、无外部 API 成本。真实 LLM 输出不稳定，也依赖 API key 和网络。Mock LLM 可以验证链路正确性，真实 provider 作为可选配置启用。

---

### Q9：这个版本是否已经能代表完整 RAG 产品？

A：它是第一版可用 RAG Answer API，但还不是完整产品。它还没有 Web UI、部署、多轮对话、streaming、document filter、Rerank 和 Hybrid Search。它的价值是先把单轮、可引用、可测试的 RAG 问答闭环打通。

---

### Q10：后续怎么提升答案质量？

A：可以从几个方向提升：Rerank 提高上下文排序质量；Hybrid Search 解决关键词和专有名词问题；document filter 缩小检索范围；Generation Evaluation 评估 answer faithfulness；streaming 和 Web UI 提升体验。

## 23. 本版本在项目中的位置

V2.7 是从基础设施到用户体验的转折版本。

它之前：

```text
V2.6.x：VectorStore、Qdrant、Benchmark、Embedding Cache、Cache Metrics
```

它解决：

```text
用户如何基于文档问问题
检索结果如何喂给 LLM
回答如何引用来源
无上下文时如何避免幻觉
默认测试如何保持低依赖
```

它之后：

```text
Web UI
Deployable Stack
VIP Search
Rerank
Hybrid Search
Agent Runtime
```

本版本最大的意义是：

> 让 AI Task Orchestrator 第一次具备可以被用户直接体验的 RAG 问答能力。
