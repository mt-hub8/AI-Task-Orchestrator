# V2.6.8 Embedding Cache / Dedup

## 1. 版本摘要

V2.6.8 的目标是为 AI Task Orchestrator 增加 **Embedding Cache / Dedup（Embedding 缓存与去重）** 能力。

在这个版本之前，系统已经具备：

- 文档上传与切分；
- EmbeddingProvider 抽象；
- Mock / OpenAI-compatible / Local Worker 等 embedding provider；
- VectorStore 抽象；
- ExactCosineVectorStore；
- QdrantVectorStore；
- Qdrant 手工验证路径；
- exact vs qdrant benchmark 结果采集能力。

但是，RAG ingestion（文档入库）链路里还有一个非常现实的问题：

> 相同文本、相同 provider、相同 model、相同 dimension 下，不应该重复生成 embedding。

如果用户重复上传同一份文档，或者不同文档里出现相同 chunk，系统如果每次都重新调用 embedding provider，会带来：

- 重复 API 成本；
- 重复 Local Worker 推理开销；
- 文档入库变慢；
- 大规模 ingestion 时成本不可控；
- 后续商业化 quota / billing / rate limit 难以精细化管理。

V2.6.8 的核心设计是：

```text
用 chunkHash + provider + model + dimension 作为 cache key，
把已经生成过的 embedding 持久化到 embedding_cache 表中。
```

只有当这四个条件都一致时，系统才复用已有 embedding。

---

## 2. 为什么现在做

Embedding Cache / Dedup 不是一开始就应该做的功能。

如果项目还没有完成文档切分、embedding provider、vector store 和检索评估，过早做 cache 容易变成空中楼阁。现在做它比较合理，因为前面版本已经完成了以下基础：

```text
V2.5.x：EmbeddingProvider 抽象、真实 embedding provider、Local Worker
V2.6.x：VectorStore 抽象、Qdrant、Benchmark、Manual Verification
```

也就是说，系统已经能完整执行：

```text
文档上传
  -> chunking
  -> embedding
  -> vector upsert
  -> search
  -> evaluation / benchmark
```

当这条链路跑通后，下一步自然会出现成本和重复计算问题。

面试官很可能会追问：

> 如果同一份文档重复上传，你会重新 embedding 一遍吗？  
> 如果多个文档里有相同段落，你怎么避免重复调用 embedding API？  
> 如果换了 embedding model，旧 cache 还能不能复用？  
> 你怎么保证不同模型的向量不会混在一起？

V2.6.8 就是对这些问题的工程化回答。

---

## 3. 本版本解决的问题

### 3.1 重复 embedding 成本问题

Embedding 生成是 RAG 系统里成本很高的一环。

如果使用外部 API，例如 OpenAI-compatible embedding provider，每次 embedding 调用都可能产生费用。如果相同文本反复出现，每次都重新生成 embedding，会造成直接成本浪费。

即使使用本地 Local Embedding Worker，也会消耗 CPU / GPU 资源，并增加 ingestion 延迟。

所以 cache 的第一层价值是：

> 避免重复计算，减少成本和延迟。

---

### 3.2 文档重复上传问题

实际业务中，重复内容很常见：

- 用户重复上传同一份文档；
- 多个用户上传相似模板；
- 企业文档里有大量重复页眉、页脚、免责声明；
- FAQ / 合同 / 说明书里存在重复段落；
- 多版本文档中大量内容相同。

如果没有 chunk 级别的 embedding dedup，即使文本完全一样，也会反复生成 embedding。

V2.6.8 不做完整 document-level dedup（文档级去重），而是先做 chunk-level embedding cache（chunk 级 embedding 缓存）。这是更克制、更容易落地的方案。

---

### 3.3 向量空间正确性问题

Embedding cache 不能只看文本是否相同。

同一段文本：

```text
AI Task Orchestrator is a backend system.
```

使用不同 provider 或不同 model 生成的向量通常不是同一个向量空间。

例如：

```text
provider = openai-compatible, model = text-embedding-3-small, dimension = 1536
provider = local-worker, model = sentence-transformers/all-MiniLM-L6-v2, dimension = 384
```

这两组 embedding 即使来自同一段文本，也不能互相替代。

因此 cache key 必须包含：

```text
chunkHash + provider + model + dimension
```

这不是普通优化细节，而是 embedding 系统的正确性边界。

---

### 3.4 后续商业化成本控制问题

未来如果项目继续做：

- VIP Search；
- quota；
- billing；
- rate limit；
- embedding usage 统计；
- provider cost 统计；

Embedding Cache 都会成为基础能力。

因为它可以回答：

> 系统因为 cache 命中了多少次？  
> 少调用了多少次 provider？  
> 节省了多少成本？  
> 哪些文档重复度最高？  
> 哪个模型的 cache hit rate 更高？

V2.6.8 先把 cache 机制做出来，为后续指标化和商业化打基础。

## 4. 核心设计

### 4.1 数据库持久化 cache

V2.6.8 采用数据库持久化 cache，而不是只做内存 cache。

推荐表：

```text
embedding_cache
```

核心字段：

| 字段 | 含义 |
|---|---|
| `id` | 主键 |
| `chunk_hash` | chunk 内容 hash |
| `provider` | embedding provider |
| `model` | embedding 模型名 |
| `dimension` | embedding 向量维度 |
| `embedding_json` | embedding 向量序列化后的 JSON |
| `created_at` | 首次写入时间 |
| `updated_at` | 更新时间 |

唯一约束：

```text
unique(chunk_hash, provider, model, dimension)
```

这个唯一约束非常关键，它保证同一个 embedding space 下的同一段文本只会保存一条 cache 记录。

---

### 4.2 为什么不用纯内存缓存

内存缓存可以很快实现，例如：

```text
Map<EmbeddingCacheKey, float[]>
```

但它有明显缺点：

- 服务重启后 cache 消失；
- 多实例之间无法共享；
- 无法作为长期证据；
- 不利于后续统计；
- 不适合大规模 ingestion；
- 面试表达上更像临时优化。

数据库持久化 cache 更适合当前项目阶段，因为它能跨重启复用，也能为后续 cache metrics 打基础。

---

### 4.3 为什么 embedding 用 JSON 存储

V2.6.8 选择把 embedding 向量序列化为 JSON 字符串，例如：

```json
[0.0123, -0.0456, 0.0789]
```

存入 `embedding_json`。

这样做的原因是：

- 实现简单；
- 不依赖 MySQL vector 扩展；
- 便于调试；
- 不改变现有 VectorStore 设计；
- 不把 MySQL 当成向量检索引擎；
- 当前 cache 只是为了复用生成结果，不负责相似度搜索。

它的缺点也要清楚：

- JSON 比二进制占空间；
- 不适合超大规模向量缓存；
- 不适合在 MySQL 中做 vector search；
- 反序列化会有一定开销。

但这些缺点可以接受，因为真正的 vector search 仍然由 `VectorStore` / Qdrant 负责。

---

### 4.4 为什么 cache key 必须包含 provider / model / dimension

错误设计：

```text
cache key = chunkHash
```

问题是：

```text
同一段文本用 model A 生成过 embedding；
后来切到 model B；
系统如果只看 chunkHash，就会错误复用 model A 的向量。
```

正确设计：

```text
cache key = chunkHash + provider + model + dimension
```

这样系统只会在同一个 embedding space 内复用向量。

面试中可以这样表达：

> chunkHash 只能说明文本一样，provider / model / dimension 才能说明向量空间一样。Embedding cache 复用的前提不是“文本一样”，而是“文本一样，并且 embedding space 一样”。

---

## 5. chunkHash 设计

### 5.1 使用 SHA-256

V2.6.8 推荐对 chunk content 计算 SHA-256：

```text
chunkHash = sha256(normalizedChunkContent)
```

选择 SHA-256 的原因：

- 稳定；
- 冲突概率极低；
- 输出长度固定；
- 适合数据库唯一键；
- 比直接把长文本作为唯一键更合理。

---

### 5.2 normalize 要克制

可以做非常轻量的 normalize，例如：

- 统一换行符；
- trim 前后空白。

但不应该做过度处理，例如：

- 转小写；
- 删除标点；
- 去停用词；
- 合并多个空格；
- 改写文本；
- 语义归一化。

原因是：

> cache 的 key 应该尽量接近实际送入 embedding model 的文本。

如果 normalize 过度，可能导致两个实际不同的文本错误复用同一个 embedding。

---

## 6. EmbeddingCacheService 设计

### 6.1 职责

`EmbeddingCacheService` 的职责是封装 cache hit / miss 逻辑。

它不应该负责文档上传，也不应该负责 VectorStore search。它只处理：

```text
给定 chunk content + provider + model + dimension，
返回 embedding，
并告诉调用方本次是否 cache hit。
```

典型流程：

```text
1. 对 chunk content 计算 chunkHash；
2. 用 chunkHash + provider + model + dimension 查询 embedding_cache；
3. 如果命中：
   - 反序列化 embedding_json；
   - 返回 cacheHit = true；
   - 不调用 EmbeddingProvider。
4. 如果未命中：
   - 调用 EmbeddingProvider；
   - 校验 provider / model / dimension；
   - 序列化 embedding；
   - 写入 embedding_cache；
   - 返回 cacheHit = false。
```

---

### 6.2 返回结果

建议返回一个明确结果对象，例如：

```text
CachedEmbeddingResult
```

字段可以包括：

- embedding；
- provider；
- model；
- dimension；
- chunkHash；
- cacheHit。

这样后续可以扩展：

- cache hit count；
- cache miss count；
- saved provider calls；
- cache hit rate；
- per-provider cache statistics。

---

### 6.3 provider / model / dimension 校验

cache miss 时，系统会调用 `EmbeddingProvider` 生成 embedding。

生成后不能只保存向量，还要确认：

```text
返回的 provider / model / dimension 是否与预期一致。
```

如果 provider 返回了不同 model 或 dimension，不能直接写 cache。否则会污染 cache 和 VectorStore。

这点对于 local-worker 很重要，因为 Java 侧配置和 Python worker 实际模型必须一致。

---

## 7. 与 VectorStore 的关系

EmbeddingCache 和 VectorStore 职责不同。

### 7.1 EmbeddingCache 负责什么

EmbeddingCache 负责：

```text
避免重复生成 embedding。
```

它回答的问题是：

> 这段文本在这个 provider / model / dimension 下，之前是否已经生成过 embedding？

---

### 7.2 VectorStore 负责什么

VectorStore 负责：

```text
保存 chunk 对应的向量索引，并支持 search。
```

它回答的问题是：

> 给定 query embedding，哪些 chunk 最相似？

---

### 7.3 cache hit 后为什么仍然要 upsert VectorStore

即使命中 cache，也不能跳过 VectorStore upsert。

原因是：

```text
同一个 embedding 可以被复用；
但当前 chunk 可能属于新的 documentId / chunkId。
```

例如两个不同文档里都有同一句话：

```text
Please read the following terms carefully.
```

这句话的 embedding 可以复用，但两个文档都需要在 VectorStore 中有自己的 chunk 记录，否则检索结果无法返回正确的 documentId / chunkId。

所以正确逻辑是：

```text
cache hit：跳过 embedding provider 调用
但仍然 upsert 当前 chunk 到 VectorStore
```

---

## 8. 并发写入设计

V2.6.8 需要考虑并发场景：

```text
线程 A 查询 cache miss
线程 B 查询 cache miss
线程 A 调用 provider 生成 embedding
线程 B 也调用 provider 生成 embedding
线程 A insert cache 成功
线程 B insert cache 触发 unique key 冲突
```

本版本采用最小可靠方案：

```text
1. 数据库 unique key 保证最终不会有重复 cache 记录；
2. 如果 insert 时发生唯一键冲突，重新查询已有 cache；
3. 如果查询成功，复用已有 cache；
4. 不引入 Redis lock；
5. 不引入分布式锁；
6. 不做 singleflight。
```

这个方案的特点是：

- 简单；
- 数据库保证最终一致；
- 不过度设计；
- 能避免重复记录；
- 但不能完全避免并发下重复 provider 调用。

如果后续需要进一步优化，可以做：

```text
in-flight request coalescing
singleflight
distributed lock
```

但这些不是 V2.6.8 范围。

---

## 9. 为什么不缓存失败结果

V2.6.8 只缓存成功 embedding。

如果 provider 调用失败，例如：

- 外部 API 超时；
- Local Worker 不可用；
- 模型加载失败；
- 返回非法 dimension；
- 网络异常；

系统不应该写入 cache。

原因：

- 失败可能是临时的；
- 缓存失败会导致后续请求直接失败；
- 失败缓存需要 TTL、重试策略和错误分类；
- 会扩大版本范围。

所以本版本原则是：

```text
只缓存成功结果，不缓存失败结果。
```

---

## 10. 为什么不做文档级 dedup

文档级 dedup 是另一个更复杂的问题。

它需要回答：

- 同一个 document 是否允许重复上传？
- 如果两个用户上传同一份文档，是复用 documentId，还是各自有一份？
- 权限如何隔离？
- 文档 metadata 不同但内容相同怎么办？
- 删除一个用户的文档是否影响其他用户？
- documentHash 如何定义？
- 是否需要版本管理？

这些都属于产品语义，不只是技术去重。

V2.6.8 只做 chunk-level embedding cache：

> 即使不同文档里出现相同 chunk，也可以复用 embedding，但 document / chunk 的业务归属仍然保留。

---

## 11. 为什么不做语义 dedup

语义 dedup 指：

```text
两个文本不完全相同，但意思很接近，所以复用 embedding。
```

例如：

```text
AI Task Orchestrator is a backend system.
The orchestrator is a backend service for AI tasks.
```

这两个句子语义接近，但不应该直接共用完全相同的 embedding。

原因：

- 语义相似不等于文本相同；
- 共用 embedding 会损失细节；
- 需要相似度阈值；
- 阈值选择会影响正确性；
- 需要额外评估；
- 容易引入不可解释错误。

所以 V2.6.8 只做 exact content hash dedup（精确内容 hash 去重）。

## 12. 测试设计

### 12.1 Hash 测试

需要覆盖：

- 同一文本 hash 一致；
- 不同文本 hash 不同；
- 换行符 normalize 行为符合预期；
- null / blank 行为符合项目规范。

这些测试保证 chunkHash 稳定。

---

### 12.2 Repository 测试

需要覆盖：

- 能按 `chunkHash + provider + model + dimension` 查询；
- 相同 key 不能重复插入；
- 相同 chunkHash 但不同 provider 可以共存；
- 相同 chunkHash 但不同 model 可以共存；
- 相同 chunkHash 但不同 dimension 可以共存；
- embedding_json 可以正确保存和读取。

这些测试证明 cache key 的唯一性和隔离性正确。

---

### 12.3 Service 测试

需要覆盖：

- cache miss 时调用 EmbeddingProvider；
- cache miss 后写入 cache；
- cache hit 时不调用 EmbeddingProvider；
- 相同文本 + 相同 provider / model / dimension 第二次命中；
- 相同文本 + 不同 provider / model / dimension 不命中；
- provider 抛异常时不写 cache；
- 并发 unique key 冲突时可以重新读取已有 cache；
- 返回结果包含 cacheHit、chunkHash、provider、model、dimension。

这些测试证明 cache 的核心行为正确。

---

### 12.4 Ingestion / Integration 测试

如果项目结构允许，还应该测试：

- 第一次处理 chunk 是 cache miss；
- 第二次处理相同 chunk 是 cache hit；
- cache hit 时 provider 不再被调用；
- cache hit 后 VectorStore 仍然执行 upsert；
- 默认测试不依赖外部 API、worker、Qdrant、Docker。

这类测试能证明 cache 没有破坏原有 ingestion 链路。

## 15. 产品化意义

### 15.1 成本控制

如果使用外部 embedding API，cache hit 可以减少 provider 调用次数，从而降低成本。

未来如果接入 usage billing，可以进一步统计：

```text
savedEmbeddingCalls
savedTokens
estimatedCostSaved
```

---

### 15.2 延迟优化

cache hit 时可以跳过 provider 调用，减少文档入库耗时。

尤其是 Local Worker 使用 CPU 时，重复 embedding 会明显拖慢 ingestion。

---

### 15.3 大规模 ingestion 基础

在企业文档、合同、FAQ、代码文档中，重复 chunk 很常见。

Embedding cache 能让系统在大量文档导入时更稳定、更可控。

---

### 15.4 检索正确性

通过 provider / model / dimension 隔离 cache，可以避免不同向量空间混用。

这比简单做一个 `Map<text, vector>` 更严谨。

---

### 15.5 商业化扩展

后续可以根据 cache 指标设计：

- 免费用户更激进复用；
- VIP 用户使用更高质量模型；
- 企业用户使用私有 Local Worker；
- 按 provider 调用次数计费；
- 按 cache 命中率优化成本。

## 16. 简历 bullet

### 16.1 成本控制表达

> 为 RAG ingestion 链路设计并实现 Embedding Cache / Dedup，基于 `chunkHash + provider + model + dimension` 构建持久化缓存，避免相同 embedding space 下重复生成向量，降低外部 embedding API 调用成本和本地 worker 推理开销。

### 16.2 正确性表达

> 将 provider、model、dimension 纳入 embedding cache key，避免不同 embedding space 的向量错误复用；即使命中 cache，仍保留当前 documentId / chunkId 并执行 VectorStore upsert，保证检索索引与业务归属正确。

### 16.3 工程可靠性表达

> 使用数据库唯一约束保证 embedding cache key 唯一，并在并发 insert 冲突时重新读取已有 cache，避免重复缓存记录；同时只缓存成功 embedding，不缓存 provider 失败结果，降低错误状态污染风险。

## 17. 面试高强度追问与回答

### Q1：为什么 cache key 不只用 chunkHash？

A：chunkHash 只能说明文本内容一样，但不能说明向量空间一样。同一段文本用不同 provider、model 或 dimension 生成的 embedding 不能混用，所以 cache key 必须包含 `chunkHash + provider + model + dimension`。

---

### Q2：命中 cache 后为什么还要写 VectorStore？

A：EmbeddingCache 只复用向量生成结果，但当前 chunk 可能属于新的 documentId / chunkId。VectorStore 需要保存当前 chunk 的检索索引和业务元数据，所以即使命中 cache，也仍然要 upsert 当前 chunk。

---

### Q3：为什么不用 Qdrant 直接当 embedding cache？

A：Qdrant 的职责是向量检索索引，EmbeddingCache 的职责是复用 embedding 生成结果。二者职责不同。独立 cache 更适合用 provider / model / dimension 做生成结果复用，也不会把成本优化逻辑耦合进检索引擎。

---

### Q4：如果两个线程同时 cache miss 怎么办？

A：本版本用数据库唯一约束保证最终只有一条 cache 记录。并发 insert 冲突时，失败线程重新查询已有 cache 并复用。这个方案不能完全避免重复 provider 调用，但可以避免重复缓存记录，且实现简单可靠。

---

### Q5：为什么不做分布式锁？

A：分布式锁可以减少并发下重复 provider 调用，但会引入 Redis、锁过期、死锁、重试等复杂问题。当前版本先用数据库唯一约束解决数据正确性，后续如果 provider 成本很高，再考虑 singleflight 或分布式锁。

---

### Q6：为什么不缓存失败结果？

A：provider 失败可能是临时问题，例如网络超时或 worker 未启动。如果缓存失败结果，后续请求可能直接失败。失败缓存需要 TTL 和错误分类策略，当前版本只缓存成功 embedding，边界更清晰。

---

### Q7：为什么不做文档级 dedup？

A：文档级 dedup 会牵涉产品语义，例如同一文档不同用户上传是否共享、权限如何隔离、metadata 不同怎么办、删除是否互相影响。V2.6.8 只做 chunk-level embedding cache，不改变文档业务语义。

---

### Q8：为什么不做语义 dedup？

A：语义相似不等于文本相同。两个文本很像，也不应该直接共用同一个 embedding。语义 dedup 需要阈值、评估和正确性控制，容易引入不可解释错误，所以当前只做精确内容 hash 去重。

---

### Q9：embedding_json 用 JSON 存会不会低效？

A：确实不是最高效的存储方式，但当前 embedding_cache 不是向量检索引擎，只是生成结果缓存。JSON 实现简单、易调试、不引入额外依赖，适合当前项目阶段。真正检索仍然交给 VectorStore / Qdrant。

---

### Q10：换模型后旧 cache 怎么办？

A：旧 cache 不能直接复用，因为 cache key 包含 model 和 dimension。换模型后会形成新的 cache key，系统会重新生成 embedding。旧 cache 可以保留，也可以后续通过清理策略删除。

---

## 19. 本版本在项目中的位置

V2.6.8 位于 RAG ingestion 成本优化层。

它之前：

```text
V2.5.x：EmbeddingProvider 与 Local Worker
V2.6.x：VectorStore、Qdrant、Benchmark、Manual Verification
```

它解决：

```text
相同文本是否重复生成 embedding
cache key 如何保证向量空间正确性
cache hit 是否跳过 provider 调用
cache hit 后是否仍然写 VectorStore
并发写入如何避免重复 cache 记录
```

它之后：

```text
Embedding Cache Metrics
Retrieval Policy / VIP Search
Rerank
Hybrid Search
Agent Runtime
```

本版本最大的意义是：

> 让 AI Task Orchestrator 的 RAG ingestion 链路开始具备成本意识和重复计算控制能力，而不是只追求“能生成 embedding”。
