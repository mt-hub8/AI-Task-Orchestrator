# V2.6.9 Embedding Cache Metrics

## 1. 版本摘要

V2.6.9 的目标是为 V2.6.8 已经实现的 **Embedding Cache / Dedup（Embedding 缓存与去重）** 增加可观测指标。

V2.6.8 解决的是：

> 相同 chunk 内容，在相同 provider、model、dimension 下，不重复生成 embedding。

V2.6.9 进一步解决的是：

> 这个 cache 到底有没有发挥作用？命中了多少次？少调用了多少次 EmbeddingProvider？不同 embedding space 的缓存表现是否不同？

也就是说，V2.6.8 是机制，V2.6.9 是证据。

本版本的核心能力包括：

- 记录 cache hit count（缓存命中次数）；
- 记录 cache miss count（缓存未命中次数）；
- 记录 write count（缓存写入次数）；
- 记录 conflict count（并发唯一键冲突次数）；
- 记录 provider call count（实际 provider 调用次数）；
- 记录 saved provider call count（节省的 provider 调用次数）；
- 按 provider / model / dimension 聚合指标；
- 计算 hit rate（命中率）；
- 提供只读查询 API：`GET /embedding-cache/metrics`。

这个版本的核心价值是：

> 让 Embedding Cache 从“做了缓存”升级为“能量化缓存效果”。

---

## 2. 为什么现在做

Embedding Cache 是一个成本优化能力。如果没有指标，只能说：

```text
我做了 cache。
```

但面试官很可能继续问：

```text
cache 命中率是多少？
它到底减少了多少 provider 调用？
如果换模型，命中率会不会变化？
local-worker 和 openai-compatible 的命中情况能不能分开看？
如果 cache hit rate 很低，你怎么发现？
```

如果没有 metrics，就很难回答这些问题。

V2.6.9 的意义是让系统能够回答：

```text
这个 embedding space 下：
hitCount = 80
missCount = 20
hitRate = 80%
savedProviderCallCount = 80
providerCallCount = 20
```

这比单纯说“我做了缓存”更有工程说服力。

---

## 3. 本版本解决的问题

### 3.1 cache 是否真的生效

V2.6.8 证明 cache 逻辑正确：

- cache hit 时不调用 EmbeddingProvider；
- cache miss 时调用 EmbeddingProvider；
- 不同 provider / model / dimension 不混用；
- cache hit 后仍然 upsert VectorStore。

但它没有直接告诉我们：

> 真实 ingestion 过程中 hit 了多少次，miss 了多少次。

V2.6.9 增加指标后，可以直接观察 cache 是否真的发挥作用。

---

### 3.2 成本优化缺少量化依据

EmbeddingProvider 调用一般是有成本的：

- 外部 API：可能按 token 或请求计费；
- Local Worker：消耗 CPU / GPU；
- 大批量文档：增加 ingestion 时间；
- 多用户场景：影响资源调度和成本核算。

一次 cache hit 可以近似理解为：

> 节省了一次 EmbeddingProvider 调用。

所以 `savedProviderCallCount` 是一个非常直观的成本优化指标。

第一版不需要估算美元成本，只统计节省的 provider 调用次数即可。

---

### 3.3 全局 hit rate 不够，需要按 embedding space 聚合

Embedding cache 的正确性边界是：

```text
provider + model + dimension
```

不同 provider、不同 model、不同 dimension 对应不同 embedding space。

所以 metrics 也不能只做一个全局统计。

错误做法：

```text
totalHitCount
totalMissCount
```

这种全局指标会掩盖关键信息。

例如：

```text
local-worker / all-MiniLM-L6-v2 / 384  命中率 80%
openai-compatible / text-embedding-3-small / 1536  命中率 10%
```

如果只看全局 hit rate，就看不出哪个模型或 provider 的重复内容更多，也无法分析成本优化重点。

所以 V2.6.9 按：

```text
provider + model + dimension
```

聚合指标。

---

### 3.4 后续 quota / billing / VIP Search 需要成本基础

未来如果继续做：

- Retrieval Policy；
- VIP Search；
- quota（配额）；
- rate limit（限流）；
- billing（计费）；
- provider cost accounting（provider 成本核算）；

就需要知道：

- 实际调用 provider 多少次；
- cache 节省了多少调用；
- 哪个 provider 成本高；
- 哪个模型命中率高；
- 不同用户计划是否应该使用不同模型或不同缓存策略。

V2.6.9 先把 provider / model / dimension 级别的统计做出来，为后续商业化能力铺路。

## 4. 核心设计

### 4.1 新增 embedding_cache_metric 表

V2.6.9 采用数据库持久化指标，而不是只打日志，也不是直接接 Prometheus。

推荐表：

```text
embedding_cache_metric
```

核心字段：

| 字段 | 中文含义 | 说明 |
|---|---|---|
| `provider` | provider（提供方） | 例如 mock / openai-compatible / local-worker |
| `model` | model（模型） | embedding 模型名称 |
| `dimension` | dimension（维度） | embedding 向量维度 |
| `hit_count` | 命中次数 | cache hit 的次数 |
| `miss_count` | 未命中次数 | cache miss 的次数 |
| `write_count` | 写入次数 | 成功写入 embedding_cache 的次数 |
| `conflict_count` | 冲突次数 | 并发写 cache 时唯一键冲突次数 |
| `provider_call_count` | provider 调用次数 | 实际调用 EmbeddingProvider 的次数 |
| `saved_provider_call_count` | 节省调用次数 | 因 cache hit 而节省的 provider 调用次数 |
| `created_at` | 创建时间 | 指标记录首次创建时间 |
| `updated_at` | 更新时间 | 指标记录最近更新时间 |

唯一约束：

```text
unique(provider, model, dimension)
```

这个唯一约束保证每个 embedding space 只有一条聚合指标记录。

---

### 4.2 为什么不用日志作为主方案

只打日志虽然简单，但问题明显：

- 不方便聚合；
- 不方便测试；
- 不方便通过 API 查询；
- 服务重启后很难整理；
- 不能直接给面试官展示；
- 不利于后续成本统计。

日志适合排查细节，但不适合作为核心指标存储。

---

### 4.3 为什么不直接接 Prometheus

Prometheus / Grafana 是更生产化的监控方案，但 V2.6.9 不直接接入，原因是：

1. 当前项目还没有完整 actuator / micrometer / dashboard 体系；
2. 引入 Prometheus 会扩大版本范围；
3. 当前最重要的是证明指标语义和业务链路正确；
4. 数据库聚合指标更容易测试；
5. 面试项目中先用 MySQL 做可查询证据更直观。

后续如果要生产化，可以把这些指标迁移为：

```text
embedding_cache_hit_total
embedding_cache_miss_total
embedding_cache_saved_provider_call_total
```

再接入 Prometheus。

## 5. 指标语义

### 5.1 hit_count（缓存命中次数）

当 `EmbeddingCacheService` 查到已有 embedding cache 时：

```text
hit_count + 1
```

中文含义：

> 本次请求没有重新生成 embedding，而是复用了已有向量。

---

### 5.2 miss_count（缓存未命中次数）

当 `EmbeddingCacheService` 没查到已有 cache 时：

```text
miss_count + 1
```

中文含义：

> 本次请求没有可复用的 embedding，需要调用 EmbeddingProvider 生成。

---

### 5.3 provider_call_count（provider 调用次数）

cache miss 后需要调用 EmbeddingProvider：

```text
provider_call_count + 1
```

它统计的是：

> 系统真实调用 embedding provider 的次数。

这对成本核算很重要。

---

### 5.4 saved_provider_call_count（节省 provider 调用次数）

cache hit 时可以认为：

```text
saved_provider_call_count + 1
```

因为本来需要调用 provider，但由于 cache 命中，系统直接复用了已有 embedding。

第一版可以认为：

```text
savedProviderCallCount ≈ cacheHitCount
```

它不是美元成本，但可以作为后续成本估算的基础。

---

### 5.5 write_count（缓存写入次数）

cache miss 后，如果 provider 成功返回 embedding，并成功写入 `embedding_cache`：

```text
write_count + 1
```

如果 provider 失败，则不增加 write_count。

原因：

> 只有真正写入成功的 embedding，才算缓存写入。

---

### 5.6 conflict_count（并发唯一键冲突次数）

如果两个线程同时 cache miss，并发写入同一个 cache key，可能出现：

```text
线程 A insert embedding_cache 成功
线程 B insert embedding_cache 触发 unique key 冲突
线程 B 重新读取已有 cache
```

这时：

```text
conflict_count + 1
```

它的含义是：

> 并发下重复写入被数据库唯一约束拦截。

注意：

```text
conflict_count 不等于 cache hit。
```

因为发生冲突的线程可能已经调用过 provider，所以它不一定节省了 provider 调用。

---

## 6. hitRate（命中率）怎么计算

Hit Rate（命中率）的公式：

```text
hitRate = hitCount / (hitCount + missCount)
```

中文解释：

> 所有 cache 查询请求中，有多少比例命中了已有 embedding。

例子：

```text
hitCount = 80
missCount = 20
hitRate = 80 / (80 + 20) = 0.8
```

也就是：

```text
命中率 = 80%
```

如果：

```text
hitCount + missCount = 0
```

第一版建议返回：

```text
hitRate = 0
```

而不是 null。这样 API 调用方和测试都更简单。

## 7. 指标更新方式

### 7.1 为什么不使用 find -> save

计数器更新如果使用：

```text
find metric
count = count + 1
save metric
```

在并发下可能出现 lost update（更新丢失）：

```text
线程 A 读取 hit_count = 10
线程 B 读取 hit_count = 10
线程 A 保存 11
线程 B 保存 11
```

最终应该是 12，但实际变成 11。

所以 V2.6.9 推荐使用数据库原子递增：

```sql
UPDATE embedding_cache_metric
SET hit_count = hit_count + 1,
    saved_provider_call_count = saved_provider_call_count + 1,
    updated_at = CURRENT_TIMESTAMP
WHERE provider = ?
  AND model = ?
  AND dimension = ?
```

这样数据库会保证计数递增是原子的。

---

### 7.2 metric row 不存在怎么办

如果 update 返回 0，说明当前 provider / model / dimension 的 metric row 还不存在。

处理方式：

```text
1. 尝试 insert 初始 metric row；
2. 如果 insert 成功，本次计数完成；
3. 如果 insert 发生唯一键冲突，说明其他线程刚创建；
4. 再 retry update。
```

这种模式能处理并发创建 metric row 的问题。

---

### 7.3 metrics 更新失败怎么办

Metrics 是可观测性辅助，不是主业务。

如果 metrics 更新失败，原则是：

```text
记录 warn 日志，但不让 embedding 主流程失败。
```

原因：

- Embedding cache / ingestion 是主流程；
- Metrics 是辅助证据；
- 不能因为指标写入失败导致文档入库失败；
- 否则系统可用性会被可观测性组件反向拖垮。

这体现了工程上的主次关系。

## 8. 与 V2.6.8 的集成

V2.6.9 不是重新写 cache，而是在 V2.6.8 的 `EmbeddingCacheService` 中接入 metrics。

典型接入点：

### 8.1 cache hit

```text
命中 embedding_cache
返回 cached embedding
记录 hit_count + saved_provider_call_count
```

### 8.2 cache miss

```text
未命中 embedding_cache
记录 miss_count + provider_call_count
调用 EmbeddingProvider
```

### 8.3 cache write

```text
provider 成功返回
写入 embedding_cache 成功
记录 write_count
```

### 8.4 cache conflict

```text
写入 embedding_cache 发生 unique key 冲突
重新读取已有 cache
记录 conflict_count
```

### 8.5 provider 失败

```text
provider 抛异常
不写 embedding_cache
不记录 write_count
```

这保证 V2.6.8 的缓存语义不被破坏，只是在关键节点增加指标。

## 9. 只读 API 设计

V2.6.9 可以新增只读 API：

```http
GET /embedding-cache/metrics
```

返回示例：

```json
{
  "items": [
    {
      "provider": "local-worker",
      "model": "sentence-transformers/all-MiniLM-L6-v2",
      "dimension": 384,
      "hitCount": 80,
      "missCount": 20,
      "writeCount": 20,
      "conflictCount": 0,
      "providerCallCount": 20,
      "savedProviderCallCount": 80,
      "hitRate": 0.8
    }
  ]
}
```

无数据时返回：

```json
{
  "items": []
}
```

这个 API 只做查询，不做管理操作。

明确不做：

- 不清空 cache；
- 不 reset metrics；
- 不删除记录；
- 不做 admin 权限；
- 不做 billing；
- 不做 quota。

真实生产环境中，这类指标 API 应该放在 internal / admin 路径下，并加权限控制。当前项目先保持简单，用于 demo 和面试展示。

## 10. 为什么不做 per-user metrics

V2.6.9 不做 per-user（按用户）统计。

原因是当前项目还没有完整的：

- 用户体系；
- 租户体系；
- 权限模型；
- billing 模型；
- userId / tenantId 在 ingestion 链路中的统一传递。

如果现在强行做 per-user metrics，会引入很多产品语义：

```text
同一份文档多个用户上传，cache 节省应该算谁的？
企业租户之间是否共享 cache？
用户删除数据后 cache 是否要删除？
```

这些问题适合后续 quota / billing 阶段解决。

V2.6.9 只按 provider / model / dimension 聚合，是更克制的选择。

---

## 11. 为什么不做真实美元成本估算

本版本不做：

```text
estimatedCostSavedUsd
```

原因：

- 不同 provider 价格不同；
- 同一 provider 的价格可能变化；
- 不同模型价格不同；
- embedding API 可能按 token 计费；
- local-worker 没有直接 API 成本；
- 需要记录文本 token 数；
- 成本估算容易失真。

所以 V2.6.9 只统计：

```text
savedProviderCallCount
```

后续如果接入 provider pricing table（价格表）和 token 统计，再做成本估算。

### 16.1 可观测性表达

> 为 Embedding Cache 增加 metrics 统计能力，按 provider / model / dimension 聚合记录 cache hit、miss、write、conflict、provider call 和 saved provider call，并计算 hit rate，使 embedding 成本优化从机制实现升级为可观测证据。

### 16.2 成本优化表达

> 通过 `savedProviderCallCount` 量化 cache hit 为系统节省的 EmbeddingProvider 调用次数，为后续 quota、billing、provider cost accounting 和 VIP Search 策略提供基础数据。

### 16.3 工程可靠性表达

> 使用数据库原子递增更新 embedding cache metrics，避免并发下 find-save 导致计数丢失；metrics 更新失败时只记录日志，不影响主 embedding / ingestion 流程，保证可观测性组件不破坏主链路可用性。

## 17. 面试高强度追问与回答

### Q1：V2.6.8 已经有 cache，为什么还要做 metrics？

A：V2.6.8 只能证明 cache 机制存在，V2.6.9 能证明 cache 产生了多少效果。没有 metrics，就无法量化命中率、miss 次数、节省 provider 调用次数，也无法判断 cache 是否值得继续优化。

---

### Q2：为什么按 provider / model / dimension 聚合？

A：因为 embedding space 由 provider、model、dimension 决定。不同 embedding space 的 cache 不能混用，指标也应该分开看。否则全局 hit rate 会掩盖模型切换或维度变化带来的影响。

---

### Q3：savedProviderCallCount 怎么理解？

A：一次 cache hit 表示系统复用了已有 embedding，没有调用 EmbeddingProvider。所以第一版可以把 savedProviderCallCount 理解为 cache 命中帮系统节省的 provider 调用次数。

---

### Q4：为什么不直接估算节省了多少钱？

A：真实成本取决于 provider、model、token 数、价格表和计费规则。当前项目还没有 token-level cost accounting 和 provider pricing table，所以先统计 savedProviderCallCount，后续再扩展 estimated cost saved。

---

### Q5：为什么不直接接 Prometheus？

A：Prometheus 更适合生产监控，但会引入 actuator、micrometer、部署和 dashboard 配置。当前阶段先用数据库聚合指标，容易测试、容易通过 API 查询，也符合项目当前复杂度。

---

### Q6：为什么 metrics 更新失败不影响主流程？

A：embedding / ingestion 是主流程，metrics 是辅助可观测性。如果指标更新失败就让文档入库失败，会降低系统可用性。所以第一版应该记录 warn 日志，但不中断主流程。

---

### Q7：为什么不用 find -> save 更新计数？

A：find -> save 在并发下可能丢失更新。例如两个线程都读到 hit_count = 10，然后都保存 11，最终少加了一次。数据库原子递增可以避免这个问题。

---

### Q8：为什么不做 per-user metrics？

A：当前项目还没有完整用户体系和租户体系。per-user metrics 会牵涉权限、计费、数据归属和删除语义，适合后续 quota / billing 阶段做。V2.6.9 先按 embedding space 聚合。

---

### Q9：为什么只做只读 API，不做 reset？

A：reset / clear 属于管理操作，需要 admin 权限和审计。当前项目没有权限体系，所以只做只读查询，避免引入安全和产品边界问题。

---

### Q10：cache hit rate 低怎么办？

A：先检查 chunking 是否稳定、文本 normalize 是否变化、provider / model / dimension 是否切换、文档是否真的重复、cache 表是否被清空。如果数据本身重复度低，hit rate 低也是正常现象。

```text
V2.6.8：Embedding Cache / Dedup
```

```text
cache hit / miss 如何统计
```

它之后：

```text
Retrieval Policy / VIP Search
Quota / Rate Limit
Cost Accounting
Rerank
```
