# V2.4.2 Evaluation Dataset Seed 面试文档

## 0. 版本摘要

V2.4.2 为 Retrieval Evaluation Harness 增加一套最小、稳定、可复现的 query-evidence benchmark seed，包括固定 markdown corpus、benchmark JSON、expectedEvidenceIds 和 seed validation test。

本版本不是新的 RAG 功能，不是 production benchmark management system，不做 Fixed vs Adaptive Chunking Comparison，不做 evidenceId -> chunkId mapper，不新增生产 API，不新增数据库表，不持久化 evaluation result，也不接真实 Embedding、Vector DB 或 Rerank。

一句话定位：

V2.4 和 V2.4.1 解决“能不能计算 retrieval metrics”，V2.4.2 解决“有没有稳定、可复现的评测输入”。

## 1. 本版本解决的问题

V2.4 / V2.4.1 已经有 Retrieval Evaluation API 和指标体系，但如果没有固定 benchmark 输入，评测仍然容易变成临时 Postman 请求和人工感觉。

当前缺口包括：

- 没有固定 corpus；
- 没有固定 query；
- 没有固定 expected evidence；
- 没有 benchmark case 文件；
- 不同版本无法复用同一套评测输入；
- 后续无法稳定比较 chunking、embedding、rerank、vector DB 策略；
- 面试时无法说明 expected evidence 是如何构造的。

V2.4.2 解决的是评测输入标准化问题：

- 用固定 markdown corpus 作为被检索语料；
- 用 benchmark JSON 记录 query cases；
- 用 expectedEvidenceIds 表达 ground truth evidence；
- 用 validation test 保证 seed 文件结构正确；
- 为后续 Fixed vs Adaptive Chunking Comparison 和 Real Embedding 对比提供固定输入。

## 2. 为什么现在做 V2.4.2

依赖关系上，V2.2 已经能检索 TopK chunks，V2.3 已经能基于检索结果做 RAG Answer，V2.4 已经能计算 retrieval metrics，V2.4.1 又补齐了 Precision@K 和 NDCG@K。到这个点，评测框架和指标都已经具备，下一步自然是补一套稳定 benchmark seed 作为后续对比实验输入。

为什么不更早做：

- 没有 retrieval evaluation API 前，benchmark seed 没有评估入口；
- 没有完整 metrics 前，seed 的评测输出结构还不稳定；
- 过早设计 benchmark 容易和后续 API / metrics 字段反复对齐。

为什么不更晚做：

- 如果直接接真实 Embedding、Vector DB 或 Rerank，就没有固定 benchmark 判断升级是否有效；
- 后续所有检索策略优化都会缺少统一基准；
- 没有固定输入时，评测结果很容易变成“这次看起来还行”，而不是可回归的工程证据。

核心表达：

先有 evaluation harness，再补 metrics，再建立 seed dataset，之后再做策略对比和真实模型升级。

## 3. 方案选择

### 方案 A：继续靠手工请求测试

做法：

- 手动上传文档；
- 手动查 chunks；
- 手动拼 expectedChunkIds；
- 手动调用 `/evaluations/retrieval`。

优点：

- 最快；
- 不需要新增 seed 文件；
- 不需要新增测试。

缺点：

- 不可复现；
- 不可自动校验；
- 不适合后续对比实验；
- chunkId 依赖数据库环境；
- 面试展示时不够专业，很难说明 ground truth 的来源。

结论：不推荐。

### 方案 B：在 src/test/resources 中放 seed corpus + benchmark JSON

做法：

- `retrieval-corpus-v1.md` 保存固定语料；
- `retrieval-benchmark-v1.json` 保存 query cases 和 expectedEvidenceIds；
- `RetrievalBenchmarkSeedTest` 校验文件结构和 marker 对应关系。

优点：

- 边界小；
- 不改生产 API；
- 不新增数据库；
- 测试稳定；
- 可复用；
- 适合后续 Fixed vs Adaptive、Real Embedding、Rerank 对比；
- 适合面试展示 evaluation dataset 设计意识。

缺点：

- 还不是正式 benchmark 管理系统；
- 不保存 evaluation run；
- expectedEvidenceIds 还不能直接被 V2.4 API 使用；
- 还需要后续 mapper 转换成 expectedChunkIds。

结论：当前推荐方案。

### 方案 C：新增生产级 Benchmark Dataset API

方向：

- `POST /evaluation-datasets`；
- `POST /evaluation-datasets/{id}/cases`；
- `POST /evaluation-runs`。

优点：

- 更产品化；
- 可以支持企业用户上传 benchmark；
- 可以保存 benchmark cases。

缺点：

- 需要新表；
- 需要新 API；
- 需要权限和所有权设计；
- 需要 evaluation run persistence；
- 当前阶段过重。

结论：暂时不选。

### 方案 D：直接做 evaluation_dataset / evaluation_case / evaluation_result 表

优点：

- 长期架构完整；
- 适合企业 benchmark 管理；
- 适合 dashboard 和历史报告。

缺点：

- 当前还没有真实 Embedding；
- 还没有 evaluation dashboard；
- 还没有多轮 run 对比需求；
- 过早固化 schema 风险高。

结论：后续 Evaluation Persistence 阶段再做。

## 4. Seed Corpus 设计

文件：

`src/test/resources/evaluation/retrieval-corpus-v1.md`

这是固定 markdown corpus，用于作为 Retrieval Evaluation 的 seed 语料。它覆盖 AI Task Orchestrator 当前项目中的核心技术点：

- Transactional Outbox；
- Atomic Task Claim；
- Task Attempt；
- RAG Retrieval Evaluation；
- RAG Answer Citation。

corpus 中使用 evidence marker 稳定标识 ground truth evidence：

- `[EVIDENCE:outbox-reason]`
- `[EVIDENCE:atomic-claim-purpose]`
- `[EVIDENCE:task-attempt-metadata]`
- `[EVIDENCE:retrieval-evaluation-purpose]`
- `[EVIDENCE:rag-citation-purpose]`

marker 的作用：

- 稳定标识某段证据；
- 不依赖数据库；
- 不依赖 chunkId；
- 后续可以通过 marker 找到所在 chunk，再映射成 expectedChunkIds。

当前 corpus 的局限：

- 语料较小；
- 人工构造；
- query 较直接；
- 没有 hard negative；
- 不能代表真实企业知识库；
- 作用是 seed / regression baseline，不是企业级 benchmark。

## 5. Benchmark JSON 设计

文件：

`src/test/resources/evaluation/retrieval-benchmark-v1.json`

字段：

- `datasetId`：benchmark 标识，目前为 `retrieval-benchmark-v1`；
- `description`：说明这是 AI Task Orchestrator retrieval evaluation 的 seed benchmark；
- `corpusFile`：指向 `evaluation/retrieval-corpus-v1.md`；
- `topKValues`：当前为 `[1, 3, 5]`；
- `cases`：query-evidence 评测样本列表。

case 字段：

- `caseId`：case 唯一标识，例如 `outbox-001`、`claim-001`；
- `query`：评测问题；
- `expectedEvidenceIds`：期望命中的 evidence marker id。

当前 benchmark cases 包括：

- `outbox-001`
- `claim-001`
- `attempt-001`
- `retrieval-eval-001`
- `citation-001`

必须强调的是，benchmark JSON 不写 expectedChunkIds，不写数据库 ID，也不写评测结果数值。它是 authoring format，用来稳定描述“哪个 query 应该命中哪段 evidence”。

## 6. 为什么用 expectedEvidenceIds，而不是 expectedChunkIds

V2.4 API 使用 expectedChunkIds 是合理的，因为 evaluation 计算最终要比较 retrieved chunkId 和 expected chunkId。

但 seed 文件不应该写死 chunkId。原因是：

- chunkId 是数据库生成的；
- chunkId 依赖上传顺序；
- 不同环境 chunkId 可能不同；
- 重新上传 corpus 后 chunkId 可能变化；
- 静态 benchmark 文件如果写 chunkId，会不可复现。

因此 V2.4.2 使用 expectedEvidenceIds 作为 benchmark authoring format。

后续路线：

```text
expectedEvidenceIds
-> 在 corpus 中找到 [EVIDENCE:id]
-> 上传 / chunk 后定位包含 marker 的 chunk
-> 转换成 expectedChunkIds
-> 调用 POST /evaluations/retrieval
```

当前版本只做 seed 和 validation，不做 mapper。

## 7. 为什么只做 Seed Validation Test

本版本新增 `RetrievalBenchmarkSeedTest`，目标是验证 seed dataset 自身质量，而不是跑完整 evaluation。

测试覆盖方向：

- corpus 文件存在；
- benchmark JSON 存在；
- JSON 可解析；
- datasetId 不为空；
- description 不为空；
- corpusFile 不为空；
- corpusFile 指向的文件存在；
- topKValues 合法；
- cases 不为空；
- caseId 唯一；
- query 不为空；
- expectedEvidenceIds 不为空；
- expectedEvidenceIds 不重复；
- corpus 中 evidence marker 唯一；
- 每个 expectedEvidenceId 都能在 corpus 中找到对应 marker；
- benchmark JSON 不使用 expectedChunkIds；
- corpus 不写死 chunkId / chunk_id / expectedChunkIds。

为什么不在本版本调用 `/evaluations/retrieval`：

- 当前 benchmark 还是 expectedEvidenceIds；
- V2.4 API 需要 expectedChunkIds；
- 中间还缺 mapper；
- 本版本边界是 seed validation；
- 完整 runner 后续单独做。

## 8. 为什么不做 evidenceId -> chunkId mapper

mapper 需要把 marker 和实际 chunking 结果关联起来。它涉及：

- 文档上传；
- chunking strategy；
- marker 所在 chunk 定位；
- chunkId 获取；
- benchmark case 转换；
- 调用 evaluation API。

这些已经超出 seed validation 范围。

当前不做的原因：

- 版本边界更清晰；
- 避免把 Seed、Runner、Chunking Comparison 混在一个版本；
- 后续 V2.4.3 / Benchmark Runner 阶段再实现更合适。

## 9. 为什么不做 Fixed vs Adaptive Chunking Comparison

V2.4.2 的目标是提供稳定 benchmark 输入，不是做策略对比。

Fixed vs Adaptive Comparison 需要：

- 用同一 corpus 生成两套 chunks；
- 将 expectedEvidenceIds 映射到各自 chunkId；
- 对两套 chunking strategy 分别检索；
- 计算 metrics；
- 比较结果。

这应该作为 V2.4.3 单独版本处理。V2.4.2 先把“输入是否稳定、标注是否一致、文件是否可校验”做好。

## 10. 为什么不做数据库持久化

当前不新增：

- evaluation_dataset；
- evaluation_case；
- evaluation_expected_evidence；
- evaluation_run；
- evaluation_result；
- retrieval_eval_case；
- retrieval_eval_result。

原因：

- 当前只是 test resource seed；
- 还没有 benchmark 管理需求；
- 还没有 evaluation history；
- 还没有 dashboard；
- 还没有企业用户上传 benchmark；
- 过早持久化会增加 schema 复杂度。

后续如果产品化，可以再设计 persistence，把 dataset、case、expected evidence、run 和 result 拆成明确的数据模型。

## 11. 核心流程

```text
新增 retrieval-corpus-v1.md
-> 在 corpus 中标记 [EVIDENCE:id]
-> 新增 retrieval-benchmark-v1.json
-> cases 使用 query + expectedEvidenceIds
-> RetrievalBenchmarkSeedTest 读取 corpus 和 benchmark JSON
-> 解析 corpus evidence markers
-> 校验 benchmark 中的 expectedEvidenceIds 都能找到 marker
-> 保证 seed dataset 可复现、可校验
```

这个流程不调用生产 API，不访问数据库，不计算 Recall / NDCG 具体结果，不做真实 retrieval，只验证 seed 输入有效。

## 16. 工程权衡

### 16.1 为什么 seed corpus 用 markdown

项目已经有 markdown / document chunking 基础，markdown 能表达标题结构，适合 Adaptive Chunking，也方便插入 evidence marker。对面试和回归测试来说，markdown 比数据库 dump 更直观，也更容易审查。

### 16.2 为什么 benchmark 用 JSON

JSON 结构清晰、易解析、适合测试读取，后续也容易扩展 topKValues、cases、expectedAnswer、tags、difficulty 等字段。它不依赖数据库，也不会把 benchmark 绑定到某个本地环境。

### 16.3 为什么不写 expectedChunkIds

chunkId 不稳定，不同数据库环境、上传顺序或重新导入都会导致 chunkId 变化。marker 更适合静态 benchmark，后续再通过 mapper 转换成 expectedChunkIds。

### 16.4 为什么只做 validation，不做 runner

Seed 和 Runner 是两个层次。当前先保证 benchmark 数据本身有效；Runner 涉及 chunking、mapping、evaluation API 调用和结果聚合，后续单独实现更清晰。

### 16.5 为什么不现在扩大 corpus

当前目标是建立 seed 结构。过早做大型 corpus 会拖慢版本，而且没有 mapper / runner / persistence 时，大型 corpus 的工程收益有限。后续可以在 larger benchmark corpus 阶段扩展 hard negatives、multi-hop 和 real queries。

## 21. 面试高强度追问与回答

### Q1：为什么已经有 Evaluation API，还要 Seed Dataset？

Evaluation API 解决的是“给我一组 query 和 expectedChunkIds，我能算指标”。Seed Dataset 解决的是“这些 query 和 expected evidence 从哪里来，是否稳定，是否能复现”。没有 seed dataset，评测很容易变成临时请求，今天用一批 chunkId，明天换一个数据库就失效。V2.4.2 把评测输入沉淀成文件，并用测试校验 marker 和 case 的一致性，为后续策略对比提供固定基准。

### Q2：seed corpus 的作用是什么？

seed corpus 是最小可复现语料。它不追求覆盖真实业务所有复杂情况，而是保证项目里有一份固定输入，可以用来验证 benchmark 结构、evidence 标注方式和后续 mapper / runner。它的价值类似回归测试中的 fixture：小，但稳定；简单，但能防止评测链路没有标准输入。

### Q3：企业真实 RAG evaluation corpus 一般怎么构造？

企业 corpus 通常来自真实知识库、FAQ、客服工单、SOP、产品文档、故障手册和用户搜索日志。构造 benchmark 时，一般会选取真实 query，标注 expected evidence passages，有时还会标注 expected answer、hard negatives、query difficulty 和 graded relevance。高风险行业还会加入合规问题、合同条款、政策解释等场景。和当前 seed 不同，企业 benchmark 重点是代表真实用户分布和业务风险。

### Q4：为什么当前 seed corpus 比较简单？

因为 V2.4.2 的目标是建立 seed 结构，不是一次性做企业 benchmark。当前还没有 mapper、runner、persistence、真实 embedding 和 rerank。如果这时直接构造大型 corpus，会把数据集建设、评测运行和策略对比混在一起。先做一个小而稳定的 seed，可以让后续每一步都有明确依赖。

### Q5：为什么用 markdown 作为 corpus？

项目已经支持 `.md` 文档上传和 Adaptive Chunking，markdown 能表达标题层级，也方便插入 `[EVIDENCE:id]` marker。相比数据库脚本或纯 JSON 文本，markdown 更贴近实际文档形态，面试时也更容易解释“marker 在文档中如何定位 evidence”。

### Q6：为什么 benchmark 用 JSON？

JSON 适合表达结构化 benchmark：datasetId、corpusFile、topKValues、cases、query、expectedEvidenceIds 都能清晰建模。它易于 Jackson 解析，也方便后续扩展字段，比如 tags、difficulty、expectedAnswer、expectedEvidenceSpans。更重要的是，JSON 不依赖数据库，适合作为 test resources 中的可复现输入。

### Q7：为什么用 expectedEvidenceIds，而不是 expectedChunkIds？

expectedChunkIds 是 evaluation API 的计算输入，但它不适合作为静态 benchmark authoring format。chunkId 来自数据库，依赖上传顺序和环境。expectedEvidenceIds 则是语义稳定的标注：它指向 corpus 中的 `[EVIDENCE:id]` marker。这样即使重新上传文档，marker 仍然稳定，后续只需要 mapper 把 marker 所在 chunk 转成实际 chunkId。

### Q8：expectedEvidenceIds 后续如何变成 expectedChunkIds？

后续可以做一个 mapper：先读取 benchmark 的 expectedEvidenceIds，再在 corpus 中找到对应 marker；上传 corpus 并完成 chunking 后，扫描 document_chunk，找出包含 marker 的 chunk，得到实际 chunkId；最后把 benchmark case 转换成 V2.4 API 需要的 expectedChunkIds。这样 seed 文件保持稳定，实际评测时再绑定当前环境的 chunkId。

### Q9：为什么不现在做 evidenceId -> chunkId mapper？

mapper 会引入上传文档、chunking、marker 定位、chunk 查询、case 转换和 evaluation API 调用，已经接近 benchmark runner。V2.4.2 的边界是 seed validation，先保证 benchmark 文件本身正确。把 mapper 放到后续版本，可以让 Seed、Mapper、Runner、Comparison 每一步都可验收。

### Q10：为什么不现在做 Fixed vs Adaptive Chunking Comparison？

Fixed vs Adaptive 对比需要同一 corpus 生成两套 chunks，并分别把 expectedEvidenceIds 映射成各自 chunkId，再运行检索评测和指标比较。这比 seed dataset 多了一整层实验执行逻辑。当前先保证有稳定输入，下一步再用这套输入做策略对比，工程边界更清楚。

### Q11：为什么不持久化 evaluation dataset？

当前数据集只是 test resource seed，还没有企业用户、dataset ownership、run history、dashboard 或 report 需求。过早持久化会引入多张表和 schema 设计成本。更稳妥的路线是先用文件形式验证 authoring format，等 runner 和真实 benchmark 成熟后，再设计 evaluation_dataset、evaluation_case、evaluation_run、evaluation_result。

### Q12：为什么不直接做企业级 benchmark 管理系统？

企业级 benchmark 管理系统需要权限、多租户、数据集版本、case 审核、标注工作流、run history、报告和 dashboard。当前项目仍处于 RAG evaluation baseline 阶段，直接做这套会偏离主线。V2.4.2 用 test resources 做 seed，是在不扩大系统复杂度的前提下，先把 benchmark 思路落地。

### Q13：这个 seed dataset 有什么局限？

它很小，query 比较直接，evidence marker 人工明确，没有 hard negative、multi-hop、真实用户 query、graded relevance 或 expectedAnswer。它不能证明当前检索效果好，也不能代表企业真实知识库。它的定位是 regression seed 和后续评测链路的起点。

### Q14：后续如何扩展成真实企业 benchmark？

可以从线上搜索日志、客服工单、FAQ 和专家标注中收集真实 query，再把 expected evidence 标注到具体 passage 或 chunk。之后增加 hard negatives、query difficulty、业务标签、expectedAnswer、graded relevance 和 citation labels。等这些数据稳定后，再加入 persistence、dashboard、定期评测和质量报告。

### Q15：VIP / Enterprise 用户怎么用这类 benchmark？

企业用户可以上传自己的知识库和 query-evidence benchmark，用它评估当前检索策略是否能找到正确证据。VIP 能进一步比较不同 embedding、chunking、rerank、topK 策略，还可以做周期性回归测试，发现某次知识库更新或模型切换后检索质量是否下降。

### Q16：如果数据量扩大 10 倍，benchmark evaluation 的瓶颈在哪里？

瓶颈会从 seed validation 转向真实 evaluation runner：embedding 计算、vector search、rerank、数据库读取、结果聚合和报告生成都会变重。如果每个 case 都要跑 maxK search，case 数量扩大后需要批处理、并发控制、缓存、任务化和结果持久化。当前 V2.4.2 还没有进入这些运行时瓶颈，因为它只做文件结构校验。

### Q17：如何避免 benchmark 过拟合？

需要把 benchmark 分层：一部分作为开发集，用来调策略；另一部分作为 holdout / regression set，不参与调参。还要引入真实用户 query、hard negatives、不同难度问题和定期更新机制。否则策略可能只对 seed corpus 表现好，但对真实业务 query 无效。

### Q18：如何从线上失败案例构造 evaluation cases？

可以收集用户搜索无结果、点击后仍追问、人工客服介入、用户点踩、RAG answer 被判错等失败样本。然后由专家标注正确 evidence，形成 query + expectedEvidenceIds / expectedEvidenceSpans。这样 benchmark 会逐渐从人工 demo seed 进化成真实 failure-driven evaluation dataset。
