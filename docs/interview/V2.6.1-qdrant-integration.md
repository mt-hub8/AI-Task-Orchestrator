# V2.6.1 Qdrant Integration

## 1. 版本摘要

V2.6.1 的目标是在 V2.6 已完成的 `VectorStore`（向量存储）抽象之下，新增一个真实外部 Vector DB（向量数据库）实现：`QdrantVectorStore`。

本版本完成的核心能力是：

- 在项目自有 `VectorStore` 接口下接入 Qdrant；
- 保留 `ExactCosineVectorStore` 作为默认 baseline（基线）；
- 只有显式配置 `app.vector-store.provider=qdrant` 时才启用 Qdrant；
- Qdrant payload（元数据）中保存 `chunkId`、`documentId`、`content`、`provider`、`model`、`dimension`、`metadata`；
- 查询时继续使用 `provider / model / dimension` 作为 correctness boundary（正确性边界），避免不同 embedding space（向量空间）混搜；
- 默认 Maven 测试不启动 Docker、不连接真实 Qdrant、不访问外部网络；
- 为后续 V2.6.2 exact search vs Qdrant ANN（近似最近邻检索）benchmark 做准备。

本版本不是 benchmark 版本，不声称 Qdrant 提升了 Recall@K（召回率@K）、NDCG@K（归一化折损累计增益@K）或 latency（延迟）。

---

## 2. 本版本解决的问题

V2.6 已经完成了 `VectorStore` 抽象，但默认实现仍然是 `ExactCosineVectorStore`。这在小规模数据和测试环境中很合适，但真实 RAG（检索增强生成）系统通常需要面对更多文档 chunk、更高查询并发和更复杂 metadata filter（元数据过滤）。

如果长期只依赖 exact cosine search（精确余弦检索），会有几个问题：

1. 数据量变大后，每次查询都要遍历候选向量，延迟会随规模上升；
2. 向量检索能力仍然停留在 Java 应用内部，缺少真实外部 Vector DB 集成经验；
3. 后续无法验证 `VectorStore` 抽象是否能承载真实外部检索系统；
4. 无法为 ANN（近似最近邻检索）、payload filter（元数据过滤）、collection 管理、VIP 检索策略打基础。

Qdrant 是一个面向向量相似度搜索的开源向量数据库，支持向量、payload 和过滤查询，适合 RAG 场景中的语义检索和 metadata filter 组合。

所以 V2.6.1 解决的真实问题是：

> 从“抽象上可替换的向量检索层”，推进到“可以接入真实外部 Vector DB 的向量检索基础设施”。

## 3. 为什么现在做 V2.6.1

现在做 Qdrant Integration 的前置条件已经具备：

- V2.4.x 已经完成 Retrieval Evaluation Harness；
- V2.4.2 / V2.4.4 已经有 query-evidence benchmark seed 和 evidence mapper；
- V2.5 已经完成 `EmbeddingProvider` 抽象；
- V2.5.1 已经具备 embedding provider benchmark comparison；
- V2.5.2 已经支持 local embedding worker；
- V2.6 已经完成 `VectorStore` 抽象和 `ExactCosineVectorStore` baseline。

因此，V2.6.1 的位置刚好在：

```text
EmbeddingProvider 可替换
        ↓
VectorStore 已抽象
        ↓
接入第一个真实外部 VectorStore：Qdrant
```

如果没有 V2.6 就直接接 Qdrant，业务代码会直接依赖具体 Qdrant API，后续替换 pgvector、Milvus、Elasticsearch 会更困难。Spring AI 自身也有独立的 VectorStore 抽象，并提供 Qdrant VectorStore 支持，这说明“用统一抽象屏蔽具体向量数据库”是成熟工程实践。

## 4. 方案选择

本版本选择：

> 基于 Qdrant REST API + Spring `RestClient` 实现 `QdrantVectorStore`，不直接引入 Qdrant 官方 Java client，也不直接使用 Spring AI Qdrant VectorStore。

### 4.1 为什么选择 Qdrant

当前项目主库是 MySQL。pgvector 更适合 PostgreSQL 主库体系，如果为了向量检索单独引入 PostgreSQL，会变成 MySQL + PostgreSQL 双数据库形态。Qdrant 作为独立 Vector DB，与 MySQL 主库分工更自然：

- MySQL 保存业务主数据；
- Qdrant 保存向量和检索 payload；
- Java 应用通过 `VectorStore` 抽象屏蔽底层实现。

Qdrant 也支持 payload 和 payload index（元数据索引），后续可以支撑 `documentId`、`tenantId`、`userPlan`、`provider`、`model`、`dimension` 等过滤字段。Qdrant 官方文档明确说明，payload index 可以用于提升过滤搜索效率。

### 4.2 为什么先用 REST API

Qdrant 官方文档说明，Qdrant 支持 REST API 和 gRPC，并建议首次使用或原型阶段优先使用 REST API；gRPC 更适合性能优化阶段。

因此 V2.6.1 选择 REST API 是克制的：

- 不新增复杂依赖；
- HTTP 请求更容易调试；
- fake client 测试更容易做；
- 适合验证第一版外部 VectorStore 接入。

### 4.3 为什么不直接用 Qdrant 官方 Java Client

Qdrant 官方 Java client 是后续生产候选，但 V2.6.1 的目标不是追求客户端封装最完整，而是验证自有 `VectorStore` 抽象能否承载真实外部 Vector DB。直接引入官方 client 会增加额外依赖和测试复杂度。

### 4.4 为什么不直接用 Spring AI Qdrant VectorStore

Spring AI Qdrant VectorStore 已经提供成熟封装，但本项目已经自建了 `EmbeddingProvider` 和 `VectorStore` 抽象。直接套用 Spring AI 的实现会让项目从“自建 AI infra 边界”变成“调框架”，不利于体现底层工程设计能力。

### 4.5 为什么不继续只保留 exact

V2.6 已经完成抽象，V2.6.1 需要用真实外部实现验证这个抽象。否则项目只能证明“设计了接口”，不能证明接口能承载真实 Vector DB。

## 5. 核心架构设计

V2.6.1 后的检索层结构是：

```text
Document / Query
    ↓
EmbeddingProvider
    ↓
VectorStore
    ├── ExactCosineVectorStore
    └── QdrantVectorStore
            ↓
      QdrantVectorStoreClient
            ↓
      RestClientQdrantVectorStoreClient
            ↓
          Qdrant
```

### 5.1 `QdrantVectorStore`

职责：

- 实现项目自有 `VectorStore` 接口；
- 将 `VectorStoreDocument` 转成 Qdrant point；
- 将 `VectorSearchRequest` 转成 Qdrant search request；
- 将 Qdrant search response 转成 `VectorSearchResult`；
- 保证 provider / model / dimension 过滤；
- 不直接暴露 Qdrant API 给上层 service。

### 5.2 `QdrantVectorStoreClient`

职责：

- 屏蔽 Qdrant REST 通信细节；
- 提供 create collection、upsert、search、delete 等内部方法；
- 让 `QdrantVectorStore` 可以使用 fake client 做默认测试。

### 5.3 `RestClientQdrantVectorStoreClient`

职责：

- 使用 Spring `RestClient` 调 Qdrant REST API；
- 处理 base-url、api-key、timeout；
- 将 HTTP 异常转换为统一异常。

### 5.4 `QdrantPayloadMapper`

职责：

- 在项目内部 `VectorStoreDocument` 和 Qdrant point / payload 之间转换；
- 保证 `chunkId`、`documentId`、`content`、`provider`、`model`、`dimension`、`metadata` 正确保存和读取。

## 6. Qdrant Collection 设计

本版本建议一个 collection 对应一个 embedding space（向量空间）。

embedding space 由以下字段共同决定：

- provider；
- model；
- dimension。

Qdrant collection 创建时需要指定向量维度和距离度量。Qdrant 支持通过 collection 管理向量配置，并在 collection 中存储 points。

本版本使用：

```text
collection-name = app.vector-store.qdrant.collection-name
distance = Cosine
dimension = 当前 embedding dimension
```

虽然 Qdrant 支持更复杂的 named vectors（命名向量）等能力，但 V2.6.1 不做，因为本版本目标是最小可用 QdrantVectorStore，不引入多向量空间管理复杂度。

---

## 7. Qdrant Payload 设计

Qdrant point 的 payload 需要保存：

```json
{
  "chunkId": 123,
  "documentId": 10,
  "content": "chunk text",
  "provider": "local-worker",
  "model": "sentence-transformers/all-MiniLM-L6-v2",
  "dimension": 384,
  "metadata": {
    "source": "document-upload"
  }
}
```

字段意义：

| 字段 | 作用 |
|---|---|
| `chunkId` | 映射回业务 chunk |
| `documentId` | 支持按文档过滤和删除 |
| `content` | 返回 search result 时可直接使用 |
| `provider` | 防止 mock / openai / local-worker 混搜 |
| `model` | 防止不同 embedding 模型混搜 |
| `dimension` | 防止不同维度向量混搜 |
| `metadata` | 为后续 tenantId、userPlan、documentType、namespace 等扩展 |

Qdrant 的 point 可以包含向量和可选 JSON payload，并支持基于 payload 的过滤查询。

---

## 8. provider / model / dimension 正确性边界

本版本延续 V2.5 和 V2.6 的核心原则：

> 不同 provider、不同 model、不同 dimension 的向量不能混搜。

原因是：

即使两个 embedding 都是 float 数组，只要它们来自不同模型或不同维度，就可能位于不同语义空间。直接混合计算相似度，结果没有可靠语义意义。

所以 Qdrant search 时必须带上：

```text
provider == request.provider
model == request.model
dimension == request.dimension
```

这不是为了字段完整，而是为了保证 retrieval correctness（检索正确性）。

## 9. 配置设计

新增或扩展配置：

```properties
app.vector-store.provider=exact

app.vector-store.qdrant.base-url=http://127.0.0.1:6333
app.vector-store.qdrant.collection-name=ai_task_orchestrator_chunks
app.vector-store.qdrant.api-key=
app.vector-store.qdrant.timeout-ms=3000
app.vector-store.qdrant.initialize-collection=false
```

配置含义：

| 配置 | 说明 |
|---|---|
| `app.vector-store.provider=exact` | 默认仍然使用 exact baseline |
| `app.vector-store.provider=qdrant` | 显式启用 Qdrant |
| `base-url` | Qdrant REST API 地址 |
| `collection-name` | 当前向量集合名称 |
| `api-key` | 本地环境可以为空 |
| `timeout-ms` | 防止外部服务卡死拖垮应用 |
| `initialize-collection` | 是否由应用尝试创建 collection |

本地 Qdrant 通常暴露 HTTP API 端口 `6333`，gRPC 端口 `6334`。

## 12. 核心流程

### 12.1 Upsert 流程

```text
Document chunk
    ↓
EmbeddingProvider 生成 embedding
    ↓
VectorStoreDocument
    ↓
QdrantPayloadMapper 转成 Qdrant point
    ↓
QdrantVectorStoreClient.upsertPoints
    ↓
Qdrant collection
```

### 12.2 Search 流程

```text
query text
    ↓
EmbeddingProvider 生成 query embedding
    ↓
VectorSearchRequest
    ↓
QdrantVectorStore 构造 filter
    ↓
QdrantVectorStoreClient.searchPoints
    ↓
Qdrant scored points
    ↓
QdrantPayloadMapper 转成 VectorSearchResult
    ↓
上层 search response / retrieval evaluation
```

### 12.3 Delete By DocumentId 流程

```text
documentId
    ↓
QdrantVectorStore.deleteByDocumentId
    ↓
构造 documentId payload filter
    ↓
Qdrant points delete
```

注意：

本版本的 Qdrant 删除只处理向量库中的 point，不代表删除 MySQL 业务数据。

## 17. 产品化 / VIP / 成本策略

V2.6.1 不实现 VIP，但它为后续检索能力分层打基础。

### 17.1 普通用户

普通用户后续可以使用：

- 较低 topK；
- 共享 collection；
- 较低查询并发；
- 较短文档保留时间；
- 基础 embedding 模型；
- 基础 VectorStore 策略。

### 17.2 VIP 用户

VIP 用户后续可以使用：

- 更高 topK；
- 更强 embedding 模型；
- Qdrant 专属 collection / namespace；
- 更高查询并发；
- 更低延迟配置；
- payload index 优化；
- rerank；
- hybrid search；
- 更长文档保留时间。

### 17.3 成本来源

接入 Qdrant 后，成本不只来自 LLM 或 embedding，还包括：

- Qdrant 存储成本；
- ANN 索引内存成本；
- payload index 成本；
- 查询 QPS 成本；
- collection 维护成本；
- 数据同步成本；
- 备份和恢复成本。

Qdrant 的 payload index 可以提升过滤搜索效率，但也意味着额外索引和资源成本。

## 21. 面试高强度追问与回答

### Q1：为什么 V2.6.1 选择 Qdrant？

A1：因为项目当前主库是 MySQL，而 pgvector 更适合 PostgreSQL 主库体系。Qdrant 作为独立 Vector DB，和 MySQL 的职责边界更清晰：MySQL 存业务主数据，Qdrant 存向量和 payload，Java 应用通过 VectorStore 抽象屏蔽底层实现。Qdrant 也支持 payload filter，适合 RAG 后续按 documentId、tenantId、provider、model、dimension 等字段过滤。

### Q2：为什么不用 pgvector？

A2：pgvector 很适合 PostgreSQL 体系项目。如果主库已经是 PostgreSQL，用 pgvector 可以减少基础设施数量。但本项目当前主库是 MySQL，如果为了向量检索单独引入 PostgreSQL，会形成 MySQL + PostgreSQL 双数据库形态。第一版外部 Vector DB 接入选择 Qdrant 更自然。

### Q3：为什么不用 Spring AI Qdrant VectorStore？

A3：Spring AI 的实现成熟，但本项目已经自建 EmbeddingProvider 和 VectorStore 抽象。V2.6.1 的目标是验证自己的 RAG infrastructure 抽象能否承载真实外部 Vector DB，而不是直接套用框架。这样更能体现底层工程设计能力。

### Q4：为什么不用 Qdrant 官方 Java client？

A4：官方 Java client 是后续生产候选，但第一版我选择 REST，是为了减少依赖、便于调试、便于 fake client 测试。Qdrant 官方文档也建议首次使用或原型阶段优先 REST，gRPC 更适合性能优化阶段。

### Q5：为什么默认测试不启动 Qdrant？

A5：默认 CI 应该稳定、快速、可重复。真实 Qdrant 属于外部服务，应该放到 optional external test 或手工验收。默认测试只验证 mapping、filter、配置和 fake client 链路，避免因为 Docker 或网络问题影响核心单元测试。

### Q6：如何防止不同 embedding 模型混搜？

A6：payload 和 request 都保留 provider、model、dimension。搜索时必须带这些过滤条件。不同 provider、不同 model、不同 dimension 代表不同 embedding space，不能混合计算相似度。

### Q7：为什么一个 collection 对应一个 embedding space？

A7：collection 内的向量应具有一致的维度和距离度量。一个 embedding space 可以由 provider、model、dimension 定义。第一版采用一个 collection 对应一个 embedding space，是最简单、安全、可解释的方式。

### Q8：Qdrant 接入后是不是 Recall@K 一定更高？

A8：不是。Qdrant 主要解决规模、延迟和外部向量检索服务化问题，不保证召回质量天然更高。Recall@K、NDCG@K、MRR 和 latency 必须通过 V2.6.2 benchmark 验证。

### Q9：如果 Qdrant ANN Recall@K 下降怎么办？

A9：可以调高 topK，调整 HNSW / search 参数，引入 rerank，或者对 VIP 用户使用更高质量检索策略。exact baseline 必须保留，用于判断 ANN 的召回损失。

### Q10：metadata filter 为什么重要？

A10：企业 RAG 不只是语义相似度搜索，还必须按文档、租户、权限、用户套餐、模型版本过滤。Qdrant 的 payload filter 和 payload index 能支撑这类场景，但索引字段选择会影响成本和性能。

### Q11：为什么不在 V2.6.1 做 payload index 自动创建？

A11：payload index 属于 Qdrant schema 和性能优化。本版本目标是接入链路，不是 schema 优化。提前做 payload index 会增加初始化逻辑和测试复杂度。后续可以在 V2.6.2 或 V2.7 根据 benchmark 和过滤字段使用情况再做。

### Q12：为什么不在 V2.6.1 做 Rerank 或 Hybrid Search？

A12：Rerank 是二阶段排序，Hybrid Search 会引入 BM25、score fusion 和权重调节。这些都会改变检索结果，导致无法判断指标变化来自 Qdrant 召回还是重排 / 混合检索。V2.6.1 只控制一个变量：外部 Vector DB 接入。

## 22. 后续演进路线

### V2.6.2 Vector Search Benchmark: Exact vs Qdrant ANN

目标：

- 固定 corpus；
- 固定 benchmark queries；
- 固定 expectedEvidenceIds；
- 对比 ExactCosineVectorStore 和 QdrantVectorStore；
- 评估 Recall@K（召回率@K）、NDCG@K（归一化折损累计增益@K）、MRR（平均倒数排名）、latency（延迟）；
- 不编造指标提升。

### V2.7 Retrieval Policy & VIP Search

目标：

- 根据 userPlan 调整 topK；
- 普通用户和 VIP 用户检索策略分层；
- 引入成本、延迟和质量权衡。

### V2.8 Rerank

目标：

- 引入二阶段 reranker；
- 独立评估 rerank 对 Recall@K、NDCG@K、ContextPrecision@K 的影响；
- 不和 Vector DB 接入混在一个版本。

### V2.9 Generation Evaluation

目标：

- 从 retrieval evaluation 进入 generation evaluation；
- 评估 Faithfulness（事实一致性）、Answer Relevance（答案相关性）；
- 引入 LLM-as-a-judge（大模型裁判）时保持可解释和可回归。

## 24. 本版本在项目中的位置

V2.6.1 是 RAG 检索基础设施阶段的关键版本。

它的位置是：

```text
V2.4.x：Retrieval Evaluation
    ↓
V2.5.x：EmbeddingProvider
    ↓
V2.6：VectorStore 抽象
    ↓
V2.6.1：Qdrant 外部 Vector DB 接入
    ↓
V2.6.2：Exact vs Qdrant Benchmark
    ↓
V2.7：Retrieval Policy & VIP Search
    ↓
V2.8：Rerank
    ↓
V2.9：Generation Evaluation
```

本版本最大的价值不是“接了一个数据库”，而是：

> 在自研 RAG infrastructure 分层下，把向量检索从本地 exact baseline 扩展到真实外部 Vector DB，同时保持默认测试稳定、检索正确性边界清晰、后续 benchmark 可归因。
