# V2.6.4 API Error Response Standardization

## 1. 版本摘要

V2.6.4 的目标是补齐 AI Task Orchestrator 的 API 错误响应标准化能力。

在这个版本之前，项目已经具备任务执行、Outbox、RAG、Embedding、VectorStore、Qdrant 原型、Retrieval Evaluation、Benchmark 和 E2E Demo 等能力。但 API 层如果仍然依赖 Spring Boot 默认错误响应，调用方会遇到几个问题：

- 错误结构不稳定；
- 业务错误没有稳定 `code`；
- 前端或外部调用方无法根据错误类型做处理；
- Demo 出错时不容易定位问题；
- 后续做 VIP Search、quota、rate limit、provider failure 时错误类型会越来越混乱。

因此 V2.6.4 引入统一错误响应结构：

```json
{
  "timestamp": "2026-07-01T21:30:00.000Z",
  "status": 404,
  "code": "TASK_NOT_FOUND",
  "message": "任务不存在",
  "path": "/tasks/999",
  "traceId": "..."
}
```

本版本的核心价值是：

> 让 API 从“能返回错误”升级为“错误可被调用方稳定消费、可定位、可扩展”。

---

## 2. 本版本解决的问题

### 2.1 Spring Boot 默认错误响应不适合业务系统

Spring Boot 默认错误响应通常只能表达 HTTP 层面的失败，例如：

```json
{
  "timestamp": "...",
  "status": 500,
  "error": "Internal Server Error",
  "path": "/tasks/999"
}
```

这种结构的问题是：

- 没有稳定业务错误码；
- 不适合前端根据错误类型处理；
- 不适合后续企业级 API 对接；
- 可能暴露过多框架细节；
- 不利于面试展示后端 API 产品化能力。

V2.6.4 通过 `ApiErrorResponse` 统一错误响应模型。

---

### 2.2 HTTP Status 不足以表达业务语义

单独依赖 HTTP status 粒度不够。

例如同样是 `400 Bad Request`，可能代表：

- JSON 请求体格式错误；
- 参数校验失败；
- `topK` 非法；
- prompt 为空；
- documentId 格式非法。

同样是 `409 Conflict`，可能代表：

- task 当前状态不允许操作；
- Atomic Finalization 被拒绝；
- 任务已经进入终态；
- 并发状态冲突。

所以需要 `ErrorCode` 表达更稳定的业务语义。

---

### 2.3 缺少 traceId 不利于排障

用户或前端反馈：

```text
这个请求失败了
```

如果没有 traceId，服务端只能靠时间、路径、日志关键字猜测。

V2.6.4 采用轻量 traceId 策略：

- 优先读取请求头 `X-Request-Id`；
- 如果没有则生成 UUID；
- 错误响应中返回 `traceId`；
- 日志排查时可以用 traceId 关联请求。

这不是完整分布式 tracing，但已经能满足当前项目的 API 排障需求。

---

## 3. 核心设计

### 3.1 ApiErrorResponse

`ApiErrorResponse` 是统一错误响应 DTO。

建议字段：

| 字段 | 含义 |
|---|---|
| `timestamp` | 错误发生时间 |
| `status` | HTTP 状态码 |
| `code` | 稳定业务错误码 |
| `message` | 面向调用方的错误说明 |
| `path` | 请求路径 |
| `traceId` | 请求追踪标识 |

设计原则：

- 字段简单；
- 不暴露 Java 堆栈；
- 调用方可稳定解析；
- 后续可扩展 details / fieldErrors。

---

### 3.2 ErrorCode

`ErrorCode` 是业务错误码枚举。

首批错误码可以包括：

- `TASK_NOT_FOUND`
- `INVALID_TASK_STATUS`
- `TASK_FINALIZATION_REJECTED`
- `DOCUMENT_NOT_FOUND`
- `INVALID_REQUEST`
- `VALIDATION_ERROR`
- `EMBEDDING_PROVIDER_ERROR`
- `VECTOR_STORE_ERROR`
- `RETRIEVAL_EVALUATION_ERROR`
- `INTERNAL_ERROR`

这些错误码覆盖当前阶段最核心的错误来源：

- 任务系统；
- 文档系统；
- 参数校验；
- Embedding provider；
- VectorStore；
- Retrieval Evaluation；
- 兜底系统异常。

---

### 3.3 BusinessException

`BusinessException` 用来表达可预期的业务错误。

它通常包含：

- `ErrorCode`
- HTTP status
- message

例如：

```java
throw new BusinessException(
    ErrorCode.TASK_NOT_FOUND,
    HttpStatus.NOT_FOUND,
    "任务不存在"
);
```

或者使用静态工厂方法：

```java
throw BusinessException.notFound(ErrorCode.TASK_NOT_FOUND, "任务不存在");
```

核心思想：

> Service 层不应该随意抛出无法区分语义的 RuntimeException，而应该在关键业务边界使用稳定的业务异常。

---

### 3.4 GlobalExceptionHandler

`GlobalExceptionHandler` 负责把不同异常统一转换成 `ApiErrorResponse`。

需要处理的类型包括：

- `BusinessException`
- `IllegalArgumentException`
- `MethodArgumentNotValidException`
- `HttpMessageNotReadableException`
- not found 类异常
- `RuntimeException` 兜底

兜底异常应该返回：

```json
{
  "status": 500,
  "code": "INTERNAL_ERROR",
  "message": "系统内部错误",
  "path": "...",
  "traceId": "..."
}
```

不要暴露内部堆栈、SQL、类名、文件路径。

---

## 4. HTTP Status 与 ErrorCode 映射

| 场景 | HTTP Status | ErrorCode |
|---|---:|---|
| 参数校验失败 | 400 | `VALIDATION_ERROR` |
| JSON 请求体不可读 | 400 | `INVALID_REQUEST` |
| 请求参数非法 | 400 | `INVALID_REQUEST` |
| 任务不存在 | 404 | `TASK_NOT_FOUND` |
| 文档不存在 | 404 | `DOCUMENT_NOT_FOUND` |
| 当前任务状态不允许操作 | 409 | `INVALID_TASK_STATUS` |
| Atomic Finalization 被拒绝 | 409 | `TASK_FINALIZATION_REJECTED` |
| Embedding Provider 失败 | 502 或 500 | `EMBEDDING_PROVIDER_ERROR` |
| VectorStore 失败 | 502 或 500 | `VECTOR_STORE_ERROR` |
| Retrieval Evaluation 失败 | 500 | `RETRIEVAL_EVALUATION_ERROR` |
| 未知异常 | 500 | `INTERNAL_ERROR` |

这里的设计重点是：

> HTTP status 表达通用协议语义，ErrorCode 表达业务语义。

---

## 5. 为什么不改成功响应结构

V2.6.4 只统一错误响应，不改成功响应。

原因：

1. 成功响应当前已经被 demo、测试、文档依赖；
2. 改成功响应会扩大影响面；
3. 当前问题主要集中在错误结构不统一；
4. 成功响应是否需要 envelope wrapper 可以后续单独评估。

如果后续要统一成功响应，可以考虑：

```json
{
  "data": {},
  "traceId": "...",
  "timestamp": "..."
}
```

但这不是 V2.6.4 范围。

---

## 6. 为什么不直接使用 ProblemDetail

Spring Framework 6 支持 `ProblemDetail`，它接近 RFC 7807 风格，是标准化方案。

但当前项目优先采用自定义 `ApiErrorResponse`，原因是：

1. 当前更需要清晰业务 `code`；
2. 当前更需要中文或业务化 message；
3. `traceId` 是项目演示和排障的重要字段；
4. 自定义结构更便于面试解释；
5. 未来仍然可以迁移或对齐 ProblemDetail。

这个选择不是否定 ProblemDetail，而是当前阶段选择更直观、成本更低、业务表达更强的方案。

---

## 7. 为什么不一次性重写所有异常

项目已经有多个模块，如果一次性把所有异常全部替换为 `BusinessException`，容易引入回归风险。

V2.6.4 采用渐进式策略：

- 核心业务边界优先改为 `BusinessException`；
- 遗留 `IllegalArgumentException` 先统一转为 `INVALID_REQUEST`；
- 未预期异常由 `INTERNAL_ERROR` 兜底；
- 后续模块迭代时逐步收敛异常类型。

这种做法更符合工程渐进演进。

---

## 8. 测试设计

### 8.1 GlobalExceptionHandler 测试

应覆盖：

- `BusinessException` 能返回指定 status / code / message；
- `IllegalArgumentException` 返回 `INVALID_REQUEST`；
- `MethodArgumentNotValidException` 返回 `VALIDATION_ERROR`；
- `HttpMessageNotReadableException` 返回 `INVALID_REQUEST`；
- not found 类异常返回对应 not found code；
- `RuntimeException` 返回 `INTERNAL_ERROR`；
- 响应包含 `timestamp` / `status` / `code` / `message` / `path` / `traceId`；
- 请求头 `X-Request-Id` 存在时，响应 traceId 使用该值；
- 请求头不存在时，自动生成 traceId。

---

### 8.2 Controller 层测试

应覆盖真实 API，而不是为了测试新增生产 API：

- 查询不存在 task -> `TASK_NOT_FOUND`；
- 创建任务参数非法 -> `VALIDATION_ERROR` 或 `INVALID_REQUEST`；
- 查询不存在 document -> `DOCUMENT_NOT_FOUND`，如果当前有对应 API；
- 非法状态操作 -> `INVALID_TASK_STATUS` 或 `TASK_FINALIZATION_REJECTED`，如果当前有对应入口。

---

### 8.3 回归测试

必须确保：

- 正常创建任务不受影响；
- 正常查询任务不受影响；
- Task Execution 测试不受影响；
- Document / Embedding / VectorStore / Retrieval Evaluation 测试不受影响；
- 默认测试不依赖外部 API、Python worker、Qdrant、Docker、OpenAI key。

## 10. 当前局限

V2.6.4 只完成第一版 API 错误响应标准化，仍然有边界：

1. traceId 是轻量实现，不是完整分布式 tracing；
2. 未接入 MDC / OpenTelemetry；
3. 未做请求日志 filter；
4. 未做错误事件持久化；
5. 未做 OpenAPI 错误模型声明；
6. 未做 i18n 国际化；
7. 未统一成功响应 envelope；
8. 未做 quota / rate limit / auth 错误；
9. 部分遗留异常可能仍通过兜底 handler 转换。

这些都是后续可演进方向。

## 11. 产品化意义

### 11.1 对前端

前端可以根据 `code` 做稳定处理，例如：

- `TASK_NOT_FOUND`：展示“任务不存在”；
- `VALIDATION_ERROR`：高亮表单；
- `INVALID_TASK_STATUS`：提示当前状态不允许操作；
- `INTERNAL_ERROR`：提示用户稍后重试并上报 traceId。

---

### 11.2 对外部 API 调用方

调用方不需要解析不稳定的 Spring Boot 默认错误结构，而是可以稳定依赖：

```text
status + code + message + traceId
```

这对后续企业级 API 对接很重要。

---

### 11.3 对商业化能力

后续如果做 VIP Search / quota / billing / rate limit，可以自然扩展错误码：

- `QUOTA_EXCEEDED`
- `RATE_LIMITED`
- `INVALID_USER_PLAN`
- `PAYMENT_REQUIRED`
- `FEATURE_NOT_AVAILABLE`

V2.6.4 是这些能力的 API 协议基础。

---

### 11.4 对工程排障

traceId 让用户反馈和后端日志之间建立最小关联：

```text
用户：请求失败，traceId = abc-123
工程侧：搜索 traceId = abc-123 的日志
```

这比只看时间和路径可靠很多。

---

## 12. 简历 bullet

### 12.1 API 产品化表达

> 设计并实现统一 API 错误响应体系，新增 `ApiErrorResponse`、`ErrorCode`、`BusinessException` 和 `GlobalExceptionHandler`，将业务异常、参数校验异常、JSON 解析异常和系统兜底异常统一转换为稳定错误结构，提升 API 可消费性与可排障性。

### 12.2 面向调用方表达

> 为 AI Task Orchestrator 统一错误协议，响应中包含 `timestamp`、`status`、`code`、`message`、`path`、`traceId`，使前端和外部调用方能够基于稳定业务错误码处理 `TASK_NOT_FOUND`、`VALIDATION_ERROR`、`INVALID_TASK_STATUS` 等场景。

### 12.3 后续商业化铺垫表达

> 通过统一 ErrorCode 体系为后续 quota、rate limit、VIP Search、provider failure 和 vector store failure 等产品化错误场景预留扩展点，避免错误处理散落在各个 Controller 中。

## 13. 面试高强度追问与回答

### Q1：为什么要做统一错误响应？

A：因为默认错误响应不能稳定表达业务语义。统一错误响应能让前端和外部调用方根据 `code` 处理错误，也能避免暴露内部异常细节，并通过 `traceId` 支持排障。

---

### Q2：为什么要有 ErrorCode，HTTP status 不够吗？

A：HTTP status 是协议层语义，粒度不够。比如 400 可能是参数校验失败、JSON 格式错误或非法 topK；409 可能是任务状态冲突或终态写入被拒绝。ErrorCode 用来表达稳定业务语义。

---

### Q3：BusinessException 和 RuntimeException 有什么区别？

A：BusinessException 表达可预期的业务错误，包含 ErrorCode、HTTP status 和 message。RuntimeException 更泛，不能稳定表达业务语义，所以只适合作为兜底异常处理。

---

### Q4：为什么不把所有异常都变成 BusinessException？

A：一次性改造所有异常影响面大，容易引入回归。更合理的方式是先建立统一 handler，再把核心业务边界逐步替换为 BusinessException，遗留 IllegalArgumentException 先统一映射为 INVALID_REQUEST。

---

### Q5：为什么兜底异常不能返回真实 exception message？

A：因为真实异常信息可能包含内部类名、SQL、路径、第三方错误细节甚至敏感配置。对外应该返回稳定、克制的 `INTERNAL_ERROR`，详细信息留在服务端日志中。

---

### Q6：traceId 是怎么生成的？

A：优先读取请求头 `X-Request-Id`，如果调用方已经传入，则原样返回；如果没有，则服务端生成 UUID。这样既支持上游 trace 传递，也支持单体服务内的最小排障。

---

### Q7：这个 traceId 和 OpenTelemetry 有什么区别？

A：这是轻量 traceId，只用于错误响应和日志关联，不是完整链路追踪。OpenTelemetry 会涉及 trace/span、上下游传播、采样、collector 和可视化平台。当前版本只做最小可用能力。

---

### Q8：为什么不直接用 ProblemDetail？

A：ProblemDetail 是标准方案，但当前项目更需要业务 `code`、中文 message、traceId 和面试可解释性。自定义 ApiErrorResponse 更贴合当前阶段，未来可以再对齐 ProblemDetail 或 OpenAPI 错误模型。

---

### Q9：这个版本和后续 VIP Search 有什么关系？

A：VIP Search 后面会涉及 quota、userPlan、feature access、rate limit 等错误。如果没有统一错误协议，这些错误会分散在 Controller 或 Service 中。V2.6.4 先建立 ErrorCode 和统一响应，为后续商业化功能打基础。

---

### Q10：如何避免 Controller 里到处 try-catch？

A：通过 `GlobalExceptionHandler` 集中处理异常。Controller 保持业务入口职责，Service 抛出 BusinessException 或标准异常，统一 handler 负责转换成 ApiErrorResponse。

## 15. 本版本在项目中的位置

V2.6.4 是项目产品化 API 能力的基础版本。

它之前：

```text
V2.6.3：项目收敛、E2E Demo、Atomic Finalization
```

```text
错误结构统一
业务错误码稳定
traceId 可排障
兜底异常不泄露细节
```

它之后：

```text
Qdrant 手工验证
Local Worker 工程化
Retrieval Policy / VIP Search
Quota / Rate Limit
```

本版本最大的意义是：

> 让 AI Task Orchestrator 的 API 从工程内部可用，进一步接近外部调用方可稳定集成的产品化接口。
