# V2.4.3 Fixed vs Adaptive Chunking Comparison 面试文档

## 0. 版本摘要

V2.4.3 在 V2.4.2 Evaluation Dataset Seed 基础上，构建 Fixed Chunking 与 Adaptive Chunking 的 comparison baseline。

一句话定位：

用同一套 query-evidence benchmark，对比不同 chunking strategy 在 retrieval evaluation metrics 下的表现。

必须明确：

- 本版本是 chunking strategy comparison baseline；
- 本版本不是 production experiment platform；
- 本版本不新增生产 API；
- 本版本不新增数据库表；
- 本版本不持久化 comparison result；
- 本版本不接真实 Embedding；
- 本版本不接 Vector DB；
- 本版本不做 Rerank；
- 本版本不做 Faithfulness / Answer Relevance；
- 本版本不声称 Adaptive 一定优于 Fixed。

## 1. 本版本解决的问题

V2.1.1 已经实现 Adaptive Chunking，但之前更多是“功能存在”：系统能按标题、段落、语义边界和 overlap 切分文档。然而在 RAG 工程里，chunking 不是孤立功能，它最终要服务 retrieval quality。

真实问题是：

- Fixed Chunking 可能切断语义边界；
- Adaptive Chunking 尝试保留 Markdown heading、段落、空行等结构；
- 但 Adaptive 是否真的更好，不能凭感觉判断；
- 需要在同一 corpus、同一 query、同一 expected evidence、同一 metrics 下比较。

V2.4.3 解决：

- 复用 V2.4.2 的 fixed benchmark seed；
- 同时生成 Fixed / Adaptive 两套 chunks；
- 在两套 chunks 上使用相同 query cases；
- 计算相同 retrieval metrics；
- 得到 comparison baseline。

核心表达：

V2.4.3 不是为了证明 Adaptive 一定更好，而是为了建立可验证的 chunking strategy comparison pipeline。

## 2. 为什么现在做 V2.4.3

依赖关系已经具备：

- V2.4 有 Retrieval Evaluation Harness；
- V2.4.1 有 Precision@K / NDCG@K；
- V2.4.2 有稳定 query-evidence benchmark seed；
- 所以现在可以比较 chunking strategy。

为什么不直接进入 Real Embedding：

- 如果直接接真实 Embedding，效果变化无法判断来自 embedding、chunking、rerank 还是 benchmark 数据变化；
- 现在先固定 corpus / query / evidence / metrics，只改变 chunking strategy；
- 这体现工程实验意识：一次只改变一个变量。

为什么不更早做：

- 没有 evaluation harness 时无法计算指标；
- 没有 seed benchmark 时无法稳定比较；
- 没有 Precision@K / NDCG@K 时排序和精度指标不完整。

## 3. 方案选择

### 方案 A：不做对比，直接进入 Real Embedding

优点：

- 推进速度快；
- 更快接近真实 RAG；
- 可以尽早使用外部 embedding 模型。

缺点：

- Adaptive Chunking 的价值没有被验证；
- 后续效果变化无法归因；
- 面试时只能说“实现了 Adaptive Chunking”，不能说“评估过它”；
- evaluation harness 没有真正用于策略对比。

结论：不推荐作为当前下一步。

### 方案 B：只写文档说明 Adaptive 更好，不写 comparison

优点：

- 风险低；
- 不影响代码；
- 文档容易写。

缺点：

- 没有真实 comparison pipeline；
- 没有自动化测试；
- 没有工程可信度；
- 容易变成纸面设计。

结论：不推荐。

方向：

`POST /evaluations/chunking-comparison`

优点：

- 手工演示方便；
- 更像产品功能；
- 可直接返回 comparison result。

缺点：

- 当前没有 evaluation run persistence；
- 没有 strategy config；
- 没有 dataset 管理；
- 没有 dashboard；
- 过早暴露 production API 容易固化接口。

结论：暂时不选。

- 构造 comparison result；
- 不新增生产 API；

优点：

- 不新增数据库；

## 4. 核心流程

当前 `ChunkingStrategyComparisonTest` 的流程是：

```text
加载 retrieval-corpus-v1.md
-> 加载 retrieval-benchmark-v1.json
-> 解析 [EVIDENCE:id] marker
-> 用 Fixed Chunking 生成 fixed chunks
-> 用 Adaptive Chunking 生成 adaptive chunks
-> 对每种 strategy 定位 expectedEvidenceIds 所在 chunk
-> 对每个 query 执行 retrieval scoring
-> 按 score 排序得到 ranked chunks
-> 标记 retrieved chunks 是否 relevant
-> 计算 Recall@K
-> 计算 Precision@K
-> 计算 HitRate@K
-> 计算 MRR / RR@K
-> 计算 NDCG@K
-> 计算 ContextPrecision@K
-> 生成 Fixed / Adaptive 两组 evaluation result
-> 生成 comparison result
```

这个流程固定了：

- 同一 corpus；
- 同一 benchmark cases；
- 同一 topKValues；
- 同一 metrics；
- 同一 deterministic scorer。

唯一变化变量是 chunking strategy。

## 5. Fixed Chunking 与 Adaptive Chunking 的区别

### 5.1 Fixed Chunking

Fixed Chunking 按固定长度或固定窗口切分文本。当前测试中的 fixed chunking 是 test-level 实现，使用固定字符窗口，`FIXED_CHUNK_SIZE = 500`，并对 evidence marker 做了最小保护，避免 marker 被切断。

优点：

- 实现简单；
- 行为稳定；
- 可控性强；
- 对任意文本都能工作；
- 工程成本低。

缺点：

- 可能切断标题和正文；
- 可能切断一个完整语义段；
- 可能把 evidence 分散到不同 chunks；
- 对 Markdown / 结构化文档利用不足。

### 5.2 Adaptive Chunking

Adaptive Chunking 尝试利用文档结构进行切分，例如 Markdown heading、段落、空行、语义边界。当前测试中的 adaptive chunking 复用项目已有 `DocumentChunker`，不是在测试里另造一套 adaptive 逻辑。

优点：

- 更容易保留语义完整性；
- 更适合结构化文档；
- 理论上更有利于 RAG context 构造；
- 可能减少被切断的 evidence。

缺点：

- 实现复杂；
- 对文档格式敏感；
- 不保证所有场景都优于 Fixed；
- 如果结构解析不稳定，可能产生过大或过小 chunks。

### 5.3 核心判断

Adaptive Chunking 不是天然更好，必须通过 benchmark 验证。V2.4.3 的价值是把“比较它们”这件事工程化，而不是提前宣布结论。

## 6. expectedEvidenceIds 如何映射到 chunks

V2.4.2 benchmark 使用 expectedEvidenceIds，而不是 expectedChunkIds。V2.4.3 对每种 chunking strategy 分别做映射：

```text
expectedEvidenceIds
-> 在 corpus 中找到 [EVIDENCE:id]
-> 生成 chunks
-> 找到包含该 marker 的 chunk
-> 记录该 strategy 下对应的 chunk index / test chunk id
-> 用这些 chunk indexes 作为 relevant set
-> 对 retrieved chunks 计算 metrics
```

当前测试要求每个 evidence marker 在每种 strategy 下都必须落在 exactly one chunk 中。如果找不到，或者匹配多个 chunk，测试会失败。

同一个 evidenceId 在 Fixed 和 Adaptive strategy 下可能对应不同 chunk index。原因是：

- 两种 chunking strategy 的切分边界不同；
- 同一段 evidence 落入的 chunk 位置可能不同；
- 所以不能复用一套固定 chunkId；
- 必须按 strategy 分别映射。

## 7. Retrieval Scoring 设计

当前实现不接真实 embedding，也不访问数据库。

`ChunkingStrategyComparisonTest` 使用 deterministic test-level lexical overlap scorer：

- 对 query 和 chunk content 做简单 tokenization；
- 计算 query tokens 和 chunk tokens 的重叠数量；
- score 越高排序越靠前；
- score 相同时按 chunk index 稳定排序；
- 返回 ranked chunks，再交给 metrics calculator。

这个 scorer 的目标是打通 comparison pipeline，不是证明真实语义检索效果。它的优点是稳定、无外部依赖、无网络调用、无 API key、无数据库依赖，适合 test-level baseline。

必须明确：

当前 retrieval scoring 是 baseline，不代表真实线上 embedding retrieval。

## 8. Metrics 设计

本版本复用 V2.4 / V2.4.1 的 retrieval metrics，并通过 `RetrievalMetricsCalculator` 计算：

- Recall@K：正确 evidence 是否被找回；
- Precision@K：TopK 中 relevant chunk 的比例；
- HitRate@K：TopK 是否至少命中一个 relevant chunk；
- MRR / RR@K：第一个 relevant chunk 排名是否靠前；
- NDCG@K：整体排序质量；
- ContextPrecision@K：实际 context 中 relevant chunk 的比例。

本版本不重新定义 metrics，重点是把同一套 metrics 应用到不同 chunking strategy 上。

## 9. 为什么不强制断言 Adaptive 优于 Fixed

这是本版本最重要的工程克制。

原因：

- 当前 seed corpus 很小；
- 当前 query 比较直接；
- 当前还不是企业真实文档；
- 当前没有 hard negative；
- 当前没有真实 embedding；
- 当前使用 test-level retrieval scoring；
- Adaptive 在小样本上不一定稳定优于 Fixed；
- 强制断言 Adaptive > Fixed 会让测试脆弱；
- 工程上应该先验证 comparison pipeline，而不是编造提升结论。

测试应该断言：

- comparison pipeline 能跑通；
- 两种 strategy 都能生成 chunks；
- 两种 strategy 都能覆盖 expected evidence；
- 两种 strategy 都能计算 metrics；
- comparison result 同时包含 FIXED 和 ADAPTIVE。

不应该断言：

- adaptiveRecallAtK > fixedRecallAtK；
- adaptiveNdcgAtK > fixedNdcgAtK；
- Adaptive 显著提升检索效果。

当前测试也是这样做的：它验证结构完整、流程可运行、指标字段齐全，但不声明 Adaptive 指标一定更高。

## 15. 工程权衡

### 15.1 为什么 test-level comparison，而不是 production API

当前目标是验证实验链路。API 需要 dataset 管理、strategy config、run persistence、权限边界和错误语义。先用测试打通比较逻辑更稳，后续需求稳定后再产品化。

### 15.2 为什么不接真实 Embedding

当前只比较 chunking strategy。接真实 embedding 会引入新变量，导致效果变化无法归因。一次只改变一个变量，是工程实验的基本纪律。真实 embedding 应放到 V2.5。

### 15.3 为什么不接 Vector DB

当前数据量小，comparison baseline 不需要 ANN / vector DB。Vector DB 会引入索引参数、召回近似误差和部署复杂度，应放到 V2.6。

### 15.4 为什么不做 Rerank

Rerank 会改变排序。当前目标是 chunking strategy comparison，Rerank 应在基础 retrieval 和 metrics 稳定后单独评估，应放到 V2.8。

### 15.5 为什么不声称 Adaptive 更好

小数据集不足以得出泛化结论，当前 scorer 也不是真实 embedding。V2.4.3 只建立 comparison baseline，后续大规模 benchmark 后再判断。

## 21. 面试高强度追问与回答

### Q1：为什么要比较 Fixed 和 Adaptive Chunking？

因为 chunking 直接决定 retriever 能看到的最小证据单元。Fixed Chunking 简单稳定，但可能切断语义；Adaptive Chunking 更关注标题、段落和结构边界，但也更复杂。比较二者不是为了证明某个策略天然更优，而是为了建立一个在同一 corpus、query、evidence 和 metrics 下做判断的实验框架。

### Q2：Chunking 为什么会影响 Retrieval？

Retriever 检索的是 chunk，而不是原始文档全文。如果 chunk 太碎，关键上下文可能分散；如果 chunk 太大，噪声会增加，query 和 chunk 的相关性被稀释。chunk 边界还会影响 citation、context 构造和后续 rerank。因此 chunking 是 RAG pipeline 里非常关键的前置设计。

### Q3：Fixed Chunking 的优缺点是什么？

Fixed Chunking 的优点是实现简单、稳定、适用于任意文本，便于作为 baseline。缺点是它不了解文档结构，可能把标题和正文切开，也可能把一个完整 evidence 切成两段。它适合作为对照组，但不一定适合所有 RAG 场景。

### Q4：Adaptive Chunking 的优缺点是什么？

Adaptive Chunking 尝试利用 Markdown heading、段落和语义边界，理论上更容易保留完整证据。它适合结构化文档，但实现复杂，对输入格式敏感，也可能因为规则不合适导致 chunk 过大或过小。所以它需要评估，而不是凭直觉宣布更好。

### Q5：为什么 Adaptive 不是天然更好？

因为检索效果由多个因素共同决定，包括 corpus、query、chunk 大小、overlap、embedding、scoring、rerank 和 topK。Adaptive 只是改变切分边界，在某些文档上可能更好，在另一些文档上可能产生更大 chunk 或引入噪声。工程上必须用 benchmark 验证。

### Q6：为什么使用同一套 benchmark？

如果 corpus、query 或 expected evidence 不同，就无法判断指标差异来自 chunking strategy 还是输入变化。同一套 benchmark 固定了实验条件，让唯一变化变量尽量只剩 chunking strategy，这样比较结果才有解释价值。

### Q7：expectedEvidenceIds 如何映射到不同 strategy 下的 chunk？

benchmark 里记录的是 evidence marker id，比如 `outbox-reason`。测试先生成某个 strategy 的 chunks，再查找包含 `[EVIDENCE:outbox-reason]` 的 chunk，把该 chunk index 作为这个 strategy 下的 expected chunk。Fixed 和 Adaptive 的 chunk index 可能不同，所以需要分别映射。

### Q8：为什么不直接使用 expectedChunkIds？

chunkId 是数据库生成的，依赖上传顺序和环境。对于静态 benchmark，写死 chunkId 会让测试不可复现。expectedEvidenceIds 更稳定，因为它锚定的是 corpus 中的 evidence marker，而不是某次导入后的数据库 ID。

### Q9：为什么当前不用真实 Embedding？

当前版本只想比较 chunking strategy。如果引入真实 Embedding，就同时改变了模型变量，结果难以归因。test-level lexical scorer 虽然不代表真实语义效果，但它足够让 comparison pipeline 跑通。真实 Embedding 应在后续 V2.5 单独接入，再复用同一套 benchmark 比较。

### Q10：为什么不强制断言 Adaptive 指标更高？

因为当前 seed corpus 小、query 直接、scorer 也不是语义 embedding。强行断言 Adaptive > Fixed 会让测试变成对某个偶然结果的绑定。V2.4.3 更关注 pipeline 完整性：两种 strategy 都能生成 chunks、映射 evidence、执行 scoring、计算 metrics。

### Q11：为什么不新增生产 API？

当前 comparison 还处于 test-level baseline，没有 dataset 管理、strategy config、run persistence、权限设计和 dashboard。直接暴露 API 容易固化一个不成熟的接口。先用测试验证实验链路，后续再产品化更稳。

### Q12：为什么不持久化 experiment result？

因为当前还没有多轮实验历史、报告、dashboard 和企业用户场景。持久化需要设计 evaluation_experiment、evaluation_run、evaluation_result 等表，会扩大版本边界。等 benchmark runner 和真实策略对比稳定后，再做 persistence 更合适。

### Q13：后续接真实 Embedding 后怎么复用？

可以保留同一 corpus、benchmark cases、expectedEvidenceIds 和 metrics，只替换 retrieval scoring 层：从 lexical overlap 换成 real embedding + vector search。这样能比较 Mock / Real Embedding 或不同 embedding 模型在同一 benchmark 下的表现。

### Q14：如果企业语料库更大，这套 comparison 怎么升级？

需要把 test-level runner 升级为正式 benchmark runner，支持批量 dataset、异步 evaluation job、结果持久化、并发控制和报告生成。chunking strategy 也需要变成可配置参数，而不是写死在测试里。

### Q15：如果 Adaptive 指标比 Fixed 差，说明什么？

不能立刻说明 Adaptive 没价值。可能是 seed corpus 太小、scorer 太弱、chunk size 参数不合适、overlap 不合适，或者 query 与 evidence 太直接。应该进一步看 per-case 结果，分析是 recall 下降、precision 下降，还是 ranking 变差，再决定调整策略。

### Q16：如何避免为了 benchmark 过拟合 chunking 策略？

需要把 benchmark 分成开发集和 holdout set，不要只针对一小组 seed cases 调参数。还要加入真实用户 query、hard negatives、多跳问题和不同文档类型。线上失败案例也应该逐步进入 benchmark，避免策略只对 demo 文档有效。

### Q17：如何用 NDCG 判断 rerank 是否有效？

NDCG 关注 relevant chunks 的整体排序位置。如果 rerank 后 relevant chunks 更靠前，NDCG@K 应该提升。相比 MRR 只看第一个 relevant，NDCG 更适合判断多个 evidence 的整体排序质量。后续 V2.8 做 rerank 时，可以用同一 benchmark 比较 rerank 前后的 NDCG。

### Q18：这个版本和完整 RAG Evaluation 的区别是什么？

V2.4.3 只做 chunking strategy 对 retrieval 的影响比较，不评估 generator。完整 RAG Evaluation 还需要 answer correctness、faithfulness、citation accuracy、LLM-as-a-judge、真实用户 query、真实 embedding、rerank 和长期报告。当前只是 comparison baseline。

### Q19：如果 Recall 高但 ContextPrecision 低，说明什么？

说明正确 evidence 被找回了，但 TopK 里混入了较多噪声。对 RAG 来说，这会增加 LLM context 噪声，可能导致回答发散或引用不精确。后续可以考虑调整 chunk size、topK、embedding、rerank 或过滤策略。

### Q20：如果 Fixed 在小 corpus 上更好，你会放弃 Adaptive 吗？

不会。小 corpus 和 test-level scorer 不能代表真实企业场景。Fixed 更好只说明当前 seed / scorer 下它表现不错，应该继续扩大 benchmark、加入真实 embedding 和更多文档类型，再做更可靠判断。工程上看趋势和 per-case failure，而不是被一次小样本结果牵着走。
