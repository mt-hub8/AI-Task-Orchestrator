# V2.6.7 Qdrant Benchmark Result Capture

## 1. 版本摘要

V2.6.7 的目标是为已有的 Qdrant 检索能力补齐 **Benchmark Result Capture（基准测试结果采集）**。

在此之前，项目已经完成：

- `VectorStore` 抽象；
- `ExactCosineVectorStore` 作为稳定 baseline（基准对照实现）；
- `QdrantVectorStore` 作为可选 Vector DB provider；
- `VectorStoreBenchmarkRunner`；
- Retrieval Evaluation Harness（检索评估框架）；
- Qdrant 本地手工验证路径；
- Local Embedding Worker 工程化封装。

但是，仅仅有 benchmark runner 还不够。面试官或 reviewer 会进一步追问：

> 你说可以对比 ExactCosineVectorStore 和 QdrantVectorStore，那真实结果在哪里？  
> 你用的是什么数据集？topK 是多少？embedding provider、model、dimension 是什么？  
> 指标怎么计算？latency 怎么记录？结果能不能复现？

V2.6.7 解决的就是这个问题：

> 把 exact vs qdrant benchmark 从“代码里可以跑”推进到“可以手工运行、输出结果文件、记录元数据、解释指标和限制”。

本版本的重点不是证明 Qdrant 一定更快，而是建立一条可信的证据采集路径。

---

## 2. 为什么现在做

当前项目已经有 Qdrant 的代码接入和手工启动方式，但如果没有真实 benchmark result capture，会出现一个可信度缺口：

```text
有 QdrantVectorStore
有 benchmark runner
有 Qdrant docker compose
但没有标准化的结果采集和输出路径
```

这会导致几个问题：

1. 面试时只能说“我有 benchmark 框架”，但不能展示结果证据；
2. reviewer 不知道如何复现 exact vs qdrant 对比；
3. 指标结果缺少 dataset / topK / model / dimension 等上下文；
4. latency 数据容易被误读；
5. 后续 README、简历、面试文档中的 benchmark 表格缺少来源。

V2.6.7 的价值是：

> 建立 benchmark evidence（基准测试证据），让系统能力从“功能实现”进一步变成“可验证结论”。

---

## 3. 本版本解决的问题

### 3.1 benchmark 能跑，但结果不可沉淀

如果 benchmark 只是在控制台打印，问题是：

- 运行后结果难保存；
- 无法和其他版本对比；
- 不方便复制到文档；
- 无法保留运行配置；
- 无法追踪数据集和模型信息。

V2.6.7 通过输出：

```text
build/reports/retrieval/qdrant-benchmark-result.json
build/reports/retrieval/qdrant-benchmark-summary.md
```

让结果可以被保存、阅读和后续引用。

---

### 3.2 结果缺少上下文会失去意义

检索 benchmark 不能只写：

```text
Recall@K = 0.8
Latency = 12ms
```

因为这些数字必须绑定上下文：

- dataset 是什么；
- query case 数量是多少；
- topK 是多少；
- embedding provider 是什么；
- model 是什么；
- dimension 是多少；
- baseline 是谁；
- candidate 是谁；
- Qdrant collection 是什么；
- distance metric 是什么；
- 是否重新 upsert；
- 是否本地 Docker；
- 是否冷启动；
- 机器环境是什么。

如果没有这些上下文，benchmark 数字没有解释力。

---

### 3.3 Qdrant 不能被随意声称更快

很多人会在项目里写：

```text
接入 Qdrant，提升检索性能
```

但这句话如果没有数据支撑，很容易被面试官追问。

而且在小数据集下，`ExactCosineVectorStore` 可能比 Qdrant 更快。原因很直接：

- Exact baseline 可能是内存扫描；
- Qdrant 需要 HTTP 调用；
- 需要序列化 / 反序列化；
- 需要经过 Docker 网络；
- 小数据集下外部服务开销可能大于 ANN 检索收益。

所以 V2.6.7 的正确表达不是：

```text
Qdrant 一定比 exact 快
```

而是：

```text
我建立了 exact vs qdrant 的手工 benchmark 结果采集流程，真实结果以固定 dataset、topK、provider、model、dimension 和本地环境为准，不做未经证明的性能外推。
```

这比盲目夸大更专业。

## 4. 核心设计

### 4.1 opt-in manual benchmark

V2.6.7 的 benchmark 必须是显式启用的 manual benchmark，而不是默认测试。

原因是 Qdrant benchmark 依赖：

- Docker；
- Qdrant 服务；
- 端口；
- collection 状态；
- 可能的 Local Embedding Worker；
- 可能的模型下载；
- 本地机器性能。

这些都不适合放进默认 `.\mvnw.cmd test`。

所以推荐使用类似：

```java
@EnabledIfSystemProperty(named = "qdrant.manual.benchmark", matches = "true")
```

手工运行时显式启用：

```powershell
.\mvnw.cmd -Dqdrant.manual.benchmark=true -Dtest=QdrantVectorStoreManualBenchmarkTest test
```

默认测试仍然保持低依赖和确定性。

---

### 4.2 baseline 与 candidate

本版本对比两个对象：

```text
baseline  = ExactCosineVectorStore
candidate = QdrantVectorStore
```

这里的 baseline 不是“低级实现”，而是稳定对照组。

`ExactCosineVectorStore` 的价值是：

- 默认测试稳定；
- 行为容易解释；
- 小数据集可控；
- 不依赖外部服务；
- 可作为 Qdrant 结果的对照。

`QdrantVectorStore` 的价值是：

- 接入真实 Vector DB；
- 支持外部向量存储；
- 为后续大规模检索、payload filter、ANN、生产化部署打基础。

两者配合后，benchmark 才有比较意义。

---

### 4.3 复用已有 retrieval dataset

V2.6.7 不应该临时写几条测试数据，而应该复用已有：

```text
retrieval-corpus-v1.md
retrieval-benchmark-v1.json
```

原因：

1. 这些数据已经和前面的 Retrieval Evaluation Harness 对齐；
2. 已有 expectedEvidenceIds；
3. 已有 Evidence Mapper；
4. 已有 Recall@K、Precision@K、MRR、NDCG@K 等指标计算；
5. 可以保持版本链的一致性。

这样做的意义是：

> 不重新造一套临时 benchmark，而是把已有评估体系延伸到 Qdrant provider 对比。

---

### 4.4 复用已有 metrics calculator

V2.6.7 不重新实现指标。

应该继续复用已有：

- `RetrievalEvaluationService`
- `RetrievalMetricsCalculator`
- `VectorStoreBenchmarkRunner`
- `BenchmarkEvidenceMapper`

原因：

- 避免重复实现产生口径不一致；
- 保证 exact 和 qdrant 使用同一套评价标准；
- 保持 V2.4.x、V2.6.2 和 V2.6.7 的指标链路一致。

## 6. 指标解释

### 6.1 Recall@K（召回率@K）

Recall@K 衡量：

> 对于一个 query，系统在前 K 个检索结果中找回了多少应该被找回的 evidence。

简单理解：

```text
应该找回 2 个证据，topK 里找回 1 个，Recall@K = 1 / 2 = 0.5
```

它关注“有没有漏掉关键证据”。

---

### 6.2 Precision@K（精确率@K）

Precision@K 衡量：

> 前 K 个检索结果中，有多少是真正相关的 evidence。

简单理解：

```text
topK 返回 5 个结果，其中 2 个相关，Precision@K = 2 / 5 = 0.4
```

它关注“返回结果里有多少是有用的”。

---

### 6.3 HitRate@K（命中率@K）

HitRate@K 衡量：

> 前 K 个结果中是否至少命中一个正确 evidence。

它通常是 0 或 1：

- 命中至少一个：1；
- 一个都没命中：0。

它适合回答：

> 用户最少能不能看到一个相关结果？

---

### 6.4 MRR（平均倒数排名）

MRR 全称 Mean Reciprocal Rank，中文可以理解为“平均倒数排名”。

它关注第一个正确结果出现得有多靠前。

如果第一个正确结果排第 1：

```text
1 / 1 = 1.0
```

如果第一个正确结果排第 3：

```text
1 / 3 = 0.333
```

如果没有正确结果：

```text
0
```

MRR 越高，说明正确答案越靠前。

---

### 6.5 NDCG@K（归一化折损累计增益@K）

NDCG@K 关注两个点：

1. 正确结果有没有出现；
2. 正确结果是不是排得足够靠前。

它比 Recall@K 更重视排序质量。

直观理解：

> 同样找回 2 个正确结果，如果它们排第 1、第 2，比排第 4、第 5 更好。

---

### 6.6 ContextPrecision@K（上下文精确率@K）

ContextPrecision@K 更贴近 RAG 场景。

它衡量：

> 放进上下文窗口的 topK chunks 中，有多少是真正有用的上下文。

它对 RAG 很重要，因为 LLM 最终会读取这些 chunks。如果上下文里噪声太多，模型可能回答错误或成本上升。

---

## 7. latency 如何理解

latency 指每次 search 的耗时。

但 latency 不能孤立解读。

需要考虑：

- 数据规模；
- topK；
- 是否本地 Docker；
- 是否 HTTP 调用；
- 是否冷启动；
- Qdrant collection 状态；
- 向量维度；
- payload filter；
- 机器 CPU / 内存；
- 是否有其他进程占用资源。

尤其在小数据集上：

```text
ExactCosineVectorStore 可能更快，因为它是内存计算；
Qdrant 可能更慢，因为它需要跨进程 HTTP 请求。
```

所以 latency 的正确用法是：

> 作为当前环境下的观测值，而不是绝对性能结论。

## 8. 为什么结果文件默认不提交

`build/reports/retrieval/` 下的文件是本地运行产物，默认不提交 Git。

原因：

- 不同机器结果不同；
- 不同模型结果不同；
- 不同 topK 结果不同；
- 不同 Qdrant collection 状态结果不同；
- 本地报告会频繁变化；
- 容易污染版本历史。

可以提交的是：

```text
docs/manual/qdrant-benchmark-result-capture.md
```

也就是运行说明。

如果未来要提交真实结果，应该单独新增一份带日期和环境说明的文档，例如：

```text
docs/benchmark/qdrant-benchmark-result-2026-xx-xx.md
```

并明确：

> 这是某次本地手工运行结果，不代表生产性能结论。

## 9. 为什么不新增生产 API

V2.6.7 不新增：

```text
POST /benchmark/qdrant
POST /admin/benchmark/run
```

原因：

1. benchmark 是工程验证工具，不是当前产品 API；
2. 项目还没有 admin / auth 体系；
3. benchmark 可能消耗资源；
4. 暴露 HTTP API 会引入安全和权限问题；
5. 当前阶段用 manual runner / opt-in test 更合适。

所以本版本只做手工 benchmark runner，不做生产 API。

## 10. 为什么不做 Testcontainers

Testcontainers 可以后续做，但 V2.6.7 不做。

原因：

- 当前目标是 result capture，不是 CI integration；
- Testcontainers 会引入依赖；
- CI 环境复杂度会上升；
- Qdrant 启动、collection 初始化和数据清理都需要额外设计；
- 当前项目已经采用 manual verification 分层策略。

后续可以单独做：

```text
Qdrant Testcontainers Integration
```

## 16. 面试高强度追问与回答

### Q1：这个版本证明 Qdrant 比 exact search 更快了吗？

A：不能这样说。这个版本建立的是可复现 benchmark result capture path。具体结果只代表当前 dataset、topK、embedding model、dimension、本地机器和 Qdrant 配置下的观测值，不能外推为生产性能结论。

---

### Q2：为什么还需要 ExactCosineVectorStore？

A：ExactCosineVectorStore 是 deterministic baseline。它不依赖外部服务，适合默认测试、小数据集验证和 Qdrant 对照。如果没有 baseline，就无法判断 Qdrant 的结果是否合理。

---

### Q3：为什么要记录 provider / model / dimension？

A：因为 embedding 向量空间由 provider、model、dimension 决定。不同模型或不同维度的向量不能直接混用。benchmark 不记录这些信息，结果就不可复现，也无法解释。

---

### Q4：为什么 benchmark 不放进默认 test？

A：真实 Qdrant benchmark 依赖 Docker、Qdrant 服务、端口、collection 状态，可能还依赖 Python worker 和模型下载。默认 test 应该稳定低依赖，所以真实 benchmark 必须显式手工启用。

---

### Q5：如果 Qdrant 指标比 exact 差，说明什么？

A：不一定说明 Qdrant 接入失败。可能是数据集太小、embedding 质量、topK 设置、collection 状态、payload filter、upsert 过程或 HTTP 开销导致。benchmark 的价值是暴露真实情况，而不是强行证明某个方案更好。

---

### Q6：为什么要输出 JSON 和 Markdown 两种报告？

A：JSON 适合机器读取和后续自动化分析；Markdown 适合人阅读、面试展示和复制到文档中。两者服务不同场景。

---

### Q7：为什么结果文件默认不提交？

A：因为 result 是本地运行产物，不同机器、模型、topK、collection 状态都会影响结果。默认提交容易污染版本历史。应提交运行说明，而不是每次运行产物。真实结果可以后续作为带日期和环境说明的独立 benchmark 文档提交。

---

### Q8：这个版本和后续 Rerank / Hybrid Search 有什么关系？

A：Rerank 和 Hybrid Search 都需要 evaluation baseline。V2.6.7 建立了结果采集方式，后续任何检索策略变化都可以用同样方式比较，而不是凭感觉说效果更好。

---
