# V2.6 VectorStore Abstraction / Vector DB Selection

## 1. 版本摘要

V2.6 新增 `VectorStore`（向量存储）抽象层，并将当前 exact cosine search（精确余弦检索）封装为默认 baseline（基线）实现 `ExactCosineVectorStore`。

本版本做了这些事：

- 新增 `VectorStore` 抽象；
- 新增 `VectorStoreDocument`、`VectorSearchRequest`、`VectorSearchFilter`、`VectorSearchResult`；
- 将当前 exact cosine search 封装为 `ExactCosineVectorStore`；
- 将上层检索逻辑从具体 exact search 实现中解耦；
- 新增 `VectorStoreProperties` 和 `VectorStoreConfiguration`；
- 默认配置为 `app.vector-store.provider=exact`；
- 保留 provider / model / dimension 正确性边界；
- 不接入外部 Vector DB；
- 为后续 Qdrant / pgvector / Milvus 等接入做准备。

必须明确：V2.6 是架构边界版本，不是外部 Vector DB 接入版本。它解决的是“向量存储和检索应该有稳定抽象”，不是“现在就把 Qdrant 或 pgvector 接进生产链路”。

## 2. 本版本解决的问题

V2.6 之前，项目已经能完成 document search 和 retrieval evaluation，但检索逻辑直接绑定当前 exact cosine search。这样短期能跑通 RAG，但后续接入 Qdrant、pgvector、Milvus、Weaviate 或 Elasticsearch dense_vector 时，业务 service 很容易直接依赖某个具体 Vector DB。

这个版本解决的真实问题是：

- 原有检索逻辑和当前 exact cosine search 耦合，不利于后续替换；
- 如果直接接 Qdrant / pgvector，业务服务会被具体 client、网络调用和 filter 语义污染；
- 没有统一 `VectorStore` 抽象时，很难做 exact search vs ANN search 对比；
- 默认测试容易被外部服务、Docker、网络和真实 Vector DB 影响；
- 后续 VIP 检索策略、metadata filter、多 Vector DB 选型缺少统一入口。

所以 V2.6 不是为了“多加一个接口”，而是为了让 RAG 检索层从具体实现升级成可替换的基础设施边界。

## 3. 为什么现在做 V2.6

前置能力已经具备：

- V2.4.x 已经完成 Retrieval Evaluation Harness；
- V2.4.2 / V2.4.4 已经有 query-evidence benchmark seed 和 evidence mapper；
- V2.5 已经完成 `EmbeddingProvider` 抽象；
- V2.5.1 已经能做 embedding provider benchmark comparison；
- V2.5.2 已经支持 local-worker embedding provider。

也就是说，文本如何生成向量已经被 `EmbeddingProvider` 抽象出来。现在需要把 `EmbeddingProvider` 之后的向量存储和检索层抽象出来。

如果跳过 V2.6 直接接 Qdrant，会同时混入：

- 外部服务；
- 网络调用；
- Docker 配置；
- ANN（近似最近邻检索）索引；
- metadata filter 语义；
- 数据同步；
- benchmark 指标变化；
- CI 稳定性问题。

这样指标变化难以归因，exact baseline 也可能丢失。V2.6 先做抽象，是为了让后续接入外部 Vector DB 时只替换实现，而不是改散业务链路。

## 4. V2.6 之前的检索链路

V2.6 之前的链路大致是：

1. 文档上传；
2. 文档切分；
3. 通过 `EmbeddingProvider` 生成 embedding；
4. 持久化 chunk 和 embedding metadata；
5. service 直接查出候选 embedding；
6. 反序列化向量；
7. 直接计算 exact cosine similarity；
8. 按 score 排序并返回 topK；
9. 进入 retrieval evaluation。

这个实现的问题是：

- 检索实现和业务 service 耦合；
- 后续接外部 Vector DB 时改动会扩散；
- 不利于 fake / mock / external provider 分层测试；
- 不利于 exact vs ANN benchmark；
- 不利于保留 exact search 作为 correctness baseline（正确性基线）。

## 5. 核心架构设计

V2.6 之后的目标分层是：

```text
Document / Query
-> EmbeddingProvider（向量生成提供者）
-> VectorStore（向量存储与检索）
-> RetrievalEvaluation（检索评估）
```

### 5.1 EmbeddingProvider

`EmbeddingProvider` 负责：

- 将文本转成向量；
- 屏蔽 `mock` / OpenAI-compatible / `local-worker`；
- 返回 provider / model / dimension metadata。

它回答的是：“文本如何变成向量？”

### 5.2 VectorStore

`VectorStore` 负责：

- 存储向量；
- 按 query embedding 检索 topK；
- 处理 provider / model / dimension 过滤；
- 返回 score / rank / metadata；
- 未来支持 Qdrant / pgvector / Milvus 等实现。

它回答的是：“向量如何存储、过滤和检索？”

### 5.3 RetrievalEvaluation

`RetrievalEvaluation` 负责：

- 根据 expectedEvidenceIds / expectedChunkIds 评价检索结果；
- 计算 Recall@K（召回率@K）、Precision@K（精确率@K）、MRR（平均倒数排名）、NDCG@K（归一化折损累计增益@K）、ContextPrecision@K（上下文精确率@K）等指标；
- 不评价生成答案事实性；
- 不做 LLM-as-a-judge（大模型裁判）。

## 6. VectorStore 与 EmbeddingProvider 的边界

`EmbeddingProvider` 解决“文本如何变成向量”。

`VectorStore` 解决“向量如何存储、过滤和检索”。

这两个职责不能混在一起。比如：

- 使用 OpenAI-compatible embedding，不代表必须使用某个 Vector DB；
- 使用 local-worker embedding，也可以搭配 exact search / Qdrant / pgvector；
- 后续可以组合 `app.embedding.provider=mock` + `app.vector-store.provider=exact`；
- 后续可以组合 `app.embedding.provider=local-worker` + `app.vector-store.provider=qdrant`；
- 后续可以组合 `app.embedding.provider=openai` + `app.vector-store.provider=pgvector`。

这种分层让 embedding 模型选择和向量存储选择可以独立演进。模型侧可以从 mock 切到 OpenAI 或本地 worker，存储侧也可以从 exact search 切到 Qdrant 或 pgvector，二者不互相绑死。

## 7. ExactCosineVectorStore 为什么作为 baseline

`ExactCosineVectorStore` 不是临时代码，也不是“还没接 Vector DB 的过渡写法”。它是后续评估 ANN（近似最近邻检索）和外部 Vector DB 的 baseline（基线）。

保留 exact baseline 的理由：

1. 小规模数据下 exact search 结果稳定；
2. 不依赖外部服务；
3. 默认测试稳定；
4. 可以作为 correctness baseline（正确性基线）；
5. 后续 Qdrant / pgvector 引入 ANN 索引后，可以对比 Recall@K（召回率@K）、NDCG@K（归一化折损累计增益@K）、latency（延迟）；
6. ANN 检索通常用速度换取部分召回，因此需要 exact baseline 做对照。

V2.6 不声称检索效果提升，只保证架构解耦和行为兼容。

## 8. provider / model / dimension 为什么仍然重要

不同 `EmbeddingProvider`、不同 model、不同 dimension 生成的向量不能混用。

原因是：即使它们都是 float 数组，只要来自不同模型或不同维度，就可能位于不同向量空间。直接混合计算 cosine similarity，结果没有可靠语义意义。

因此 `VectorStoreDocument` 和 `VectorSearchRequest` 中保留：

- provider；
- model；
- dimension。

这不是字段冗余，而是 correctness boundary（正确性边界）。

举例：

- `mock / fake embedding` 不应和 OpenAI embedding 混搜；
- `sentence-transformers/all-MiniLM-L6-v2` 的 384 维向量不应和 1536 维模型混搜；
- 同一个 query embedding 只能检索同一个 embedding space 下的 chunk。

## 9. 候选 Vector DB 选型分析

### 9.1 pgvector

pgvector 是 PostgreSQL 生态中的向量扩展，适合主库已经是 PostgreSQL 的系统。

优点：

- 复用 PostgreSQL；
- SQL / 事务 / 备份 / 权限体系成熟；
- 适合中小规模 RAG；
- 可以减少基础设施数量。

缺点：

- 本项目当前主库是 MySQL；
- 如果 V2.6.1 直接选 pgvector，会引入 MySQL + PostgreSQL 双数据库形态；
- 对当前项目不是最自然的第一选择。

结论：pgvector 是重要候选，但更适合 PostgreSQL 体系项目。

### 9.2 Qdrant

Qdrant 是专用 Vector DB，适合作为 MySQL 主库之外的向量检索服务。

优点：

- 和 MySQL 主库分工明确；
- 适合 RAG 检索服务化；
- 支持 payload / metadata filter；
- 适合后续 documentId、tenantId、userPlan、provider、model、dimension 等过滤场景；
- 适合作为 V2.6.1 第一候选。

缺点：

- 多一个外部服务；
- 需要 Docker / 运维 / 数据同步；
- 默认 CI 不能依赖真实 Qdrant；
- 需要 fake / optional external test 分层。

结论：本项目后续 V2.6.1 倾向优先接入 Qdrant。

### 9.3 Milvus

Milvus 更适合大规模向量检索、多模态、分布式场景。

优点：

- 大规模能力强；
- 适合海量向量；
- 工业级向量检索能力完整。

缺点：

- 当前项目阶段过重；
- 部署和运维复杂度更高；
- 会让 V2.6.1 膨胀。

结论：Milvus 适合后续规模化阶段，不适合当前作为第一接入目标。

### 9.4 Weaviate

Weaviate 产品化能力强，支持更完整的 AI search 场景，尤其适合后续 hybrid search。

优点：

- 产品化能力强；
- hybrid search 能力清晰；
- 适合更完整的 RAG 搜索产品。

缺点：

- 当前阶段容易混入 Vector DB、BM25、Hybrid Search、Rerank 等多个变量；
- 不利于保持 V2.6 的变量单一。

结论：Weaviate 可作为后续 Hybrid Search 阶段候选，不作为 V2.6.1 首选。

### 9.5 Elasticsearch dense_vector

Elasticsearch 适合已有搜索平台的企业，能够结合全文检索和向量检索。

优点：

- 关键词检索生态成熟；
- 过滤、聚合、日志检索能力强；
- 适合后续统一搜索平台或 hybrid search。

缺点：

- 如果只是纯向量检索，系统复杂度偏高；
- 当前项目没有必要为了 `VectorStore` 抽象直接引入 Elasticsearch。

结论：Elasticsearch 更适合后续 Hybrid Search / 企业搜索集成阶段。

## 10. 为什么本版本不直接接入外部 Vector DB

V2.6 只改变一个核心变量：`VectorStore` 抽象。

如果直接接入 Qdrant / pgvector，会同时改变：

- 外部服务依赖；
- 网络调用；
- Docker 配置；
- ANN 索引；

这样指标变化不可归因。所以当前版本不做外部 Vector DB 接入，是有意控制工程边界，不是能力缺失。

## 14. 核心流程

V2.6 后的流程是：

1. 文档上传；
2. 文档切分；
3. `EmbeddingProvider` 生成 chunk embedding；
4. embedding metadata 持久化；
5. 查询文本生成 query embedding；
6. 构造 `VectorSearchRequest`；
7. `VectorStore` 执行 search；
8. `ExactCosineVectorStore` 作为默认实现计算 cosine similarity；
9. 按 score 排序，返回 topK；
10. 上层 service 转换为原有 search response；
11. `RetrievalEvaluationService` 继续使用检索结果计算指标。

核心变化是：上层 service 依赖 `VectorStore` 抽象，而不是直接依赖 exact cosine search 细节。

## 22. 面试高强度追问与回答

### Q1：为什么 V2.6 先做 VectorStore 抽象，而不是直接接 Qdrant？

因为 V2.6 要先控制变量。如果直接接 Qdrant，会同时引入外部服务、网络调用、Docker、ANN 索引、payload filter 和数据同步，指标变化很难归因。

先做 `VectorStore` 抽象可以防止业务 service 绑定具体 Vector DB，同时保留 `ExactCosineVectorStore` 作为 baseline。后续接 Qdrant 时，只需要新增一个 `VectorStore` 实现，而不是改散上层检索链路。默认测试也能继续不依赖外部服务。

### Q2：VectorStore 和 EmbeddingProvider 的边界是什么？

`EmbeddingProvider` 负责文本转向量，屏蔽 mock、OpenAI-compatible、local-worker 等不同向量生成方式。

`VectorStore` 负责向量存储、过滤和检索，屏蔽 exact search、Qdrant、pgvector 等不同检索后端。

两者可以独立替换，这是 RAG infrastructure 分层设计的核心。

### Q3：为什么 exact cosine search 还要保留？

exact cosine search 是 correctness baseline。小数据集下它稳定、不依赖外部服务、默认测试可控。

后续 ANN 检索可能用速度换取部分 Recall@K（召回率@K），所以必须有 exact baseline 作为对照，否则无法判断 Qdrant 或 pgvector 的检索结果是否损失了召回。

### Q4：provider / model / dimension 为什么不能混用？

不同模型的向量空间不同。维度不同的向量无法比较；即使维度相同，不同模型生成的语义空间也可能不同。

如果把 `mock`、OpenAI、local-worker 或不同 model 的向量混在一起做 cosine similarity，结果没有可靠语义意义。因此 provider / model / dimension 是 correctness boundary。

### Q5：为什么当前项目后续倾向 Qdrant，而不是 pgvector？

当前项目主库是 MySQL。pgvector 更适合 PostgreSQL 主库体系，如果现在选 pgvector，会引入 MySQL + PostgreSQL 双数据库形态。

Qdrant 作为独立 Vector DB，和 MySQL 主库分工更自然，也适合 RAG 服务化。它的 payload / metadata filter 能力适合后续 documentId、tenantId、userPlan、provider、model、dimension 等过滤场景。

### Q6：如果数据规模扩大 10 倍，V2.6 会遇到什么瓶颈？

`ExactCosineVectorStore` 仍然是 exact search，全量候选扫描会让延迟上升，CPU 和内存压力也会增加。metadata filter 如果只靠当前持久化结构，也很难支撑复杂过滤和高 QPS。

这时需要 ANN 索引和专用 Vector DB，例如 Qdrant 或 Milvus，并用 benchmark 验证延迟和召回质量的权衡。

### Q7：metadata filter 为什么会影响 Vector DB 选择？

企业 RAG 不只是向量相似度，还需要按 tenantId、documentId、userPlan、权限、文档类型等条件过滤。

filter 选择率会影响 ANN 查询性能。Vector DB 是否支持 payload index、filter pushdown、过滤后召回质量，都会影响最终选型。

### Q8：为什么不在 V2.6 做 Hybrid Search？

Hybrid Search 会引入 BM25、score fusion、权重调节和关键词索引。它改变的是检索策略，不只是向量存储抽象。

如果和 V2.6 混在一起，就无法判断变化来自 VectorStore 抽象，还是来自 Hybrid Search。因此应该作为独立版本评估。

### Q9：为什么不在 V2.6 做 Rerank？

Rerank 是二阶段排序，VectorStore 是候选召回。如果一起做，无法判断指标变化来自召回质量，还是来自重排模型。

所以 V2.6 只做 VectorStore 抽象，V2.8 再单独做 Rerank，并独立评估它对 NDCG@K（归一化折损累计增益@K）和 MRR（平均倒数排名）的影响。

### Q10：如何验证 Qdrant 比 exact search 更值得引入？

需要固定 corpus、固定 benchmark queries、固定 expectedEvidenceIds，再用同一套 evaluation harness 对比 `ExactCosineVectorStore` 和 `QdrantVectorStore`。

指标不能只看速度，还要看 Recall@K（召回率@K）、NDCG@K（归一化折损累计增益@K）、MRR（平均倒数排名）和 latency（延迟）。如果速度提升但召回明显下降，不一定值得默认启用。

### Q11：如果 ANN 检索 Recall@K 下降怎么办？

可以调整 HNSW / `ef_search` / index 参数，提高召回质量；也可以提高 topK，让 rerank 接收更多候选。

对于 VIP 用户，可以使用更高质量策略，比如更高 topK、更强 embedding 模型、rerank 或更保守的 ANN 参数。exact baseline 仍然用于回归验证。

### Q12：这个版本怎么写进简历才不夸大？

应该写架构抽象和扩展点，不写性能提升，也不写“接入多个 Vector DB”。

没有 benchmark 数据前，不写提升百分比。可以写“设计并实现可插拔 VectorStore 抽象，将 exact cosine retrieval 封装为 baseline，为后续 Vector DB 接入和 ANN benchmark 提供统一扩展点”。
