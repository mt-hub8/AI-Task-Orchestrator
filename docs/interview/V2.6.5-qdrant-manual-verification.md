# V2.6.5 Qdrant Manual Verification

## 1. 版本摘要

V2.6.5 的目标是为已有的 `QdrantVectorStore` 补齐本地手工验证闭环。

在 V2.6.1 中，项目已经接入了 Qdrant provider；在 V2.6.2 中，项目已经有 VectorStore Benchmark Harness。但在此之前，Qdrant 仍然主要停留在：

```text
代码里有 QdrantVectorStore
测试里有 fake client / mock path
默认测试不连接真实 Qdrant
```

这对稳定测试是正确的，但对项目展示还不够。面试官或 reviewer 可能会追问：

> 你说接入了 Qdrant，那我怎么在本地确认它真的能启动、连接、写入和检索？

V2.6.5 解决的就是这个问题。

本版本新增或完善：

- 独立 `docker-compose.qdrant.yml`
- 独立 `application-qdrant.properties`
- Qdrant 手工验证文档
- 可选 `qdrant-flow.http`
- 可选默认 disabled 的 manual integration test

核心原则：

> 默认测试继续保持 deterministic（确定性）和低依赖；真实 Qdrant 通过 manual verification（手工验证）路径隔离验证。

---

## 2. 为什么现在做

当前项目已经有：

- VectorStore 抽象；
- ExactCosineVectorStore baseline；
- QdrantVectorStore 原型接入；
- Retrieval Evaluation Harness；
- VectorStore Benchmark Harness；
- E2E Demo Golden Path；
- API Error Response Standardization。

但 Qdrant path 的可信度还缺一环：

```text
默认测试证明逻辑可测；
手工验证证明真实外部组件可跑。
```

如果没有 V2.6.5，项目容易被认为：

- 只是写了 Qdrant 相关类；
- 没有真实服务验证；
- provider 切换路径不明确；
- reviewer 不知道怎么复现；
- Qdrant 是“简历词”，不是可运行能力。

V2.6.5 的价值是：

> 把 Qdrant 从“代码原型”推进到“本地可复现验证”。

---

## 3. 本版本解决的问题

### 3.1 Qdrant 缺少独立启动方式

Qdrant 是可选 VectorStore provider，不应该强行进入主 `docker-compose.yml`。

主 compose 应保持核心开发依赖：

- MySQL；
- RabbitMQ；
- 其他核心任务系统基础设施。

Qdrant 属于可选 RAG / VectorStore 基础设施，应通过独立文件启动：

```powershell
docker compose -f docker-compose.qdrant.yml up -d
```

这样可以避免所有开发者、所有测试、所有 demo 都默认依赖 Qdrant。

---

### 3.2 Qdrant 缺少独立 profile 配置

如果 Qdrant 配置混在默认 `application.properties` 中，容易出现两个问题：

1. 默认启动误连 Qdrant；
2. 测试和 demo 的 provider 行为不清楚。

所以 V2.6.5 通过 `application-qdrant.properties` 隔离配置：

```properties
app.vector-store.provider=qdrant
app.vector-store.qdrant.base-url=http://localhost:6333
app.vector-store.qdrant.collection-name=...
```

具体字段必须以当前项目实际配置类为准。

启动时通过 profile 切换：

```powershell
.\mvnw.cmd spring-boot:run "-Dspring-boot.run.profiles=docker,qdrant"
```

---

### 3.3 缺少手工验证文档

真实项目不能只依赖口头说明：

> 你本地起一个 Qdrant，然后跑一下就行。

手工验证文档必须说明：

- 验证目标；
- 本验证不是什么；
- 前置条件；
- 如何启动 Qdrant；
- 如何确认 Qdrant 可访问；
- 如何用 qdrant profile 启动 Spring Boot；
- 如何准备测试文档；
- 如何触发 chunk / embedding / vector upsert；
- 如何执行 search topK；
- 如何确认走的是 QdrantVectorStore；
- 如何切回 exact baseline；
- 常见问题；
- 如何清理容器和 volume；
- 为什么默认 test 不依赖 Qdrant。

这让 reviewer 能直接复现，而不是猜测。

---

## 4. 核心设计

### 4.1 独立 docker-compose.qdrant.yml

设计原则：

```text
Qdrant 是可选依赖，不进入主 docker-compose.yml。
```

好处：

- 默认开发环境更轻；
- 默认 test 不受影响；
- Qdrant 手工验证路径独立；
- 后续可单独扩展 Qdrant volume / config / snapshot；
- 不污染主任务系统基础设施。

典型命令：

```powershell
docker compose -f docker-compose.qdrant.yml up -d
docker compose -f docker-compose.qdrant.yml ps
docker compose -f docker-compose.qdrant.yml down
```

如果需要清理 volume，应在文档中明确警告：

```text
清理 volume 会删除本地 Qdrant 数据。
```

---

### 4.2 独立 application-qdrant.properties

设计原则：

```text
通过 profile 显式选择 qdrant provider，而不是默认启用。
```

好处：

- 默认 exact baseline 保持稳定；
- qdrant provider 可显式切换；
- 面试时能清楚解释 provider 策略；
- 本地验证和默认测试隔离。

示例语义：

```properties
app.vector-store.provider=qdrant
app.vector-store.qdrant.base-url=http://localhost:6333
app.vector-store.qdrant.collection-name=ai_task_orchestrator_chunks
```

具体配置名以项目实际代码为准。

---

### 4.3 手工验证文档

文档是本版本的核心产物。

它应该明确：

> 本验证是 manual verification，不是默认 CI integration test。

也应该明确：

> 本版本不声明 Qdrant 性能优于 exact search，只验证真实 Qdrant path 可运行。

这可以避免过度承诺。

---

### 4.4 可选 qdrant-flow.http

如果当前 API 能支持完整流程，可以提供 `.http` 文件：

- 上传文档；
- 触发 chunking；
- 触发 embedding；
- search topK。

但必须遵守：

```text
只写真实存在的 API。
```

不能把 benchmark runner 或测试类伪装成 HTTP API。

如果当前没有完整 HTTP 入口，应在文档中如实说明：

```text
当前某些验证步骤通过测试 / runner / service path 验证，尚未暴露为生产 HTTP API。
```

---

### 4.5 可选 Manual Integration Test

允许新增默认 disabled 的测试，例如：

```java
@Disabled("Manual Qdrant integration test. Requires local Qdrant.")
class QdrantVectorStoreManualIntegrationTest {
}
```

要求：

- 默认 `.\mvnw.cmd test` 不运行；
- 不启动 Docker；
- 不依赖真实 Qdrant；
- 不依赖 OpenAI API；
- 使用固定测试向量或 mock embedding；
- 类名明确带 Manual。

这个测试的价值是：

> 给工程师一个可选的手工验证入口，但不破坏默认 CI 稳定性。

---

## 5. 为什么默认测试不连接 Qdrant

这是 V2.6.5 的重要工程原则。

默认测试应该具备：

- deterministic（确定性）；
- 低依赖；
- 不需要 Docker；
- 不需要外部服务；
- 不需要网络；
- 不需要 API key；
- 能在 CI 和本地稳定运行。

真实 Qdrant 属于外部基础设施。如果默认 test 连接 Qdrant，会产生：

- 环境依赖；
- 启动顺序问题；
- 端口冲突；
- 数据残留问题；
- CI 配置复杂度；
- 偶发失败。

所以正确策略是：

```text
默认测试验证抽象、逻辑和 fake client；
真实组件通过 manual verification 隔离验证。
```

这比“所有东西都塞进 test”更成熟。

---

## 6. 为什么保留 ExactCosineVectorStore

ExactCosineVectorStore 不是临时代码，而是 baseline。

它的价值包括：

1. 默认测试稳定；
2. 小数据集下行为可解释；
3. 可作为 Qdrant 的对照组；
4. 可验证 provider / model / dimension 过滤逻辑；
5. 可用于 benchmark delta 计算；
6. 不依赖外部 Vector DB。

Qdrant 的价值是可扩展的 Vector DB provider；Exact 的价值是稳定 baseline。

二者不是互相替代关系，而是：

```text
ExactCosineVectorStore = deterministic baseline
QdrantVectorStore = external vector database provider
```

---

## 7. 为什么不接 Testcontainers

Testcontainers 是一个合理的后续方向，但 V2.6.5 不做。

原因：

1. 当前目标是手工验证闭环，不是 CI integration test；
2. Testcontainers 会引入依赖和 CI 配置复杂度；
3. 当前项目仍在面试项目和工程原型阶段；
4. 默认测试低依赖优先级更高；
5. 后续可以专门做 Qdrant Testcontainers Integration。

面试表达：

> 我没有把外部基础设施强行塞进默认测试，而是先通过 manual verification 验证真实 Qdrant path，后续如果 CI 环境稳定，再引入 Testcontainers 做自动化集成测试。

---

## 8. 为什么不做性能结论

V2.6.5 不能声称：

```text
Qdrant 性能显著优于 exact search
Qdrant recall 更高
Qdrant 一定更适合生产
```

因为这些结论需要：

- 固定数据集；
- 固定 embedding provider；
- 固定 model；
- 固定 dimension；
- 固定 topK；
- 固定 collection 参数；
- 固定 Qdrant 配置；
- 多次运行；
- 记录 latency / recall / precision / NDCG；
- 对比 baseline。

V2.6.5 只做：

```text
真实 Qdrant path 可启动、可配置、可验证。
```

性能和召回对比应该放到后续 benchmark result capture。

## 9. 手工验收路径

典型手工验收流程：

### 9.1 启动 Qdrant

```powershell
cd E:\code\ai-task-orchestrator
docker compose -f docker-compose.qdrant.yml up -d
docker compose -f docker-compose.qdrant.yml ps
```

### 9.2 检查 Qdrant 可访问

可以通过浏览器、curl 或 PowerShell 访问 Qdrant HTTP endpoint。

具体命令以文档为准。

### 9.3 启动 Spring Boot with qdrant profile

```powershell
.\mvnw.cmd spring-boot:run "-Dspring-boot.run.profiles=docker,qdrant"
```

### 9.4 执行业务验证

按 `qdrant-verification.md` 或 `qdrant-flow.http`：

1. 准备测试文档；
2. 上传文档；
3. 触发 chunking；
4. 触发 embedding；
5. upsert 到 Qdrant；
6. search topK；
7. 确认返回 chunk；
8. 切回 exact baseline 对比。

### 9.5 清理

```powershell
docker compose -f docker-compose.qdrant.yml down
```

如需清理 volume，应谨慎执行对应命令。

## 15. 面试高强度追问与回答

### Q1：为什么 Qdrant 不进入主 docker-compose？

A：因为 Qdrant 是可选 VectorStore provider，不是任务系统的核心依赖。主 docker-compose 保持轻量，Qdrant 通过独立 compose 文件手工启动，避免所有开发、测试和 demo 都默认依赖 Vector DB。

---

### Q2：为什么默认 test 不连接 Qdrant？

A：默认测试需要 deterministic 和低依赖。真实 Qdrant 属于外部基础设施，会引入 Docker、端口、启动顺序、数据残留和 CI 稳定性问题。因此默认测试验证抽象和逻辑，真实 Qdrant 通过 manual verification 单独验证。

---

### Q3：这是不是说明 Qdrant 没有真正接入？

A：不是。QdrantVectorStore 已经作为可选 provider 接入，本版本补充了独立启动、profile 配置和手工验证路径。区别只是默认测试不依赖真实 Qdrant，这是为了测试稳定性，而不是没有接入。

---

### Q4：为什么保留 ExactCosineVectorStore？

A：ExactCosineVectorStore 是 deterministic baseline，可以用于默认测试、小规模验证和 benchmark 对照。Qdrant 是可扩展 Vector DB provider，二者分别承担 baseline 和外部 provider 的角色。

---

### Q5：这个版本能证明 Qdrant 比 exact search 更快吗？

A：不能。这个版本只证明 Qdrant path 可以真实运行。性能结论需要固定数据集、embedding model、dimension、topK 和多次 benchmark 输出，不能凭手工验证下结论。

---

### Q6：为什么不直接上 Testcontainers？

A：Testcontainers 是后续可以做的自动化集成测试方案，但当前阶段优先补手工验证路径，降低依赖和 CI 复杂度。等 Qdrant path 稳定后，再引入 Testcontainers 更合理。

---

### Q7：如何确认应用真的走的是 QdrantVectorStore？

A：通过 qdrant profile 设置 `app.vector-store.provider=qdrant`，启动日志、配置打印、请求结果和 Qdrant collection 数据都可以作为确认依据。文档中应给出具体检查步骤。

---

### Q8：如果 Qdrant 启动失败，系统应该怎么办？

A：在 qdrant profile 下启动失败是合理的，因为该 profile 明确依赖 Qdrant。默认 profile 不应受影响。后续可以做更细粒度的 health check 和降级策略，但当前版本不做自动 fallback。

---

### Q9：为什么不做 payload index 自动创建？

A：payload index 和 collection migration 属于生产化 Vector DB 运维策略，会扩大范围。V2.6.5 只做手工验证闭环，后续可以单独做 Qdrant production hardening。

---

### Q10：这个版本和后续 VIP Search 有什么关系？

A：VIP Search 会依赖更稳定的 retrieval infrastructure。Qdrant 作为可选 provider 完成手工验证后，后续才能更可信地做不同 userPlan 下的 topK、rerank、hybrid search 或更高性能检索策略。
