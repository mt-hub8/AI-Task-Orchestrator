# V2.4.4 Benchmark Runner / Evidence Mapper 面试文档

## 0. 版本摘要

V2.4.4 在 V2.4.2 Evaluation Dataset Seed 基础上，补充 test-level Benchmark Runner / Evidence Mapper，把稳定的 expectedEvidenceIds 自动映射成运行时真实 expectedChunkIds，并调用现有 RetrievalEvaluationService 跑通 retrieval benchmark。

一句话定位：

V2.4.4 把静态 query-evidence benchmark seed 转换成可执行的 retrieval benchmark run。

必须明确：

- 本版本是 test-level benchmark runner；
- 本版本不是 production benchmark API；
- 本版本不是完整 evaluation platform；
- 本版本不新增生产 API；
- 本版本不新增数据库表；
- 本版本不持久化 evaluation run；
- 本版本不接真实 Embedding；
- 本版本不接 Vector DB；
- 本版本不做 Rerank；
- 本版本不做 Faithfulness / Answer Relevance；
- 本版本不修改 seed corpus / benchmark JSON。

## 1. 本版本解决的问题

V2.4.2 已经提供稳定 benchmark seed，但它不能直接调用 V2.4 的 RetrievalEvaluationService。

原因是 V2.4.2 benchmark JSON 使用：

- expectedEvidenceIds

而 V2.4 RetrievalEvaluationService / API 使用：

- expectedChunkIds

核心矛盾：

- expectedEvidenceIds 是稳定 authoring format；
- expectedChunkIds 是 evaluation service 的运行时计算输入；
- chunkId 来自数据库或运行时 chunk 生成；
- chunkId 不适合写死在静态 benchmark JSON 中。

V2.4.4 解决的是这个桥接问题：

```text
expectedEvidenceIds
-> [EVIDENCE:id] marker
-> containing chunk
-> runtime chunkId
-> RetrievalEvaluationRequest
-> RetrievalEvaluationService
-> evaluation response
```

核心表达：

V2.4.2 解决 benchmark 输入稳定性；V2.4.4 解决 benchmark 输入可执行性。

## 2. 为什么现在做 V2.4.4

依赖关系已经具备：

- V2.4 已有 RetrievalEvaluationService；
- V2.4.1 已有 Recall@K、Precision@K、HitRate@K、MRR、NDCG@K、ContextPrecision@K；
- V2.4.2 已有 query-evidence benchmark seed；
- V2.4.3 已验证 evidence marker 可以和 chunking strategy 结合；
- 所以现在可以做 expectedEvidenceIds -> expectedChunkIds 的自动映射。

为什么不直接进入 V2.5 Real Embedding：

- 接入真实 Embedding 后，仍然需要 benchmark 自动化；
- 如果 expectedEvidenceIds 还要手工转 chunkId，评测链路不完整；
- Real Embedding 的效果也无法稳定复现评估；
- 所以先补 Benchmark Runner / Evidence Mapper，再进入真实检索质量升级更稳。

核心表达：

先把 benchmark seed 自动跑起来，再替换检索底层能力。

## 3. 方案选择

### 方案 A：跳过 V2.4.4，直接做 Real Embedding

优点：

- 更快接近真实 RAG；
- 可以尽早使用真实语义向量；
- 对项目展示有吸引力。

缺点：

- benchmark 仍然不能自动运行；
- expectedEvidenceIds 仍要手工转换成 expectedChunkIds；
- Real Embedding 效果无法稳定评估；
- evaluation seed 的价值没有完全释放。

结论：不推荐作为当前下一步。

### 方案 B：test-level Benchmark Runner / Evidence Mapper

做法：

- 在测试中加载 corpus 和 benchmark JSON；
- 解析 evidence markers；
- 生成或读取实际 chunks；
- 将 expectedEvidenceIds 映射成真实 chunkId；
- 构造 RetrievalEvaluationRequest；
- 调用 RetrievalEvaluationService；
- 验证 response 中的 case results 和 summary metrics。

优点：

- 边界小；
- 不新增生产 API；
- 不新增数据库；
- 不持久化 run；
- 能真正打通 benchmark seed -> evaluation service；
- 为 Real Embedding / Vector DB / Rerank 提供可复用评测链路。

缺点：

- 不是正式产品能力；
- 没有 run history；
- 没有 dashboard；
- 不能给企业用户直接上传 benchmark。

结论：当前推荐方案。仓库当前实现采用 `BenchmarkRunnerEvidenceMapperTest`。

### 方案 C：新增生产 API

方向：

`POST /evaluations/benchmark-runs`

优点：

- 更像产品能力；
- 可以手工调用；
- 后续可以接 dashboard 和 report。

缺点：

- 需要 API schema；
- 需要 dataset 管理；
- 需要 run persistence；
- 需要权限与所有权设计；
- 当前阶段容易过早固化接口。

结论：暂时不选。

### 方案 D：新增 evaluation persistence 表

方向：

- evaluation_dataset；
- evaluation_case；
- evaluation_expected_evidence；
- evaluation_run；
- evaluation_result。

优点：

- 更接近企业级 evaluation platform；
- 可以保存历史 run；
- 可以支持报告和 dashboard。

缺点：

- 版本过大；
- 当前还没有真实 Embedding；
- 还没有大规模 benchmark；
- 还没有企业 dashboard；
- 过早设计 schema 风险高。

结论：后续 Evaluation Persistence 阶段再做。

## 4. 核心流程

当前 `BenchmarkRunnerEvidenceMapperTest` 的真实流程是：

```text
加载 retrieval-corpus-v1.md
-> 加载 retrieval-benchmark-v1.json
-> 解析 [EVIDENCE:id] marker
-> 使用 DocumentChunker 生成 chunks
-> 保存 DocumentEntity
-> 保存 DocumentChunkEntity
-> 调用 DocumentEmbeddingService.embedDocument(documentId)
-> 遍历本次保存的 chunks，查找包含 marker 的 chunk
-> 获取真实 chunkId
-> 将每个 case 的 expectedEvidenceIds 转换为 expectedChunkIds
-> 构造 RetrievalEvaluationRequest
-> 调用 RetrievalEvaluationService.evaluate(request)
-> 获取 RetrievalEvaluationResponse
-> 验证 case results 非空
-> 验证 summary metrics 非空
-> 验证 summary 包含 Recall@K / Precision@K / HitRate@K / MRR / NDCG@K / ContextPrecision@K
```

必须强调：

- 不重写 metrics；
- 不绕过 RetrievalEvaluationService；
- 不依赖固定数据库自增 ID；
- 不依赖已有数据库数据；
- chunkId 来自本次测试实际生成 / 保存的 chunks。

## 5. 为什么 benchmark JSON 不直接写 expectedChunkIds

expectedChunkIds 看起来可以直接调用 evaluation API，但它不适合作为静态 benchmark authoring format。

原因：

- chunkId 是数据库生成的；
- 不同环境 chunkId 不一致；
- 重新上传 corpus 后 chunkId 会变化；
- 不同 chunking strategy 下同一个 evidence 可能落在不同 chunk；
- 静态 benchmark 文件写死 chunkId 会不可复现；
- benchmark 应该表达“应该命中哪段证据”，而不是“某次数据库里的具体自增 ID”。

所以 V2.4.2 / V2.4.4 采用：

- expectedEvidenceIds 作为静态格式；
- expectedChunkIds 作为运行时格式。

## 6. expectedEvidenceIds 如何映射为 expectedChunkIds

映射规则：

1. 对每个 benchmark case 读取 expectedEvidenceIds；
2. 将每个 evidenceId 转成 marker：

```text
[EVIDENCE:<id>]
```

3. 在本次生成 / 保存的 chunks 中查找包含该 marker 的 chunk；
4. 取该 chunk 的真实 chunkId；
5. 加入 expectedChunkIds；
6. 用 expectedChunkIds 构造 RetrievalEvaluationRequest。

当前测试中的异常处理语义：

- 如果 marker 找不到，测试失败；
- 如果 marker 出现在多个 chunks 中，测试失败；
- 如果 expectedEvidenceIds 为空，测试失败；
- 转换后的 expectedChunkIds 会去重，但 seed 内重复 evidence id 本身不应出现。

## 7. 为什么调用 RetrievalEvaluationService，而不是测试里重写 metrics

V2.4 / V2.4.1 已经实现 metrics 计算，包括：

- Recall@K；
- Precision@K；
- HitRate@K；
- MRR / RR@K；
- NDCG@K；
- ContextPrecision@K。

V2.4.4 的职责是：

- 映射 evidence；
- 构造 request；
- 调用现有 evaluation service。

它不应该：

- 重新实现 Recall；
- 重新实现 Precision；
- 重新实现 NDCG；
- 复制一套 metrics 逻辑。

原因：

- 避免两套 evaluation 逻辑分叉；
- 保证 benchmark runner 和正式 evaluation API 使用同一套计算逻辑；
- 后续 V2.5 / V2.6 / V2.8 可以直接复用。

核心表达：

Evidence Mapper 负责把 ground truth 转成 service 输入；RetrievalEvaluationService 负责计算指标。

## 14. 工程权衡

### 14.1 为什么 expectedEvidenceIds 是 authoring format

expectedEvidenceIds 稳定，不依赖数据库，不依赖 chunkId，适合写在静态 benchmark JSON，适合跨环境复现。它表达的是“这个 query 的正确证据在哪里”，而不是“某次运行里的数据库行 id 是多少”。

### 14.2 为什么 expectedChunkIds 是 runtime format

RetrievalEvaluationService 最终需要和 retrieved chunkId 比较。chunkId 只能在文档实际 chunking / 保存后确定，所以它应该运行时生成，而不是静态写死。

### 14.3 为什么不重写 metrics

不重写 metrics 可以避免逻辑分叉。benchmark runner 复用正式 evaluation service，后续指标扩展、bug 修复或公式调整只需要维护一套逻辑。

### 14.4 为什么 test-level runner 优于现在做 production API

当前需求还不稳定。先验证核心映射链路，再抽象 API，可以降低版本风险。生产 API 需要 dataset 管理、权限、持久化、异步任务和成本控制，这些都不该在 V2.4.4 一次性完成。

### 14.5 为什么不做 persistence

当前只有小规模 seed benchmark，没有历史对比、dashboard 和企业报告需求。现在不应该过早建表，避免把还没稳定的 evaluation run 模型固化进 schema。

## 20. 面试高强度追问与回答

### Q1：为什么 benchmark JSON 不直接写 expectedChunkIds？

因为 expectedChunkIds 是运行时概念，它来自某次文档上传和 chunk 保存后的数据库 ID。不同环境、不同上传顺序、不同 chunking 策略都会产生不同 chunkId。benchmark JSON 应该表达稳定的 ground truth evidence，而不是绑定到某一次数据库状态。

### Q2：expectedEvidenceIds 的作用是什么？

expectedEvidenceIds 是 benchmark authoring format。它让标注者用稳定 id 表达“这个 query 应该命中哪段证据”。这些 id 对应 corpus 里的 `[EVIDENCE:id]` marker，不依赖数据库，也适合跨环境复现。

### Q3：expectedEvidenceIds 如何映射到 expectedChunkIds？

测试读取 corpus 和 benchmark 后，先用 DocumentChunker 生成 chunks 并保存到数据库，得到真实 chunkId。然后对每个 expectedEvidenceId 构造 marker，在本次保存的 chunks 中查找包含该 marker 的 chunk，把该 chunk 的 id 加入 expectedChunkIds。

### Q4：为什么 chunkId 必须运行时生成？

chunkId 取决于数据库自增 ID 和当前 chunking 结果。重新运行测试、换数据库或换策略，都可能改变 chunkId。运行时生成可以保证 benchmark 文件稳定，同时 evaluation request 使用真实当前环境的 chunkId。

### Q5：为什么调用 RetrievalEvaluationService，而不是测试里重新算 metrics？

因为 V2.4 / V2.4.1 已经实现正式 metrics 逻辑。V2.4.4 的目标是桥接 seed 和 service，不是复制公式。复用 RetrievalEvaluationService 可以保证 benchmark runner 与正式 API 使用同一套计算逻辑，避免测试和生产指标分叉。

### Q6：为什么当前只做 test-level runner？

因为生产 benchmark runner 需要 dataset 管理、权限、run persistence、异步任务、报告和 dashboard。当前阶段先验证核心映射和执行链路，边界更小，也为后续产品化提供依据。

### Q7：为什么不新增 production API？

API 一旦暴露，就要考虑请求 schema、错误语义、鉴权、限流、成本和兼容性。当前 benchmark runner 还处于验证阶段，先不把它固化成生产接口，更符合渐进式工程演进。

### Q8：为什么不做 evaluation persistence？

因为当前没有 run history、dashboard 或企业报告需求。持久化需要 evaluation_dataset、case、run、result 等 schema，这些应该等 benchmark runner 和真实策略对比稳定后再设计。

### Q9：如果一个 evidence marker 落在多个 chunks 里怎么办？

当前测试要求 exactly one chunk 命中。如果一个 marker 出现在多个 chunks，说明 corpus marker 重复、chunking overlap 导致重复包含，或切分逻辑需要明确处理。此时应该 fail fast，而不是悄悄取第一个 chunk。

### Q10：如果一个 evidence marker 被 chunking 切断怎么办？

mapper 会找不到完整 marker，测试失败。这是有价值的信号：说明 chunking 可能破坏了 benchmark evidence marker。后续可以调整 chunking、marker 放置方式，或在 mapper 中支持 span-level 定位，但当前版本保持简单严格。

### Q11：后续接 Real Embedding 后怎么复用？

只要保留同一 corpus、benchmark JSON 和 evidence mapper，就可以把 embedding/search 底层从 MockEmbedding 换成真实 provider。Evaluation request 和 metrics 仍然复用现有 RetrievalEvaluationService，从而比较 Mock vs Real 或不同模型效果。

### Q12：后续接 Vector DB 后怎么复用？

Vector DB 改变的是 retrieval backend。只要 search service 返回 chunkId、score、metadata，benchmark runner 仍然可以把 expectedEvidenceIds 映射成 expectedChunkIds，并用同一 metrics 评估 exact search 与 ANN search 的差异。

### Q13：企业 benchmark runner 还需要哪些能力？

需要 dataset 管理、case 管理、权限隔离、run history、异步任务、报告导出、dashboard、失败诊断、成本统计、定期回归评测和告警。V2.4.4 只是 test-level bridge，还没有这些产品化能力。

### Q14：如何支持大规模 benchmark？

需要把 runner 任务化，按 case 批处理，控制并发，缓存 embeddings，支持断点续跑，持久化 run result，并记录 latency、cost 和失败原因。大规模 benchmark 不适合直接同步跑在一个测试方法里。

### Q15：这个版本和 V2.4.3 的区别是什么？

V2.4.3 关注 Fixed vs Adaptive Chunking Comparison，验证不同 chunking strategy 都能基于同一 seed benchmark 计算指标。V2.4.4 关注 benchmark runner / evidence mapper，把 expectedEvidenceIds 自动转成真实 expectedChunkIds，并调用现有 RetrievalEvaluationService。

### Q16：这个版本和完整 RAG Evaluation 的区别是什么？

V2.4.4 只做 retrieval benchmark run，而且是 test-level。完整 RAG Evaluation 还包括 generation quality、faithfulness、answer relevance、citation accuracy、LLM-as-a-judge、run history、dashboard 和企业 dataset 管理。

### Q17：如果 benchmark runner 映射失败，应该怎么排查？

先检查 benchmark expectedEvidenceIds 是否在 corpus 中存在，再检查 DocumentChunker 是否生成了包含 marker 的 chunk。如果 marker 被切断或重复出现，需要检查 chunk 边界、overlap、corpus marker 位置和 seed 文件唯一性。当前测试 fail fast，可以尽早暴露这些问题。

### Q18：为什么说 expectedEvidenceIds 是 authoring format，expectedChunkIds 是 runtime format？

authoring format 是人写 benchmark 时使用的稳定标注，不应该依赖环境。runtime format 是服务运行时计算指标需要的输入，必须绑定当前实际 chunks。expectedEvidenceIds 稳定，expectedChunkIds 精确，二者通过 mapper 衔接。

### Q19：如果企业用户要上传 benchmark，你会如何设计 API？

我会先设计 dataset 资源：上传 corpus、上传 cases、定义 expected evidence，再提供 run API。run 可以异步执行，结果持久化到 evaluation_run / evaluation_result，并支持按 embedding model、chunking strategy、topK、rerank policy 做对比。还需要权限、配额、成本和报告。

### Q20：如果要保存历史 evaluation run，你会怎么设计表？

可以拆成 evaluation_dataset、evaluation_case、evaluation_expected_evidence、evaluation_run、evaluation_case_result、evaluation_summary_result。dataset 保存输入，run 保存一次实验配置，result 保存每个 case 和 summary metrics。这样既能复现，也能做趋势对比。
