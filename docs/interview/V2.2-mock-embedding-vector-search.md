# V2.2 Mock Embedding & Vector Search 面试文档

## 0. 版本摘要

V2.2 实现了 RAG 检索层 baseline：抽象 `EmbeddingClient`，使用 `MockEmbeddingClient` 对 `document_chunk` 生成 128 维向量，将向量持久化到 `document_chunk_embedding` 表，并通过 Java 内存 cosine similarity 实现 TopK chunk 检索。

本版本明确不接真实 Embedding Provider，不接 OpenAI / BGE / Qwen / Jina / E5 / Voyage，不接 pgvector / Qdrant / Milvus / Elasticsearch，不做 RAG Answer，不做 Citation，不做 Retrieval Evaluation。

它的价值不是“证明 mock embedding 效果好”，而是先跑通：

```text
chunk -> embedding -> persistence -> query embedding -> similarity -> TopK
```

## 1. 本版本解决的问题

V2.1 / V2.1.1 已经完成文档上传、chunking、Adaptive Chunking 和 chunk metadata，但 `document_chunk` 还只是静态文本片段，不能被 query 检索。

V2.2 解决的问题是：把 `document_chunk` 从“可存储文本片段”升级为“可检索知识片段”。

核心链路：

```text
document_chunk
-> embedding vector
-> vector persistence
-> query embedding
-> cosine similarity
-> TopK retrieval
```

没有检索，就没有 RAG Answer；没有 score，就无法做 Retrieval Evaluation；没有 `headingPath` / `startOffset` / `endOffset` / `chunkStrategy`，后续就很难做 Citation 和 Chunking 对比。

## 2. 为什么现在做 V2.2

版本依赖关系：

- V2.1 完成 Document Upload & Chunking。
- V2.1.1 完成 Adaptive Chunking。
- V2.3 要做 RAG Answer with Citation。
- 所以 V2.2 必须先完成检索层 baseline。

RAG Answer 可以拆成：

```text
RAG Answer = Retrieval + Prompt Assembly + LLM Generation + Citation
```

现在还没有 Retrieval，所以不能直接做 RAG Answer。否则只能把固定文本硬塞给 LLM，无法根据用户 query 找到最相关的上下文，也无法解释答案来自哪些 chunk。

## 4. Embedding 基础理解

### 4.1 Embedding 是什么

Embedding 是把文本、图片、代码等非结构化对象映射成稠密向量，使语义相近的对象在向量空间中距离更近。

在 RAG 中，query 和 chunk 都会被映射到同一个向量空间，然后通过 cosine similarity、dot product 或 L2 distance 做相似度计算。

可以简单理解为：

```text
query: "退款申请多久内提交"
chunk: "用户需要在订单完成后 7 天内提交退款申请"
```

它们字面不完全一样，但语义接近，所以理想情况下向量相似度应该较高。

### 4.2 Embedding 算法经历了哪几代

第一代：静态词向量

- Word2Vec
- GloVe
- FastText

特点：

- 一个词一个向量。
- 不能很好处理上下文语义。
- FastText 能利用 subword，对未登录词更友好。

局限：

- “苹果”在“苹果手机”和“苹果水果”中是同一个向量。
- 更适合词级表示，不是现代 RAG 句子级语义检索的主流方案。

第二代：上下文表示

- ELMo
- BERT

特点：

- 同一个词在不同上下文中可以有不同表示。
- 更适合理解句子语义。
- 原始 BERT 理解能力强，但不是专门为向量检索优化的。

局限：

- 如果直接用 BERT Cross-Encoder 做大规模第一阶段召回，需要 query 和每个候选 chunk 两两拼接，在线成本太高。

第三代：句向量 / 检索向量模型

- SBERT
- SimCSE
- BGE
- E5
- OpenAI Embedding
- Qwen Embedding
- Jina Embedding
- Voyage Embedding

特点：

- 面向 sentence/document embedding。
- 更适合 query-document 相似度。
- document chunk 和 query 可以独立编码。
- 文档向量可以提前离线计算。
- query 在线只算一次。
- 可用于大规模向量检索和 RAG。

### 4.3 RAG 为什么主流用 Bi-Encoder

Bi-Encoder 结构：

```text
query -> vector
chunk -> vector
similarity(query_vector, chunk_vector)
```

优点：

- chunk embedding 可以提前离线计算。
- query 来了只需要计算 query embedding。
- 支持向量数据库 ANN 检索。
- 适合大规模召回。

缺点：

- query 和 document 分开编码，交互弱。
- 精排能力不如 Cross-Encoder。

Cross-Encoder 结构：

```text
[query, chunk] -> model -> relevance score
```

优点：

- query 和 chunk 深度交互。
- relevance 判断更准。
- 适合 rerank。

缺点：

- 每个 query-chunk pair 都要跑一次模型。
- 无法提前为 chunk 独立建索引。
- 不适合第一阶段大规模召回。

结论：

RAG 常用 Bi-Encoder 做第一阶段 TopK 召回，再用 Cross-Encoder / reranker 做第二阶段重排。

### 4.4 Bi-Encoder 与 Cross-Encoder 的工程差异

是否能预计算：

- Bi-Encoder 可以预计算 document chunk embedding。
- Cross-Encoder 不能预计算 query-chunk pair，只能在线计算。

是否能建向量索引：

- Bi-Encoder 可以建 ANN 向量索引。
- Cross-Encoder 通常不能直接用于向量索引召回。

延迟：

- Bi-Encoder 在线只算 query embedding，再做向量检索。
- Cross-Encoder 需要对多个 query-chunk pair 逐个打分。

成本：

- Bi-Encoder 更适合大规模低成本召回。
- Cross-Encoder 更适合小候选集精排。

适用阶段：

- Bi-Encoder：retrieval / recall。
- Cross-Encoder：rerank / precision improvement。

大规模检索可行性：

- Bi-Encoder 可行。
- Cross-Encoder 直接做全量召回不可行。

rerank 场景：

- 先 Bi-Encoder 找 TopK。
- 再 Cross-Encoder 对 TopK 或 TopN rerank。

## 5. 当前方案选择

当前方案是：

```text
MockEmbeddingClient + MySQL TEXT + Java 内存 cosine similarity
```

具体包括：

- `EmbeddingClient` 抽象。
- `MockEmbeddingClient` 实现。
- hash-based bag-of-words / char n-gram。
- 128 维向量。
- `document_chunk_embedding` 表。
- `embedding_vector` 用 MySQL `TEXT` 存逗号分隔向量。
- Java 侧计算 cosine similarity。
- 支持 `documentId` 可选过滤。
- `topK` 默认 5，最大 20。
- 返回 `score`、`content`、`headingPath`、`startOffset`、`endOffset`、`chunkStrategy` 等 metadata。

这是一个 baseline，不是生产级 retrieval engine。

## 6. 为什么先用 MockEmbedding

V2.2 的目标不是验证某个真实 embedding 模型效果，而是先把 RAG 检索链路的工程骨架打通。

MockEmbedding 的意义：

1. 解耦工程链路和外部模型依赖。
2. 避免 API key、网络、费用、限流影响开发。
3. 让测试稳定可重复。
4. 方便先验证数据库表、API、相似度计算、TopK 返回结构。
5. 后续替换真实 Embedding Provider 时，只需要替换 `EmbeddingClient` 实现。

这是阶段性工程取舍，不是因为不会接真实模型。

MockEmbedding 在这个阶段承担的是“工程占位符”和“可重复测试工具”的角色。

## 7. 为什么暂时不接真实 Embedding

候选模型：

- OpenAI Embedding
- BGE-large-zh
- BGE-M3
- Qwen Embedding
- Jina Embedding
- E5
- Voyage

暂时不接的原因：

- 当前还没有 query-evidence 评测集。
- 没有 Recall@K / MRR 数据，无法判断哪个模型更适合。
- 接真实模型会引入 API key、成本、延迟、限流、部署复杂度。
- 过早接真实 Provider 容易把问题混在一起：链路问题、模型问题、网络问题、数据问题。
- V2.2 先做接口抽象，V2.5 再做真实 Embedding Provider 选型更合理。

更合理的节奏是：

```text
先跑通链路
-> 再构造评测集
-> 再接多个真实模型
-> 再根据 Recall@K / MRR / latency / cost 做选型
```

## 8. 为什么暂时不用向量数据库

### 8.1 为什么不用 pgvector

- 当前数据量小，Java 内存 exact scan 足够。
- 当前重点是打通检索链路，不是压测向量索引。
- pgvector 适合 PostgreSQL 生态，但当前项目主库是 MySQL。
- 现在引入 PostgreSQL 会增加本地环境和 Flyway 管理复杂度。
- 后续 V2.6 会专门做 Vector DB Selection。

### 8.2 为什么不用 Qdrant

- Qdrant 是独立向量数据库，适合 metadata filter + ANN。
- 但当前还没做真实 embedding 和检索评测。
- 过早引入会增加服务依赖和运维成本。

### 8.3 为什么不用 Milvus

- Milvus 更适合大规模向量检索。
- 部署和运维成本更高。
- 对当前 baseline 阶段过重。

### 8.4 为什么不用 Elasticsearch

- Elasticsearch 更适合 BM25 / hybrid search。
- 当前 V2.2 重点是 dense vector baseline。
- hybrid search 应该放到后续 Retrieval Policy 版本。

## 9. 向量数据库分层选型

小规模 / Demo / 单机 baseline：

- MySQL TEXT
- Java exact scan
- 优点：简单、可控、低依赖
- 缺点：不可扩展

中小规模 / PostgreSQL 体系：

- pgvector
- 优点：关系数据和向量数据放一起，事务和 metadata filter 简单
- 缺点：极大规模和复杂 ANN 能力不如专用向量库

中大规模 / 独立向量数据库：

- Qdrant
- Milvus
- Weaviate

适用场景：

- 大规模知识库
- ANN 检索
- metadata filter
- 多租户
- 高 QPS
- 横向扩展

搜索增强 / Hybrid Search：

- Elasticsearch / OpenSearch
- BM25 + vector
- 适合关键词强匹配、日志、文档检索

这里不展开成完整 V2.6 方案，只说明未来选型方向。

## 10. 数据库设计

重点表：`document_chunk_embedding`

字段：

- `id`
- `document_id`
- `document_chunk_id`
- `embedding_provider`
- `embedding_model`
- `vector_dimension`
- `distance_metric`
- `embedding_vector`
- `created_at`

说明：当前实际实现字段是 `created_at`，没有额外加入 `updated_at`。如果后续需要跟踪 embedding 重算更新时间，可以再增加 `updated_at`。

为什么 `document_chunk_embedding` 和 `document_chunk` 分表：

embedding 是派生数据，不是原始 chunk 数据。独立成表更适合后续多模型、多版本、重建和评测。

分表后可以支持：

- 同一个 chunk 多个 embedding model。
- embedding 重算。
- A/B test。
- 不同 provider 对比。
- 删除或重建 embedding 不影响原始 chunk。

约束设计：

- `document_id` 支持按文档过滤。
- `document_chunk_id` 关联原文 chunk。
- `embedding_provider` 区分 mock / openai / bge / qwen / jina。
- `embedding_model` 支持同 provider 下多个模型。
- `vector_dimension` 支持不同模型维度。
- `distance_metric` 支持 COSINE / IP / L2 后续扩展。
- unique 约束避免重复生成同模型 embedding。

## 11. API 设计

### 11.1 POST /documents/{documentId}/embeddings

用途：

为指定 document 下的 chunks 生成 embedding，并写入 `document_chunk_embedding`。

响应字段：

- `documentId`
- `embeddingProvider`
- `embeddingModel`
- `vectorDimension`
- `embeddedChunkCount`

设计意图：

- 先显式生成 embedding。
- 不在上传文档时自动生成。
- 方便控制成本、重试、异步化升级。
- 后续可以改成 MQ 异步 embedding job。

### 11.2 POST /documents/search

用途：

根据 query 生成 query embedding，并在已有 chunk embedding 中做 TopK 检索。

请求字段：

- `query`
- `topK`
- `documentId` 可选

响应字段：

- `score`
- `documentId`
- `chunkId`
- `content`
- `headingPath`
- `startOffset`
- `endOffset`
- `chunkStrategy`
- `embeddingProvider`
- `embeddingModel`
- `distanceMetric`

这些 metadata 不是装饰，而是后续 Citation / Evaluation 的基础。

## 12. 核心流程

### 12.1 生成 Embedding 流程

```text
POST /documents/{documentId}/embeddings
-> 查询 document
-> 查询 document_chunk
-> 对每个 chunk 调用 EmbeddingClient
-> 得到 vector
-> 写入 document_chunk_embedding
-> 返回生成结果
```

重复生成时，会先删除同一 documentId + provider + model 下已有 embedding，再重新保存，避免重复记录。

### 12.2 Search TopK 流程

```text
POST /documents/search
-> 校验 query / topK
-> 调用 EmbeddingClient 生成 query vector
-> 加载候选 chunk embeddings
-> 可选按 documentId 过滤
-> 计算 cosine similarity
-> 排序
-> 截取 TopK
-> 回查 chunk content 和 metadata
-> 返回 search results
```

当前 search 是 exact scan，不是 ANN 检索。

## 19. 面试高强度追问与回答

Q1：Embedding 是什么？

A1：Embedding 是把文本、图片、代码等非结构化对象映射成连续稠密向量的技术。向量空间里的距离可以表达相似度：语义越接近，向量越接近。在项目里，Embedding 用来把 `document_chunk` 和用户 query 映射到同一个向量空间，然后用 cosine similarity 找 TopK chunks。当前用 MockEmbedding 验证工程链路，局限是它不代表真实语义效果。后续可以接 BGE、Qwen、OpenAI、Jina 等真实 Provider。

Q2：Embedding 算法经历了哪几代？

A2：第一代是 Word2Vec、GloVe、FastText 这类静态词向量，一个词一个向量；第二代是 ELMo、BERT 这类上下文表示，同一个词在不同上下文中可以有不同表示；第三代是 SBERT、SimCSE、BGE、E5、OpenAI Embedding、Qwen、Jina、Voyage 这类句向量或检索向量模型，更适合 query-document 检索。在项目里，V2.2 先用 MockEmbedding 占位，后续 V2.5 再接真实第三代 embedding 模型。

Q3：为什么 RAG 主流用 Bi-Encoder，而不是 Cross-Encoder？

A3：Bi-Encoder 可以让 query 和 chunk 分别编码，chunk embedding 可以离线预计算，query 在线只算一次，然后做向量检索，适合大规模召回。Cross-Encoder 需要把 `[query, chunk]` 拼起来对每个候选单独打分，效果更准但成本高，不适合第一阶段全量召回。工程上常用 Bi-Encoder 做 TopK recall，再用 Cross-Encoder 或 reranker 对小候选集精排。

Q4：为什么项目里先用 MockEmbedding？

A4：因为 V2.2 的目标是先跑通检索链路，而不是证明某个真实 embedding 模型效果好。MockEmbedding 不需要 API key，不依赖网络，没有费用和限流，测试也稳定可重复。这样可以先验证表结构、API、cosine similarity、TopK 返回和 metadata。等链路稳定后，再替换 `EmbeddingClient` 为真实 Provider。

Q5：MockEmbedding 有什么意义？

A5：MockEmbedding 是工程 baseline。它让系统可以在没有外部模型的情况下验证从 chunk 到向量、从 query 到 TopK 的完整闭环。它还帮助稳定单元测试，因为相同文本会生成相同向量。它的局限是语义效果有限，不能替代真实 embedding 模型。后续升级时，只需要保留接口契约并替换实现。

Q6：为什么不用真实 OpenAI / BGE / Qwen Embedding？

A6：当前还没有 query-evidence 评测集，也没有 Recall@K、MRR 等评价数据。过早接真实模型会引入 API key、成本、延迟、限流和部署复杂度。如果效果不好，很难判断是 chunking、模型还是检索实现的问题。所以 V2.2 先做抽象和 baseline，V2.5 再基于评测集做真实模型选型。

Q7：为什么 embedding_vector 用 MySQL TEXT？

A7：因为这是 baseline 阶段，MySQL TEXT 存逗号分隔向量实现简单，不需要引入额外基础设施，也方便理解向量检索底层过程。它的局限是不能高效做 ANN 索引，不适合大规模生产检索。后续 V2.6 可以迁移到 pgvector、Qdrant、Milvus 或 Elasticsearch。

Q8：为什么不用 pgvector？

A8：pgvector 适合 PostgreSQL 技术栈和中小规模 RAG，也方便和业务表 JOIN。但当前项目主库是 MySQL，引入 PostgreSQL 会增加本地环境、Flyway 和部署复杂度。V2.2 当前目标是 baseline，不是数据库选型。后续 V2.6 会专门评估 pgvector。

Q9：为什么不用 Milvus？

A9：Milvus 适合百万、千万级甚至更高规模的分布式向量检索，索引能力强，但部署和运维复杂。当前还没有真实 embedding、评测集和规模压力，引入 Milvus 会让基础设施复杂度超过当前阶段目标。所以先用 MySQL + Java exact scan 跑通链路。

Q10：当前方案瓶颈在哪里？

A10：主要瓶颈是 Java 内存全量扫描 embedding，数据量大后延迟会变高；向量存在 MySQL TEXT 中，也无法使用 ANN 索引；MockEmbedding 本身语义质量有限。此外，目前没有 rerank、hybrid search、多租户隔离和评测体系。后续需要真实 Provider、向量数据库和 Retrieval Evaluation。

Q11：chunk 数量扩大 10 倍怎么办？

A11：短期可以通过 `documentId` filter 缩小候选集、限制 topK、分页加载 embedding、缓存 query embedding 等方式缓解。中期可以接 pgvector 或 Qdrant 做 ANN 检索。长期如果是多知识库、多租户、高 QPS，就需要专门向量数据库、metadata filter、hybrid search 和可观测指标。

Q12：后续如何评估真实 Embedding 模型？

A12：需要构造 query-evidence 评测集：每个 query 标注一个或多个相关 chunk。然后对不同 embedding 模型、chunking 策略和检索参数计算 Recall@K、Hit@K、MRR、Context Precision，同时记录 latency、成本、输入长度限制和中文效果。没有评测集之前，不应该说某个模型更好。

Q13：Hit@K / Recall@K / MRR 怎么算？

A13：Hit@K 表示 TopK 结果里是否命中至少一个相关 chunk；Recall@K 表示 TopK 找回的相关 chunk 数量占全部相关 chunk 数量的比例；MRR 是第一个相关结果排名倒数的平均值，例如第一个相关结果排第 2，贡献就是 1/2。query-evidence 评测集是这些指标的前提，因为它告诉系统哪些 chunk 是标准答案证据。没有评测就说检索效果提升，是不严谨的。

Q14：VIP 用户和普通用户在检索策略上可以怎么区分？

A14：普通用户可以使用较小 topK、baseline embedding、不 rerank、较小 context budget 和标准队列。VIP 用户可以使用更大 topK、更强 embedding model、rerank、更大 context budget、更详细 citation、更高优先级和更高 quota。这个项目当前还没做 VIP 策略，但 V2.7 可以基于现有 SearchRequest 和 metadata 扩展。

Q15：下一步如何升级？

A15：下一步 V2.3 可以做 RAG Answer with Citation，把 TopK chunk 拼成 context prompt，调用 MockLlmClient 返回 answer 和 citations。V2.4 做 Chunking Evaluation，V2.5 接真实 Embedding Provider，V2.6 做向量数据库选型，V2.7 再做 Retrieval Policy 和 VIP Search。

## 21. 可观测性指标

Embedding 指标：

- embedding request count
- embedding latency
- embedding failure count
- embedding vector dimension mismatch
- embedding provider/model 分布

Search 指标：

- search request count
- search latency P50 / P95 / P99
- candidate chunk count
- topK
- average score
- zero result count
- documentId filter 使用比例

Evaluation 指标：

- Recall@1
- Recall@3
- Recall@5
- MRR
- Context Precision
- Citation accuracy

这些是后续应该采集和评估的指标；V2.2 当前没有真实观测数据，因此不能编造 latency 或 Recall 指标。
