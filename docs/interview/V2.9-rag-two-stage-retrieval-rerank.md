# V2.9 RAG Two-Stage Retrieval & Rerank 面试文档

## 1. 版本摘要

V2.9 的版本名称是：

```text
V2.9 RAG Two-Stage Retrieval & Rerank
```

中文可以表述为：

```text
RAG 两阶段检索与重排序
```

这个版本的核心目标是：在已有 RAG Answer API 和 V2.8 检索质量评估框架基础上，引入 two-stage retrieval（两阶段检索）和 rerank（重排序）能力。

在 V2.9 之前，RAG 检索链路主要是：

```text
query
  -> query embedding
  -> VectorStore search topK
  -> topK chunks
  -> citations
  -> RAG prompt
  -> LLM generation
  -> answer
```

V2.9 将链路升级为：

```text
query
  -> query embedding
  -> VectorStore search candidateTopK
  -> Reranker rerank candidates
  -> select finalTopK
  -> citations
  -> RAG prompt
  -> LLM generation
  -> answer
```

也就是说，本版本把检索拆成两个阶段：

```text
第一阶段：VectorStore 召回更多候选 chunk；
第二阶段：Reranker 对候选 chunk 重新排序；
最后只选择 finalTopK 进入 prompt 和 citations。
```

这个版本不是直接接入复杂 CrossEncoder 模型，而是先建立可替换的 Reranker 抽象，并提供一个低依赖、确定性的 lexical reranker（词项重叠重排序器），确保默认测试稳定、链路可验证、后续可扩展。

同时，V2.9 复用 V2.8 的 evaluation benchmark（评估基准），支持 baseline vs rerank 对比，输出 before / after / delta 报告，用数据判断 rerank 是否改善检索质量。

---

## 2. 为什么现在做 V2.9

V2.8 已经完成了 RAG Retrieval Evaluation & Quality Benchmark（RAG 检索质量评估与基准测试）。

这意味着项目已经有了“量尺”：

```text
HitRate@K（命中率@K）
Recall@K（召回率@K）
Precision@K（精确率@K）
RR@K（倒数排名@K）
MRR（平均倒数排名）
per-case result（逐样本结果）
summary metrics（汇总指标）
JSON / Markdown report
```

有了评估能力之后，下一步就应该做真正的检索质量优化。

如果继续只做页面、文档或 demo，就会偏离 RAG 工程主线。

所以 V2.9 的合理位置是：

```text
V2.8：先建立检索质量评估能力；
V2.9：再引入重排序，用评估框架判断是否有改善。
```

这符合真实 RAG 系统的演进顺序：

```text
先能检索
再能评估
再做优化
再用评估证明优化是否有效
```

## 3. 本版本解决的问题

### 3.1 解决单阶段向量检索排序不够精细的问题

单阶段 VectorStore search 通常依赖向量相似度。

它适合快速召回语义相近的 chunk，但可能存在几个问题：

```text
相关 chunk 被召回了，但排名不靠前；
topK 里混入语义相近但细节不匹配的 chunk；
query 里有关键术语，但向量相似度没有充分体现；
一些重要片段排在 candidateTopK 内，但没有进入最终 topK。
```

V2.9 的思路是：

```text
第一阶段扩大召回；
第二阶段重新排序；
最终只选择更合适的 finalTopK。
```

这样可以在不改变底层 VectorStore 的前提下，增强检索排序能力。

---

### 3.2 解决“有评估但没有优化动作”的问题

V2.8 提供了评估框架，但它本身不改变检索质量。

V2.9 利用 V2.8 的评估框架来验证 rerank 策略：

```text
baseline：直接 VectorStore search topK；
rerank：VectorStore search candidateTopK -> rerank -> finalTopK；
```

然后比较：

```text
HitRate@K 是否变化；
Recall@K 是否变化；
Precision@K 是否变化；
MRR 是否变化；
哪些 case improved；
哪些 case regressed；
哪些 case unchanged。
```

这样项目不再只是“实现一个功能”，而是形成了完整的优化闭环。

---

### 3.3 解决后续接入强 reranker 的架构基础问题

如果一开始直接接 CrossEncoder，会出现很多额外复杂度：

```text
模型下载；
Python worker 协议；
推理延迟；
CPU / GPU 边界；
默认测试依赖；
服务部署；
超时与失败处理；
```

V2.9 先抽象 Reranker interface，把调用点、输入输出、metadata、evaluation 对比流程打通。

后续可以替换成：

```text
CrossEncoderReranker
LLM-based Reranker
ExternalApiReranker
Hybrid Reranker
```

但当前版本不需要承担这些复杂度。

## 4. 核心设计

### 4.1 Two-stage retrieval（两阶段检索）

V2.9 的核心链路是 two-stage retrieval：

```text
query
  -> embedding
  -> candidate retrieval
  -> rerank
  -> final selection
```

第一阶段：

```text
VectorStore search candidateTopK
```

它负责扩大召回范围。

第二阶段：

```text
Reranker rerank candidates
```

它负责在候选结果中重新排序。

最后：

```text
take finalTopK
```

只把 finalTopK 用于 citations 和 RAG prompt。

这种设计的好处是：

```text
不破坏现有 VectorStore；
不要求立刻接复杂模型；
可以在候选集合内做更细粒度排序；
可以用 V2.8 evaluation 量化前后变化；
后续可替换更强 reranker。
```

---

### 4.2 candidateTopK 与 finalTopK

V2.9 引入两个关键参数：

```text
candidateTopK：第一阶段召回候选数量；
finalTopK：最终进入 RAG prompt 的数量。
```

例如：

```text
candidateTopK = 20
finalTopK = 5
```

含义是：

```text
先从向量库召回 20 个候选 chunk；
再通过 reranker 重新排序；
最后选择前 5 个 chunk 构造 citations 和 prompt。
```

这个设计解决了一个常见问题：

```text
如果直接 topK=5，某些相关 chunk 可能排在第 6 到第 20 位，无法进入 prompt。
```

通过 candidateTopK 扩召回，reranker 有机会把这些相关 chunk 提到前面。

但 candidateTopK 不能无限大，因为：

```text
候选越多，rerank 成本越高；
后续如果接 CrossEncoder，延迟会明显增加；
候选过多也可能引入更多噪声。
```

所以 candidateTopK 和 finalTopK 需要配置化。

---

### 4.3 Reranker 抽象

V2.9 新增 Reranker 抽象。

它的核心职责是：

```text
输入 query 和 candidates；
输出 reranked results；
暴露 rerankerName 和 latency；
保持实现可替换。
```

输入一般包含：

```text
query
candidate results
finalTopK
```

candidate result 通常包含：

```text
originalRank
documentId / documentTitle
chunkId
content / contentSnippet
original vector score
metadata
```

输出包含：

```text
reranked results
rerankerName
latencyMs
```

这个抽象的价值是：

```text
RAG Service 不直接绑定某个具体 reranker；
当前可以用 lexical reranker；
后续可以替换 CrossEncoder；
evaluation 也可以复用同一套 reranker 接口；
测试可以 mock 或使用低依赖实现。
```

---

### 4.4 LexicalOverlapReranker（词项重叠重排序器）

V2.9 默认实现一个低依赖 reranker，例如：

```text
LexicalOverlapReranker
```

它基于 query 和 chunk content 的词项重叠进行打分。

大致思路是：

```text
对 query 做简单 token 化；
对 candidate content 做简单 token 化；
计算 query token 与 chunk token 的 overlap；
结合原始 vector score；
得到 rerank score；
按 rerank score 重新排序。
```

示例公式可以理解为：

```text
rerankScore = vectorScoreWeight * normalizedVectorScore
            + lexicalWeight * lexicalOverlapScore
```

这个 reranker 的优点是：

```text
无外部依赖；
不需要 Python worker；
不下载模型；
结果确定性强；
单元测试容易覆盖；
适合验证 rerank 链路。
```

它的局限也很明确：

```text
它不是语义级 reranker；
不能理解复杂同义表达；
对中文分词能力有限；
不保证所有 case 都提升；
主要用于建立工程骨架。
```

面试时要强调：

```text
这个版本不是宣称 lexical reranker 很强，而是先建立可替换的 rerank 架构和评估闭环。
```

## 5. RAG Answer API 集成

### 5.1 默认行为不变

V2.9 的一个关键边界是：

```text
默认 rerank disabled，不改变现有 RAG 行为。
```

也就是说，如果配置为：

```properties
rag.rerank.enabled=false
```

RAG Answer API 仍然按原来的方式执行：

```text
query embedding
  -> VectorStore search topK
  -> citations
  -> prompt
  -> answer
```

这样可以保证兼容性。

---

### 5.2 启用 rerank 后的行为

如果配置为：

```properties
rag.rerank.enabled=true
```

则执行：

```text
query embedding
  -> VectorStore search candidateTopK
  -> Reranker rerank candidates
  -> take finalTopK
  -> citations
  -> prompt
  -> answer
```

此时 citations 的顺序必须来自 rerank 后顺序。

也就是说：

```text
用户看到的 citation[1] 应该是 reranker 排在第一的 chunk。
```

这点很重要，因为 citations 是用户理解答案来源的入口。

---

### 5.3 不改变必填请求字段

V2.9 不强制修改 `POST /rag/answers` 的必填请求字段。

原因是：

```text
V2.7 已经定义了 RAG Answer API；
V2.7.1 Web UI 已经调用该接口；
如果强行增加必填字段，会破坏现有调用方；
rerank 属于检索策略，优先通过配置控制。
```

所以本版本更推荐通过配置控制：

```properties
rag.rerank.enabled
rag.rerank.candidate-top-k
rag.rerank.final-top-k
rag.rerank.provider
```

而不是要求前端或用户每次请求都传 rerank 参数。

## 6. Retrieval Metadata 设计

V2.9 需要增强 retrieval metadata（检索元数据），让用户和 reviewer 能看到 rerank 是否启用。

建议 metadata 包含：

```text
rerankEnabled
rerankerName
candidateTopK
finalTopK
rerankLatencyMs
```

如果实现支持，也可以在 retrieved items 或 citations 中展示：

```text
originalRank
rerankedRank
originalScore
rerankScore
```

这些字段的意义是：

```text
originalRank：向量检索原始排名；
rerankedRank：重排序后排名；
originalScore：向量检索分数；
rerankScore：重排序分数。
```

这让排序变化变得可解释。

例如，一个 chunk 原来排第 8，rerank 后排第 2，就能在 metadata 中看到这种变化。

这对调试和面试展示都很有价值。

## 7. Evaluation 对比设计

V2.9 复用 V2.8 的 evaluation framework（评估框架），新增 baseline vs rerank 对比。

### 7.1 Baseline

baseline 表示原始检索策略：

```text
VectorStore search topK
```

不经过 reranker。

### 7.2 Rerank

rerank 表示两阶段检索策略：

```text
VectorStore search candidateTopK
  -> Reranker rerank
  -> finalTopK
```

### 7.3 对比报告

报告中应该包含：

```text
baseline summary metrics
rerank summary metrics
delta metrics
```

delta 表示前后变化，例如：

```text
delta HitRate@K
delta Recall@K
delta Precision@K
delta MRR
```

per-case 层面可以展示：

```text
caseId
baseline hit / recall / precision / rr
rerank hit / recall / precision / rr
status: improved / regressed / unchanged
baseline ranks
reranked ranks
```

这个对比能力是 V2.9 的关键价值。

因为 rerank 不应该靠感觉判断，而应该用 V2.8 的指标体系来验证。

## 9. 关键取舍

### 9.1 为什么不直接接 CrossEncoder

CrossEncoder reranker 通常比简单 lexical reranker 更强，但不适合在 V2.9 直接接入。

原因是：

```text
需要额外模型；
可能需要 Python worker；
模型下载和推理成本高；
延迟更大；
测试和部署复杂；
Java 与 Python 协议需要重新设计；
失败和超时处理更复杂。
```

所以 V2.9 的选择是：

```text
先抽象 Reranker；
先打通 two-stage retrieval 链路；
先用低依赖 reranker 做可测实现；
后续再替换强模型。
```

这是一种工程上更稳的演进方式。

---

### 9.2 为什么不做 Hybrid Search

Hybrid Search（混合检索）通常涉及：

```text
dense vector search；
sparse / keyword search；
BM25；
score fusion；
RRF / DBSF；
Qdrant sparse vector；
```

这是另一个大的检索系统升级。

如果把 Hybrid Search 和 Rerank 混在同一个版本，会导致边界过大。

所以 V2.9 只做 two-stage retrieval + rerank。

Hybrid Search 可以作为后续版本：

```text
V3.x RAG Hybrid Search
```

---

### 9.3 为什么不改 Web UI

V2.9 不改 Web UI。

原因是：

```text
rerank 属于后端检索策略；
Web UI 只负责输入 query 和展示 answer/citations/metadata；
现有 metadata 已经可以展示 rerank 信息；
不需要为了本版本增加前端开关。
```

如果以后需要前端选择检索策略，可以再做独立版本。

---

### 9.4 为什么不做 answer quality evaluation

V2.9 优化的是 retrieval quality（检索质量）。

Answer quality evaluation（答案质量评估）涉及：

```text
标准答案；
事实一致性；
faithfulness（忠实度）；
citation correctness（引用正确性）；
LLM-as-judge；
人工评分；
```

复杂度更高。

当前阶段应先把检索链路优化和评估闭环做好，再考虑答案质量评估。

## 13. 面试高强度追问与回答

### 13.1 为什么要做 two-stage retrieval？

回答：

```text
单阶段向量检索适合快速召回，但排序不一定足够精细。有些相关 chunk 可能被召回到 candidateTopK 中，但没有进入最终 topK。

所以我把链路拆成两阶段：第一阶段用 VectorStore 扩大候选召回，第二阶段用 Reranker 对候选结果重新排序，最后选择 finalTopK 进入 prompt。这样可以在不改底层 VectorStore 的前提下提升排序灵活性。
```

---

### 13.2 candidateTopK 和 finalTopK 的区别是什么？

回答：

```text
candidateTopK 是第一阶段向量检索召回的候选数量，目的是扩大召回范围。finalTopK 是最终进入 citations 和 RAG prompt 的数量。

比如 candidateTopK=20、finalTopK=5，表示先召回 20 个候选 chunk，然后 rerank，再选前 5 个进入最终回答上下文。
```

---

### 13.3 为什么不直接把 topK 调大？

回答：

```text
直接把 topK 调大会把更多 chunk 放进 prompt，但这会增加上下文噪声和 token 成本。Rerank 的思路是先扩大候选范围，但最终仍然只选择 finalTopK。

这样既有机会找回更多相关候选，又不会把所有候选都塞进 prompt。
```

---

### 13.4 为什么不直接接 CrossEncoder？

回答：

```text
CrossEncoder 通常更强，但会引入模型下载、Python worker、推理延迟、部署和测试复杂度。当前版本的目标是先建立可替换的 rerank 架构和 two-stage retrieval 链路。

所以我先抽象 Reranker 接口，并实现一个低依赖的 lexical reranker。等链路、metadata 和 evaluation 对比都稳定后，再接 CrossEncoder 会更安全。
```

---

### 13.5 LexicalOverlapReranker 有什么局限？

回答：

```text
它主要基于 query 和 chunk content 的词项重叠，不是真正的深度语义 reranker。它对同义词、复杂语义改写和中文分词的处理能力有限。

它的价值不是证明 lexical reranker 最强，而是提供一个低依赖、确定性、可测试的默认实现，用来打通 Reranker 抽象、RAG 集成和 evaluation 对比链路。
```

---

### 13.6 rerank 一定能提升指标吗？

回答：

```text
不一定。V2.9 不承诺 rerank 一定提升所有指标。不同数据集、query 和 scoring 策略下，rerank 可能提升 MRR，也可能让某些 case 退化。

所以我把 V2.8 的 evaluation framework 复用进来，输出 baseline vs rerank 的 before / after / delta，让指标真实反映效果，而不是主观判断。
```

---

### 13.7 如果 Recall 提升但 Precision 下降，怎么理解？

回答：

```text
这说明系统找回了更多 expected relevant items，但同时也引入了更多无关结果。对于 RAG 来说，这可能意味着上下文覆盖更全，但 prompt 噪声也更大。

是否接受这种变化，要结合具体场景。如果任务更怕漏召回，可以接受 Recall 提升；如果任务更怕噪声干扰，就要更关注 Precision。
```

---

### 13.8 如果 MRR 提升说明什么？

回答：

```text
MRR 提升说明第一个相关结果通常排得更靠前。对 RAG 来说，这很重要，因为排在前面的 chunk 更可能影响 prompt 构造和最终答案。

如果 HitRate 没变但 MRR 提升，说明系统命中的 case 数量没变，但相关结果排序更好了。
```

---

### 13.9 为什么默认关闭 rerank？

回答：

```text
为了保证兼容性和可控性。V2.7 已经有稳定的 RAG Answer API，如果默认开启 rerank，就会改变已有行为，也可能影响测试和演示结果。

所以我把 rerank 设计成配置可选。默认关闭时保持原始 VectorStore topK 行为；显式启用后才走 candidateTopK -> rerank -> finalTopK。
```

---

### 13.10 retrieval metadata 为什么要暴露 rerank 信息？

回答：

```text
因为 rerank 会改变检索排序。如果只返回最终 answer 和 citations，使用者不知道排序是怎么变化的，也无法排查某个 chunk 为什么进入或没进入 prompt。

通过 metadata 暴露 rerankEnabled、rerankerName、candidateTopK、finalTopK、originalRank、rerankedRank、originalScore 和 rerankScore，可以让检索链路更可解释。
```

---

### 13.11 这个版本体现什么工程能力？

回答：

```text
它体现的是 RAG 系统优化闭环能力。V2.8 先建立 evaluation benchmark，V2.9 再引入 two-stage retrieval 和 rerank，并用 benchmark 对比优化前后指标。

同时我没有直接引入复杂 CrossEncoder，而是先做 Reranker 抽象、低依赖实现、metadata 暴露和 evaluation 对比。这体现了可替换架构、边界控制和质量工程思维。
```

### 13.12 后续如何升级成更强 reranker？

回答：

```text
后续可以新增 CrossEncoderReranker 或 Python Reranker Worker。由于 V2.9 已经抽象了 Reranker 接口，RAG service 和 evaluation runner 不需要重写，只需要新增 provider 实现，并处理模型加载、超时、批量推理和降级逻辑。

然后继续用 V2.8/V2.9 的 baseline vs rerank benchmark 验证它是否真的提升指标。
```

## 16. 总结表达

这个版本可以用一段话总结：

```text
在 V2.9 中，我将 RAG 检索链路从单阶段 VectorStore topK 检索升级为 two-stage retrieval：先用 candidateTopK 扩大候选召回，再通过 Reranker 对候选 chunk 重排序，最后选择 finalTopK 构造 citations 和 RAG prompt。

我没有直接接入复杂 CrossEncoder，而是先抽象 Reranker 接口，并实现一个低依赖的 LexicalOverlapReranker，保证默认测试不依赖 Qdrant、Docker、Python worker、外部模型或 API。同时，我复用 V2.8 的 evaluation benchmark，支持 baseline vs rerank 对比，输出 HitRate@K、Recall@K、Precision@K、MRR 的 before / after / delta 报告。

这个版本体现的是 RAG 系统的检索优化闭环：先评估，再优化，再用指标验证优化效果。
```
