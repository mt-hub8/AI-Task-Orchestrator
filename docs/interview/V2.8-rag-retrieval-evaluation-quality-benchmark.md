# V2.8 RAG Retrieval Evaluation & Quality Benchmark 面试文档

## 1. 版本摘要

V2.8 的版本名称是：

```text
V2.8 RAG Retrieval Evaluation & Quality Benchmark
```

中文可以表述为：

```text
RAG 检索质量评估与基准测试
```

这个版本的核心目标是：让项目不再只是“能检索、能回答”，而是能够**量化评估 RAG 检索质量**。

在 V2.7 中，系统已经完成了 RAG Answer API with Citations，可以完成：

```text
query
  -> query embedding
  -> VectorStore search
  -> topK chunks
  -> citations
  -> RAG prompt
  -> LLM generation
  -> answer
```

在 V2.7.1 和 V2.7.2 中，系统又补齐了 Web Demo 和可复现演示路径，使用户能够在浏览器中体验 RAG 问答。

但是这些能力仍然有一个不足：

```text
系统能回答，不代表检索质量一定稳定；
单次回答成功，不代表整体检索效果可靠；
没有评估指标，就无法判断改动是否真的提升了 RAG 质量。
```

所以 V2.8 新增了 RAG retrieval evaluation（RAG 检索评估）能力。它通过 evaluation dataset（评估数据集）批量运行检索，计算 HitRate@K（命中率@K）、Recall@K（召回率@K）、Precision@K（精确率@K）、RR@K（倒数排名@K）和 MRR（平均倒数排名），并输出 per-case result（逐样本结果）、summary metrics（汇总指标）、JSON report 和 Markdown report。

这个版本的定位是：

> 把 RAG 系统从“功能可用”推进到“质量可评估”。

---

## 2. 为什么现在做 V2.8

在 V2.8 之前，项目已经具备了比较完整的 RAG 工程链路：

```text
文档 / chunk
embedding
VectorStore
Qdrant manual verification
Qdrant benchmark result capture
Embedding Cache
Embedding Cache Metrics
RAG Answer API
citations
Minimal Web UI
Manual Demo Golden Path
```

这些版本解决了“系统能不能跑通”的问题。

但企业级 RAG 系统还必须回答另一个问题：

```text
检索效果到底好不好？
```

例如：

- topK 设置为 3、5、10 时，检索质量有什么差异？
- 调整 chunk size 后，Recall@K 是否提升？
- 更换 embedding model 后，MRR 是否提升？
- Qdrant 检索和 exact baseline 的质量差距是多少？
- 某些 query 为什么没有命中 expected chunk？
- 相关 chunk 是排在第 1 位，还是排在第 5 位？
- 返回结果里有多少噪声？

如果没有 evaluation benchmark，这些问题只能靠人工感觉判断。

V2.8 的意义就是建立一套可复用的评估框架，让后续 RAG 优化可以有数据依据。

---

## 3. 本版本解决的问题

### 3.1 解决“只看单次效果”的问题

单次 RAG 问答成功，并不能说明系统整体质量好。

可能只是某个 query 比较简单，也可能是 demo 数据刚好命中。

V2.8 引入 evaluation dataset，每条 case 都包含：

```text
caseId
query
topK
expected relevant items
```

系统会对每条 query 执行检索，然后把检索结果和 expected relevant items 进行匹配，最后计算指标。

这样可以从单点验证变成批量评估。

---

### 3.2 解决“无法质量回归”的问题

后续如果修改：

```text
chunk size
chunk overlap
embedding model
VectorStore 实现
Qdrant 参数
topK
similarity threshold
rerank
hybrid search
```

都可能影响检索效果。

如果没有评估系统，就很难判断改动是提升还是退化。

V2.8 提供了固定数据集和固定指标，所以后续每次修改 RAG 策略，都可以重新运行 benchmark，比较 summary metrics 和 per-case result。

这就是 quality regression（质量回归）能力。

---

### 3.3 解决“面试表达不够硬”的问题

普通项目说：

```text
我做了 RAG。
```

工程化项目可以说：

```text
我不仅实现了 RAG Answer API，还设计了 retrieval evaluation benchmark，通过带 expected relevant chunks 的数据集批量评估检索质量，计算 HitRate@K、Recall@K、Precision@K 和 MRR，并输出 JSON / Markdown 报告，用于后续 embedding、chunking、topK 和 VectorStore 策略优化。
```

这体现的不只是功能实现，而是系统评估、质量度量和工程迭代能力。

## 4. 核心设计

### 4.1 Evaluation Dataset 设计

V2.8 新增 evaluation dataset，用来描述一组可评估的检索样本。

每个 case 至少包含：

```text
caseId：样本 ID
query：用户问题
topK：检索返回数量
expected：期望命中的相关内容
```

expected relevant item 不建议强依赖固定 chunkId。

原因是：

```text
chunkId 可能随着数据库重建、文档重新摄入、chunking 策略调整而变化。
```

因此更稳妥的设计是支持：

```text
documentId / documentTitle，可选；
expectedChunkId，可选；
chunkContains 文本片段匹配。
```

其中 `chunkContains` 是一个比较实用的设计：只要 retrieved chunk 内容中包含指定文本片段，就认为命中了该 expected item。

这样评估数据集不会和数据库物理 ID 强绑定，更适合本地重建和长期维护。

---

### 4.2 Dataset Loader 设计

Dataset Loader 负责从 JSON 文件读取 evaluation cases。

它需要完成：

```text
读取 JSON；
解析 caseId / query / topK / expected；
校验必要字段；
字段缺失时给出清晰错误；
把原始 JSON 转成 Java 内部 evaluation case 对象。
```

这个模块是低依赖的，不应该依赖 Qdrant、worker 或真实 embedding model。

因此它可以被默认单元测试覆盖。

---

### 4.3 Retrieval Evaluation Runner 设计

Retrieval Evaluation Runner 是实际执行评估的入口。

它的核心流程是：

```text
加载 evaluation dataset
  -> 遍历每个 case
  -> 对 query 生成 embedding
  -> 调用 VectorStore search(topK)
  -> 获得 retrieved results
  -> 和 expected items 匹配
  -> 计算 per-case metrics
  -> 聚合 summary metrics
  -> 输出 JSON report
  -> 输出 Markdown report
```

Runner 必须是 opt-in（显式启用），不能默认执行。

原因是它可能依赖真实 Qdrant、Local Embedding Worker 和 embedding model，这些都属于重依赖，不适合污染默认测试和普通启动流程。

---

### 4.4 Expected Matching 设计

Expected matching 是本版本的关键设计之一。

它要判断某个 retrieved chunk 是否命中某个 expected item。

推荐匹配方式包括：

```text
expectedChunkId 匹配；
documentId / documentTitle 匹配；
chunkContains 文本片段匹配。
```

其中最重要的是 `chunkContains`，因为它对本地 demo 和数据库重建更稳定。

示例：

```json
{
  "caseId": "rag-eval-001",
  "query": "RAG Answer API 的核心流程是什么？",
  "topK": 5,
  "expected": [
    {
      "documentTitle": "RAG Demo Source",
      "chunkContains": "query embedding"
    },
    {
      "documentTitle": "RAG Demo Source",
      "chunkContains": "citations"
    }
  ]
}
```

如果 topK 结果中有 chunk 内容包含 `query embedding` 或 `citations`，就可以判断命中对应 expected item。

## 5. 指标设计

V2.8 的指标聚焦 retrieval quality（检索质量），不是 answer quality（答案质量）。

### 5.1 HitRate@K（命中率@K）

HitRate@K 关注的是：

```text
topK 检索结果中是否至少命中一个 expected relevant item。
```

单个 case 的 hit 判断：

```text
只要 topK 中有一个 relevant result，则 hit=true；
否则 hit=false。
```

整体 HitRate@K：

```text
HitRate@K = hit case 数 / total case 数
```

它回答的问题是：

```text
系统是否至少能找回一个相关结果？
```

这个指标适合快速判断检索是否“基本可用”。

---

### 5.2 Recall@K（召回率@K）

Recall@K 关注的是：

```text
所有 expected relevant items 中，有多少被 topK 检索结果找回了。
```

单个 case 的 Recall@K：

```text
Recall@K = matched expected item 数 / expected item 总数
```

例如某个 query 有 4 个 expected relevant chunks，topK 中命中了 3 个：

```text
Recall@K = 3 / 4 = 0.75
```

它回答的问题是：

```text
相关内容找回得全不全？
```

在 RAG 系统里，Recall@K 很重要，因为如果相关内容没有被检索回来，后面的 LLM generation 再强也无法基于正确上下文回答。

---

### 5.3 Precision@K（精确率@K）

Precision@K 关注的是：

```text
topK 返回结果中，有多少是 relevant。
```

常见计算方式有两种：

```text
Precision@K = relevant retrieved count / K
```

或者：

```text
Precision@K = relevant retrieved count / actual retrieved count
```

如果系统返回数量一定等于 K，两种通常等价。

如果实际返回数量可能小于 K，需要在代码和文档中明确采用哪一种。

Precision@K 回答的问题是：

```text
返回结果里噪声多不多？
```

在 RAG 场景中，如果 Precision@K 很低，说明 prompt 中可能混入大量无关上下文，影响生成质量。

---

### 5.4 RR@K（Reciprocal Rank@K，倒数排名@K）

RR@K 关注的是：

```text
第一个 relevant result 出现在第几名。
```

计算方式：

```text
如果第一个 relevant result 排名为 r，则 RR = 1 / r；
如果 topK 中没有 relevant result，则 RR = 0。
```

例如：

```text
第 1 位命中：RR = 1
第 2 位命中：RR = 0.5
第 5 位命中：RR = 0.2
没有命中：RR = 0
```

它回答的问题是：

```text
相关结果是不是排得足够靠前？
```

这对 RAG 很重要，因为排名越靠前的 chunk 通常越可能被放在 prompt 的重要位置。

---

### 5.5 MRR（Mean Reciprocal Rank，平均倒数排名）

MRR 是所有 case 的 RR 平均值。

计算方式：

```text
MRR = 所有 case 的 RR 总和 / total case 数
```

它衡量的是整体 ranking quality（排序质量）。

如果 MRR 较高，说明相关结果通常排得靠前。

如果 HitRate@K 高但 MRR 低，说明系统能找到相关内容，但经常排在后面，这可能影响最终 RAG answer 的质量。

## 6. Report 输出设计

V2.8 输出两类报告。

### 6.1 JSON Report

JSON report 面向机器读取和后续对比。

它适合用于：

```text
保存 benchmark 原始结果；
后续做多次实验对比；
给自动化脚本读取；
作为后续 dashboard 的数据来源。
```

JSON report 通常包含：

```text
datasetName / datasetPath
runAt
topK
provider
model
dimension
vectorStore
caseResults
summaryMetrics
latency
```

其中 caseResults 记录每个 case 的详细结果，summaryMetrics 记录整体指标。

---

### 6.2 Markdown Report

Markdown report 面向人工阅读和面试展示。

它适合提交到：

```text
docs/evaluation/reports/
```

内容可以包括：

```text
评估时间；
数据集路径；
运行配置；
summary metrics；
per-case result table；
failed / missed cases；
指标解释；
注意事项。
```

Markdown report 的价值是：

```text
让 reviewer 不需要读 JSON，也能快速看懂本次评估结果。
```

这对项目展示非常有帮助。

## 8. Manual / Opt-in 运行设计

V2.8 的真实 evaluation 应该通过显式配置运行，例如：

```powershell
.\mvnw.cmd spring-boot:run "-Dspring-boot.run.profiles=docker,qdrant,evaluation" "-Dspring-boot.run.arguments=--rag.eval.enabled=true"
```

具体 profile 和参数以项目实际实现为准。

```text

```

这样可以避免 evaluation 影响日常开发，同时保留真实链路验证能力。

## 9. 关键取舍

### 9.1 为什么只评估 retrieval quality，不评估 answer quality

V2.8 聚焦 retrieval quality（检索质量），不评估 LLM answer quality（答案质量）。

原因是 RAG 系统可以拆成两层：

```text
retrieval：有没有找回正确上下文；
generation：模型是否基于上下文生成好答案。
```

如果检索没做好，生成阶段很难补救。

因此先评估 retrieval 是更基础、更稳定、也更低依赖的选择。

Answer quality evaluation 通常还需要：

```text
标准答案；
人工评分；
LLM-as-judge；
事实一致性判断；
引用支持度判断；
幻觉检测。
```

这些复杂度更高，适合作为后续版本。

---

### 9.2 为什么不做 Web UI Dashboard

本版本不做 evaluation dashboard。

原因是：

```text
当前目标是建立评估能力本身；
JSON / Markdown report 已经足够支持分析和面试展示；
Web dashboard 会引入前端、API、数据管理和权限问题；
会扩大版本边界。
```

后续如果评估结果足够多，再做 dashboard 才有意义。

---

### 9.3 为什么不接 Prometheus / Grafana

Prometheus / Grafana 更适合监控线上运行时指标，例如：

```text
QPS
latency
error rate
cache hit rate
resource usage
```

V2.8 的 retrieval evaluation 更偏离线 benchmark 和质量分析。

所以当前先输出 report，不接监控系统。

---

### 9.4 为什么不做 rerank / hybrid search

Rerank 和 hybrid search 都是优化检索质量的策略。

但在引入优化策略之前，应该先有评估能力。

更合理的顺序是：

```text
先建立 evaluation benchmark；
再引入 rerank / hybrid search；
最后用同一套 benchmark 判断是否提升。
```

所以 V2.8 不做检索优化，而是先做评估框架。

## 13. 面试高强度追问与回答

### 13.1 为什么要做 RAG evaluation？

回答：

```text
因为 RAG 系统不能只看单次问答效果。单次回答成功，可能只是某个 query 刚好命中，并不能说明整体检索质量稳定。

所以我在 V2.8 设计了 retrieval evaluation benchmark，通过带 expected relevant items 的数据集批量评估检索结果，计算 HitRate@K、Recall@K、Precision@K 和 MRR。这样后续修改 chunking、embedding model、topK 或 VectorStore 策略时，可以用同一套 benchmark 做质量回归。
```

---

### 13.2 为什么先评估 retrieval，而不是直接评估 answer？

回答：

```text
RAG 可以拆成 retrieval 和 generation 两层。retrieval 负责找上下文，generation 负责基于上下文生成答案。如果 retrieval 没有找回正确内容，后面的 LLM 再强也很难生成可靠答案。

Answer quality evaluation 还涉及标准答案、人工评分、LLM-as-judge、事实一致性和引用支持度判断，复杂度更高。所以我先从 retrieval quality 入手，这是更基础、更可控的一层。
```

---

### 13.3 HitRate@K 和 Recall@K 有什么区别？

回答：

```text
HitRate@K 看的是 topK 结果中是否至少命中一个相关结果。只要命中一个，这个 case 就算 hit。

Recall@K 看的是所有 expected relevant items 中，有多少被 topK 找回。比如一个 case 有 4 个 expected chunks，检索结果命中 3 个，那么 Recall@K 是 0.75。

简单说，HitRate@K 关注“有没有命中”，Recall@K 关注“命中了多少”。
```

---

### 13.4 Precision@K 评价什么？

回答：

```text
Precision@K 评价 topK 返回结果里有多少是相关结果。它反映检索结果的噪声程度。

在 RAG 场景里，Precision@K 很重要，因为如果 topK 中混入太多无关 chunk，这些内容会进入 prompt，可能干扰 LLM 生成，导致回答不准确。
```

---

### 13.5 MRR 评价什么？

回答：

```text
MRR 评价相关结果是否排得靠前。它先计算每个 case 的 RR，也就是第一个相关结果排名 r 对应 1/r；如果没有命中则是 0。然后对所有 case 的 RR 求平均。

如果 MRR 高，说明相关结果通常排在前面。对 RAG 来说，这很重要，因为排在前面的 chunk 更可能对最终 prompt 和答案产生影响。
```

---

### 13.6 为什么 expected item 不强依赖 chunkId？

回答：

```text
因为 chunkId 很容易变化。比如重新导入文档、重建数据库、调整 chunking 策略，都可能导致 chunkId 变化。如果 evaluation dataset 强依赖固定 chunkId，就会变得很脆弱。

所以我支持 documentTitle / documentId 和 chunkContains 这种更稳定的匹配方式。这样即使底层 ID 变化，只要内容语义和片段还在，评估仍然可以运行。
```

---

### 13.7 为什么输出 JSON 和 Markdown 两种报告？

回答：

```text
JSON report 适合机器读取，后续可以用于自动对比、多次实验记录或 dashboard 数据源。

Markdown report 适合人工阅读和面试展示，可以直接查看 summary metrics、per-case result 和 failed cases。

一个服务自动化，一个服务可读性，所以两种都保留。
```

---

### 13.8 为什么真实 evaluation 不放进默认 mvn test？

回答：

```text
因为真实 evaluation 依赖 Qdrant、Python Embedding Worker、真实 embedding model，可能还需要 Docker 和模型下载。这些依赖会让默认测试变慢、不稳定，也不适合普通开发环境和 CI。

所以我的设计是默认测试只覆盖 dataset loader、metrics calculator、result aggregation 等低依赖逻辑；真实 Qdrant / worker evaluation 通过 manual opt-in 执行。这样既保证日常测试稳定，也保留真实链路验证能力。
```

---

### 13.9 这个版本体现了什么工程能力？

回答：

```text
它体现的是 RAG 系统的质量工程能力。很多项目只做到“能问答”，但缺少评估闭环。我在这个版本里把 evaluation dataset、expected matching、metrics calculator、evaluation runner 和 report writer 串起来，让检索效果可以被量化、比较和回归。

这说明我不是只会调 API，而是在考虑如何让 RAG 系统可评估、可优化、可演进。
```

---

### 13.10 如果后续要优化 RAG，V2.8 怎么发挥作用？

回答：

```text
后续如果我要调整 chunk size、更换 embedding model、修改 topK、引入 rerank 或 hybrid search，都可以先跑 V2.8 的 benchmark。然后比较 HitRate@K、Recall@K、Precision@K 和 MRR 是否提升。

如果 Recall 提升但 Precision 下降，就说明召回更多相关内容的同时也引入了更多噪声。如果 HitRate 不变但 MRR 提升，说明相关结果排得更靠前。这些指标可以指导后续优化方向。
```

## 14. 后续演进路线

V2.8 之后，可以沿以下方向继续演进。

### 14.1 Rerank Evaluation

引入 reranker 后，可以比较：

```text
VectorStore 原始排序
Rerank 后排序
MRR 是否提升
Precision@K 是否提升
```

这会让系统从 dense retrieval 进一步升级为 retrieval + rerank 架构。

---

### 14.2 Hybrid Search Evaluation

后续可以引入 hybrid search：

```text
BM25 / keyword search
dense vector search
score fusion
```

然后用 V2.8 benchmark 比较 hybrid search 是否提升 Recall@K 和 HitRate@K。

---

### 14.3 Chunking Strategy Benchmark

可以比较不同 chunking 策略：

```text
chunk size = 300 / 500 / 800
overlap = 50 / 100 / 150
按段落切分
按标题切分
semantic chunking
```

用同一套 evaluation dataset 判断质量变化。

---

### 14.4 Answer Quality Evaluation

在 retrieval evaluation 稳定后，再做 answer quality evaluation。

可能指标包括：

```text
faithfulness（忠实度）
answer relevance（答案相关性）
context support（上下文支持度）
citation correctness（引用正确性）
hallucination rate（幻觉率）
```

这通常需要标准答案、人工评分或 LLM-as-judge。

---

### 14.5 Evaluation Dashboard

当 report 数量增多后，可以做 dashboard：

```text
展示历史 benchmark；
比较不同配置；
查看失败 case；
查看指标趋势；
筛选 bad cases。
```

但这应该建立在 evaluation 数据和报告已经稳定的基础上。

## 16. 总结表达

这个版本可以用一段话总结：

```text
在 V2.8 中，我为 RAG 系统新增了 Retrieval Evaluation & Quality Benchmark。它通过 evaluation dataset 批量运行 VectorStore 检索，并基于 expected relevant items 计算 HitRate@K、Recall@K、Precision@K、RR@K 和 MRR，同时输出 per-case result、summary metrics、JSON report 和 Markdown report。

这个版本不评估 LLM answer quality，也不做 dashboard、rerank 或 hybrid search，而是先建立低依赖、可复用的 retrieval quality evaluation 框架。默认 `mvn test` 只覆盖 dataset loader、metrics calculator 和结果聚合等纯逻辑，真实 Qdrant / worker evaluation 通过 manual opt-in 执行。它让项目从“RAG 能回答”进一步升级为“RAG 检索质量可以被量化评估和回归”。
```
