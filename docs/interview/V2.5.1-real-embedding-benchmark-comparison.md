# V2.5.1 Real Embedding Benchmark Comparison 面试文档

## 0. 版本摘要

V2.5.1 新增测试级 Embedding Provider Benchmark Comparison Harness，复用 V2.4.x 的 retrieval benchmark，对 baseline provider 和 candidate provider 做同基准 retrieval metrics 对比。

一句话定位：

```text
V2.5.1 把 EmbeddingProvider 从“可接入”推进到“可评估”。
```

本版本边界：

- 不新增生产 API；
- 不新增数据库表；
- 不做 Evaluation Persistence；
- 不做 Vector DB；
- 不做 ANN；
- 不做 Rerank；
- 不做 Hybrid Search；
- 不做 Local Embedding Worker；
- 不做 Faithfulness / Answer Relevance；
- 不做 LLM-as-a-judge；
- 不改 README；
- 默认测试不调用外部 API；
- 默认测试不需要 `OPENAI_API_KEY`；
- 不声称 Real Embedding 一定优于 Mock。

真实代码中，V2.5.1 的核心测试类是：

```text
src/test/java/com/tuoman/ai_task_orchestrator/evaluation/EmbeddingProviderBenchmarkComparisonTest.java
```

它使用 `MockEmbeddingClient` 作为 baseline provider，使用测试内部的 `FakeCandidateEmbeddingProvider` 作为 candidate provider。这个 fake provider deterministic、无外部依赖，只验证 comparison pipeline，不代表真实 OpenAI embedding 效果。

## 1. 本版本解决的问题

V2.5 已经完成真实 Embedding Provider 接入，包括：

- `EmbeddingProvider` 抽象；
- 默认 `mock` provider；
- `OpenAiCompatibleEmbeddingProvider`；
- provider / model / dimension metadata；
- search 时避免不同 embedding space 混用。

但是 V2.5 只说明“可以接入 provider”，还没有说明“不同 provider 能不能在同一套 benchmark 下评估”。

V2.5.1 解决的是 provider comparison 闭环：

```text
same corpus
same benchmark cases
same topKValues
same chunking strategy
same RetrievalEvaluationService
different EmbeddingProvider
-> metrics comparison
```

核心回答：

接入真实 Embedding 后，不能只说“真实模型理论上更好”，而要用固定 benchmark 对比：

- Recall@K；
- Precision@K；
- HitRate@K；
- MRR / RR@K；
- NDCG@K；
- ContextPrecision@K。

V2.5.1 先把这个 comparison harness 打通，但默认不调用真实 OpenAI API。

## 2. 为什么现在做 V2.5.1

依赖关系如下：

```text
V2.4 Retrieval Evaluation Harness
-> V2.4.1 Metrics Completion
-> V2.4.2 Evaluation Dataset Seed
-> V2.4.3 Fixed vs Adaptive Chunking Comparison
-> V2.4.4 Benchmark Runner / Evidence Mapper
-> V2.5 Real Embedding Provider
-> V2.5.1 Provider Benchmark Comparison
```

没有 V2.4.x，V2.5 只能证明：

```text
Embedding API 能调通。
```

有了 V2.4.x，V2.5.1 可以证明：

```text
不同 provider 可以被同一套 benchmark 评估。
```

这一步很关键。真实 embedding provider 的价值不应该靠主观判断，而应该放到固定 corpus、固定 query、固定 expected evidence、固定 metrics 的实验框架里观察。

## 3. 方案选择

### 方案 A：默认测试直接调用 OpenAI

优点：

- 最接近真实 provider 对比；
- 能直接生成真实 embedding；
- 可以手工观察真实 metrics。

缺点：

- CI 不稳定；
- 需要 API key；
- API 调用会收费；
- 外部网络不稳定；
- 服务可能限流；
- 模型服务行为可能变化；
- API key 有泄露风险；
- 不适合作为默认 Maven test。

结论：不推荐作为默认测试。

### 方案 B：只写手工说明，不写 comparison harness

优点：

- 最安全；
- 不产生外部费用；
- 不引入测试复杂度。

缺点：

- 没有工程闭环；
- 无法验证 provider comparison pipeline；
- 面试表达停留在“理论上可以比较”。

结论：不推荐。

### 方案 C：test-level comparison harness + fake candidate provider + optional external benchmark

当前采用这个方向。

做法：

- 默认测试使用 `MockEmbeddingClient` 和 test-only `FakeCandidateEmbeddingProvider`；
- 验证 comparison pipeline；
- 复用 retrieval benchmark；
- 计算 baseline / candidate metrics；
- 计算 delta metrics；
- 不断言 candidate 一定优于 baseline；
- 真实 OpenAI-compatible benchmark 作为手工 external 流程，不进入默认 CI。

优点：

- CI 稳定；
- 不需要 API key；
- 不产生外部费用；
- comparison pipeline 可测试；
- 真实 provider 可手工运行；
- 不提前引入生产 API、持久化、权限和成本控制。

缺点：

- `FakeCandidateEmbeddingProvider` 不代表真实语义质量；
- 真实 provider comparison 需要手工配置 API key。

结论：当前推荐方案。

### 方案 D：新增生产 API 做 provider comparison

例如：

```text
POST /evaluations/provider-comparison
```

优点：

- 更接近产品能力；
- 方便前端或平台调用；
- 适合后续企业 dashboard。

缺点：

- 需要 API schema；
- 需要权限；
- 需要成本控制；
- 需要异步任务；
- 需要持久化 run history；
- 当前版本过早。

结论：后置。

## 4. 核心设计

真实实现为测试级 harness：

```text
EmbeddingProviderBenchmarkComparisonTest
```

它复用：

- `src/test/resources/evaluation/retrieval-corpus-v1.md`
- `src/test/resources/evaluation/retrieval-benchmark-v1.json`
- `DocumentChunker`
- `DocumentEmbeddingService`
- `RetrievalEvaluationService`
- `RetrievalMetricsCalculator`

测试内部定义了 test-only helper / record：

- `BenchmarkDataset`
- `BenchmarkCase`
- `EmbeddingProviderComparisonResult`
- `EmbeddingProviderBenchmarkResult`
- `EmbeddingMetricDelta`
- `FakeCandidateEmbeddingProvider`

### baseline result

baseline result 是 baseline provider 在同一 benchmark 下得到的 evaluation result。

当前 baseline provider：

```text
MockEmbeddingClient
```

baseline result 记录：

- provider；
- model；
- dimension；
- caseCount；
- topKValues；
- summary metrics；
- case metrics。

### candidate result

candidate result 是 candidate provider 在同一 benchmark 下得到的 evaluation result。

当前 candidate provider 是测试内部类：

```text
FakeCandidateEmbeddingProvider
```

metadata：

- provider：`fake-real`
- model：`fake-real-embedding-v1`
- dimension：`64`

它 deterministic，不访问外部网络，不代表真实 OpenAI embedding。

### delta metrics

delta metrics 的定义：

```text
delta = candidate metric - baseline metric
```

当前测试会为每个 benchmark `topKValues` 计算 delta。

### topKValues

`topKValues` 来自 benchmark JSON，而不是测试里临时写死。这样 baseline 和 candidate 在同一组 K 上比较。

### provider / model / dimension metadata

provider / model / dimension 是 comparison 的 correctness boundary。测试会验证 baseline 和 candidate result 中 metadata 非空，并验证 search result 使用对应 provider / model。

## 5. 为什么默认测试不调用真实 API

默认测试不能调用真实 OpenAI-compatible API，原因包括：

- CI 环境通常没有 API key；
- API 调用会收费；
- 外部网络不稳定；
- 服务可能限流；
- 模型服务行为可能变化；
- 测试结果不完全 deterministic；
- API key 有泄露风险；
- 单元测试和默认集成测试不应该依赖第三方服务可用性。

所以默认测试只验证：

- comparison pipeline 能跑通；
- 两个 provider 都能生成 embedding；
- baseline metrics 能计算；
- candidate metrics 能计算；
- delta metrics 能计算；
- provider / model / dimension 不混用；
- 不需要 `OPENAI_API_KEY`；
- 不访问外部网络。

真实 API benchmark 应该作为 manual external 流程。

## 6. 为什么使用 FakeCandidateProvider

`FakeCandidateEmbeddingProvider` 不是为了模拟真实 OpenAI embedding 语义效果，而是为了验证 provider comparison pipeline。

它的价值：

- deterministic；
- 无外部依赖；
- provider metadata 与 mock 不同；
- model metadata 与 mock 不同；
- dimension 与 mock 不同；
- 可验证不同 embedding space 的隔离；
- 可用于默认 CI；
- 可验证 comparison result 和 delta metrics 结构。

必须强调：

- `FakeCandidateEmbeddingProvider` 不代表 OpenAI；
- `FakeCandidateEmbeddingProvider` 指标不代表真实 embedding 质量；
- 它只验证 comparison harness。

这是一种测试替身，而不是效果替身。

## 7. 为什么不强制断言 candidate 优于 baseline

不强制断言 candidate 优于 baseline，原因是：

- seed corpus 很小；
- query 比较直接；
- benchmark 不是企业真实数据集；
- chunking 策略会影响结果；
- topK 会影响结果；
- 不同 embedding model 适合的任务不同；
- `FakeCandidateEmbeddingProvider` 本来就不代表真实语义模型；
- 即使是真实 API，也需要多数据集、多 query、多场景验证。

正确断言应该是：

- baseline metrics 非空；
- candidate metrics 非空；
- delta metrics 非空；
- 每个 topK 都有 comparison；
- provider / model / dimension metadata 正确；
- search 不混用不同 embedding space；
- candidate 不一定优于 baseline。

如果在测试里写：

```text
candidateRecallAtK > baselineRecallAtK
```

那不是工程严谨，而是在把实验结论硬编码进测试。

## 8. Metrics 解释

V2.5.1 复用 `RetrievalEvaluationService` 和 `RetrievalMetricsCalculator`，不重新实现 metrics。

### Recall@K（召回率@K）

衡量 expected relevant chunks 中有多少出现在 TopK 结果里。

它回答：

```text
应该找回的证据，有多少被找回来了？
```

### Precision@K（精确率@K）

衡量 TopK 结果里 relevant chunks 的比例。

它回答：

```text
返回的 TopK 里，有多少是真正相关的？
```

### HitRate@K（命中率@K）

衡量 TopK 里是否至少命中一个 relevant chunk。

它回答：

```text
系统有没有至少给 LLM 提供一条有效证据？
```

### MRR / RR@K（平均倒数排名 / 倒数排名@K）

衡量第一个 relevant chunk 出现得有多靠前。

它回答：

```text
第一条正确证据排得靠不靠前？
```

### NDCG@K（归一化折损累计增益@K）

衡量排序质量，越靠前命中 relevant chunk 越好。

它回答：

```text
多个相关证据是否整体排在更好的位置？
```

### ContextPrecision@K（上下文精确率@K）

衡量实际返回 context 中 relevant chunk 的比例。

它回答：

```text
塞给 LLM 的上下文里，有多少是真正有用的？
```

V2.5.1 的 comparison harness 只负责组织不同 provider 的 run 和计算 delta。

## 9. Delta Metrics 设计

delta 定义：

```text
delta = candidate metric - baseline metric
```

当前 delta 字段包括：

- `recallAtKDelta`
- `precisionAtKDelta`
- `hitRateAtKDelta`
- `mrrDelta`
- `ndcgAtKDelta`
- `contextPrecisionAtKDelta`

解释：

- delta > 0：candidate 在当前 benchmark / K 上更高；
- delta < 0：candidate 在当前 benchmark / K 上更低；
- delta = 0：当前 benchmark / K 下没有差异。

必须强调：

delta 只能说明当前 benchmark 下的相对表现，不能泛化为某个 provider 在所有场景都更强或更弱。

例如：

```text
candidate 的 NDCG@5 更高
```

只能说明：

```text
在当前 seed corpus、当前 query、当前 chunking、当前 scoring 和当前 topK 下，candidate 的排序指标更高。
```

不能直接写：

```text
candidate provider 比 baseline provider 更强。
```

## 10. Search 隔离与 metadata

同一次 search 必须在同一 provider / model / dimension 下比较。

错误例子：

```text
query 用 candidate provider；
chunk embedding 来自 mock provider；
然后混在一起算 cosine similarity。
```

这是错误的，因为不同 provider / model 的 vector space 不同。

V2.5.1 的 comparison 逻辑是：

- baseline run 使用 baseline provider 生成 chunk embeddings；
- baseline query 只搜索 baseline provider / model / dimension 对应的 embeddings；
- candidate run 使用 candidate provider 生成 chunk embeddings；
- candidate query 只搜索 candidate provider / model / dimension 对应的 embeddings；
- comparison 比较的是两个独立 retrieval run 的 evaluation result。

provider / model / dimension metadata 是 correctness boundary，不只是展示字段。

当前测试中还会通过 search result 验证：

- baseline result 的 `embeddingProvider` 等于 baseline provider；
- baseline result 的 `embeddingModel` 等于 baseline model；
- candidate result 的 `embeddingProvider` 等于 candidate provider；
- candidate result 的 `embeddingModel` 等于 candidate model。

## 11. 如何手工调用真实 OpenAI-compatible Embedding API

这一节是 manual external benchmark 说明，不是默认 CI 流程。

### 11.1 前提条件

手工调用真实 OpenAI-compatible embedding API 前，需要满足：

- 有有效 API key；
- API key 不能写死；
- API key 不要提交到 Git；
- API key 通过环境变量传入；
- 真实 API 调用通常会产生费用；
- 需要确认所选 provider 支持 embeddings endpoint；
- 当前项目默认 provider 仍然是 `mock`。

### 11.2 PowerShell 设置环境变量

PowerShell session 级别设置：

```powershell
$env:OPENAI_API_KEY="your_api_key_here"
```

说明：

- 这是当前 PowerShell session 级别设置；
- 关闭终端后会失效；
- 不要把真实 key 写进脚本、代码或 Git。

### 11.3 配置 OpenAI-compatible provider

项目真实配置字段如下：

```properties
app.embedding.provider=openai
app.embedding.openai.base-url=https://api.openai.com/v1
app.embedding.openai.api-key=${OPENAI_API_KEY:}
app.embedding.openai.model=text-embedding-3-small
app.embedding.openai.timeout-ms=10000
app.embedding.openai.dimension=1536
```

说明：

- `base-url` 指向 OpenAI-compatible API 根路径；
- 实际请求路径为 `{baseUrl}/embeddings`；
- `model` 可以替换为其他 embedding model；
- `dimension` 必须与模型实际返回维度一致；
- 如果 dimension mismatch，provider 应抛出明确错误；
- 不要把真实 key 写进 `application.properties`。

### 11.4 手工运行方式

当前仓库没有新增默认运行的 external OpenAI benchmark test。因此建议手工方式是通过配置启动项目后，走现有 document embedding / search / evaluation 链路。

大致流程：

1. 设置 `OPENAI_API_KEY`；
2. 设置 `app.embedding.provider=openai`；
3. 启动 Spring Boot；
4. 上传或准备测试 corpus；
5. 生成 document chunks；
6. 调用 embedding generation；
7. 调用 document search；
8. 调用 retrieval evaluation / benchmark runner；
9. 查看 provider / model / dimension metadata；
10. 对比 metrics。

这一步不进入默认 CI，也不应该影响 `.\mvnw.cmd test`。

### 11.5 手工调用 curl 示例

下面是通用 OpenAI-compatible embeddings API 示例，用于理解 API 形状，不是项目默认测试：

```bash
curl https://api.openai.com/v1/embeddings \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "text-embedding-3-small",
    "input": ["What is retrieval evaluation?"]
  }'
```

说明：

- 项目中实际由 `OpenAiCompatibleEmbeddingProvider` 发起请求；
- 不要把 API key 写进代码；
- Windows PowerShell 下 `curl` 命令格式可能需要调整；
- 真实调用可能产生费用。

### 11.6 真实 API benchmark 结果如何解读

如果 Real provider 指标更高，只能说明它在当前 benchmark 上更高。

如果 Real provider 指标更低，也不一定说明模型差，可能原因包括：

- corpus 太小；
- query 太直接；
- chunking 不适配；
- benchmark 设计不足；
- topK 不合适；
- domain 与模型不匹配。

真实上线前需要：

- 更大企业语料；
- 更真实 query；
- hard negatives；
- 多轮 benchmark；
- 成本评估；
- 延迟评估；
- 稳定性评估。

不要用一次小 benchmark 直接决定生产上线。

## 14. 产品化 / VIP / 商业化思考

普通用户：

- 平台默认选择成本 / 效果平衡的 embedding provider；

## 17. 面试高强度追问与回答

### Q1：V2.5 已经接入 Real Embedding，为什么还需要 V2.5.1？

V2.5 解决的是 provider 接入问题，也就是系统能从 mock-only 变成可配置 provider 架构。但接入不等于评估。V2.5.1 的价值是把 provider 放进同一套 retrieval benchmark 中比较，让我们可以观察 baseline 和 candidate 在 Recall@K、Precision@K、MRR、NDCG@K 等指标上的差异。它把“能接入”推进到“能评估”。

### Q2：为什么不能只说真实 Embedding 理论上更好？

因为 embedding model 的效果和语料、query、chunking、topK、领域高度相关。真实模型通常比 mock 更有语义能力，但不能在没有 benchmark 的情况下直接宣称它在当前项目一定更好。工程上应该用固定 corpus、固定 query、固定 expected evidence 和固定 metrics 评估，而不是依靠理论判断。

### Q3：为什么默认测试不能调用 OpenAI API？

默认测试应该稳定、可重复、低成本。OpenAI API 调用依赖网络、API key、额度、限流和外部服务状态，还可能产生费用。如果把它放进默认 CI，测试会因为外部因素失败。V2.5.1 默认用 fake candidate provider 验证 comparison harness，把真实 API benchmark 留给 manual external 流程。

### Q4：FakeCandidateProvider 的意义是什么？

它是测试替身，用来验证 provider comparison pipeline。它 deterministic、无外部依赖、provider/model/dimension 与 mock 不同，因此可以验证 baseline 和 candidate 两套 embedding space 是否独立、metrics 是否能分别计算、delta 是否能生成。它不是为了模拟 OpenAI 的真实语义效果。

### Q5：FakeCandidateProvider 能证明真实 embedding 更好吗？

不能。FakeCandidateProvider 只能证明 comparison harness 能跑通，不能证明真实 OpenAI、BGE、Jina 或其他模型更好。真实效果必须用真实 provider、真实语料、真实 query 和更完整 benchmark 评估。

### Q6：为什么不强制断言 candidate 指标高于 baseline？

因为 candidate 是否更好是实验结果，不应该写死在测试里。当前 candidate 是 fake provider，本身不代表真实语义模型；即使换成真实 provider，也可能因为 corpus 小、query 简单、chunking 不适配导致指标不升反降。测试应该验证 comparison pipeline 正确，而不是假设结果一定提升。

### Q7：delta metrics 怎么解释？

delta = candidate metric - baseline metric。delta 大于 0 表示 candidate 在当前 benchmark 和当前 K 上更高；小于 0 表示更低；等于 0 表示没有差异。但 delta 只对当前 benchmark 有意义，不能泛化为某个 provider 在所有场景都更好。

### Q8：Recall@K / Precision@K / NDCG@K / MRR 分别评价什么？

Recall@K 看 expected evidence 找回了多少，偏向“找全”；Precision@K 看 TopK 中相关结果比例，偏向“返回是否干净”；MRR 看第一个相关结果排得多靠前；NDCG@K 看多个相关结果整体排序质量。它们回答的问题不同，所以 provider comparison 不能只看一个指标。

### Q9：为什么要复用 RetrievalEvaluationService，而不是重写 metrics？

因为 V2.4 / V2.4.1 已经实现了 retrieval metrics。如果在 V2.5.1 测试里重新实现一套公式，会产生逻辑分叉，后续指标改动时两边可能不一致。V2.5.1 的职责是组织 provider run 和 delta，不应该复制 metrics 逻辑。

### Q10：为什么要复用 retrieval-corpus-v1.md 和 retrieval-benchmark-v1.json？

provider comparison 需要固定输入。如果每次换 corpus 或 query，就无法判断结果差异来自 provider 还是输入变化。复用 V2.4.2 的 seed corpus 和 benchmark JSON，可以保证 baseline 和 candidate 在同一套 query-evidence benchmark 下比较。

### Q11：expectedEvidenceIds 如何映射到 expectedChunkIds？

benchmark JSON 中写的是稳定的 `expectedEvidenceIds`，因为 chunkId 是运行时数据库 ID，不适合写死。测试会在 corpus 中找到 `[EVIDENCE:id]` marker，经过 chunking 后定位包含 marker 的 chunk，再取该 chunk 的真实 id，构造 `RetrievalEvaluationRequest.expectedChunkIds`。

### Q12：不同 provider 的 embedding 为什么不能混用？

embedding vector 属于特定 provider / model 的向量空间。不同模型即使 dimension 相同，也不代表语义空间兼容。query 用 candidate provider，chunk 用 mock provider，然后计算 cosine similarity，是没有语义意义的。V2.5.1 验证 baseline 和 candidate run 分别只使用自己的 provider/model embedding。

### Q13：provider / model / dimension metadata 在 comparison 中有什么作用？

它们是隔离 embedding space 的边界。provider 表示来源，model 表示具体模型，dimension 表示向量长度。comparison 中必须记录这些信息，否则无法确认 baseline 和 candidate 是否真的在不同 provider 下运行，也无法防止搜索混用不同向量空间。

### Q14：如果 candidate 指标更低，说明什么？

首先说明在当前 benchmark 下 candidate 更低，但不能马上说明模型不好。可能是 benchmark 太小、query 太直接、chunking 不适配、topK 设置不合适、fake provider 本身不代表语义能力，或者真实 provider 与当前领域不匹配。应该看 per-case result，再扩大 benchmark。

### Q15：如果 candidate 指标更高，能不能直接上线？

不能直接上线。指标更高只说明它在当前 benchmark 下表现更好，还需要看成本、延迟、稳定性、数据合规、失败率、限流、监控和更大规模数据集。上线决策要综合质量、成本和风险，而不是只看一次 seed benchmark。

### Q16：企业真实场景怎么构建 embedding benchmark？

企业 benchmark 应该来自真实知识库、真实搜索日志、客服问题、工单、FAQ、SOP 和专家标注 evidence。需要包含 hard negatives、同义改写、多跳问题、歧义问题、长文档问题和高风险合规问题。expected evidence 最好由领域专家标注，而不是开发者临时写几个简单 query。

### Q17：怎么把这个 test-level harness 产品化？

需要新增 dataset 管理、provider config、evaluation run、result persistence、权限、成本控制、异步任务、dashboard 和报告导出。当前 test-level harness 证明核心链路可行，产品化时可以抽象成 benchmark runner service，再引入数据库表和 API。

### Q18：为什么不现在做生产 API？

因为生产 API 会带来 schema、权限、成本、异步任务、历史结果、错误处理和 dashboard 设计。当前阶段先验证 comparison harness，避免过早固化 API。等 benchmark、provider、持久化需求稳定后再做生产 API 更稳。

### Q19：为什么不现在做 Evaluation Persistence？

当前只做测试级 comparison，没有历史 run、dashboard、多用户 dataset 管理和企业报告需求。现在建表容易过早设计。后续如果要支持趋势分析、A/B test 和报告，再设计 `evaluation_run`、`evaluation_result` 等表。

### Q20：为什么不现在做 Vector DB / Rerank？

V2.5.1 只比较 provider。如果同时引入 Vector DB 或 Rerank，指标变化无法归因。Vector DB 会引入 ANN index 和召回近似误差，Rerank 会改变排序。应该先稳定 provider comparison，再分别评估 Vector DB 和 Rerank。

### Q21：如果要比较 OpenAI、BGE、Jina、Voyage，架构怎么扩展？

每个模型或服务都实现 `EmbeddingProvider`，统一暴露 `embed`、`embedBatch`、`provider`、`model`、`dimension`。comparison harness 可以把 providers 列表循环跑同一 benchmark，生成多个 result，再计算 pairwise delta 或排名。关键仍然是 metadata 隔离和统一 benchmark。

### Q22：如何控制 benchmark 成本？

可以限制 case 数、限制 corpus 大小、使用 batch embedding、缓存已生成 embedding、只对变更文档增量生成、对 external benchmark 加手工触发或预算限制。真实 provider benchmark 不应在每次 CI 默认运行，而应作为 manual 或 scheduled job。

### Q23：如何防止 benchmark 结果被小数据集误导？

需要扩大数据集，覆盖不同业务主题、不同难度、hard negatives、多跳 query 和真实用户 query。还要看 per-case result，不只看 summary。benchmark 也要定期更新，避免模型或策略只针对少量 seed case 过拟合。

### Q24：这个版本对面试有什么价值？

它展示了工程实验意识：不是接了真实 provider 就宣称成功，而是先建立同一 benchmark 下的比较框架；不是让 CI 调外部 API，而是用 fake provider 保持稳定；不是只看单一指标，而是复用 Recall、Precision、MRR、NDCG、ContextPrecision。它体现了边界控制、可测试性和评估意识。

### Q25：下一步为什么是 Local Embedding Worker 或 Vector DB？

V2.5.1 已经有 provider comparison harness。下一步可以接 Local Embedding Worker，比较私有模型和外部 provider；也可以进入 Vector DB Selection，比较 exact cosine search 和 ANN search。两者都能复用当前 benchmark 和 metrics。

### Q26：真实 OpenAI-compatible API 怎么手工调用？

先设置 `OPENAI_API_KEY`，再配置 `app.embedding.provider=openai`、`base-url`、`model` 和 `dimension`。然后通过现有 document embedding、search、retrieval evaluation 链路运行。它不进入默认 CI，不写死 key，不预设指标一定提升，且真实调用可能产生费用。

### Q27：如果真实 API 返回 dimension 和配置不一致怎么办？

应该失败并给出明确错误。dimension mismatch 是 correctness issue，不是普通 warning。如果把错误维度的向量写入同一 embedding space，后续 search 可能污染结果。当前 `OpenAiCompatibleEmbeddingProvider` 已经会对 dimension mismatch 抛出异常。

### Q28：如果某个 provider 是 OpenAI-compatible，但不是 OpenAI 官方，怎么接？

只要它提供兼容 `/embeddings` endpoint，就可以配置 baseUrl、apiKey、model 和 dimension 接入。仍然要保留 provider/model/dimension metadata，仍然要通过 benchmark 验证效果。API 兼容只代表调用协议兼容，不代表 embedding 质量等价。

## 19. 参考资料与工程依据

这些资料是工程依据，不代表本项目完全照搬这些框架。

1. OpenAI Embeddings Documentation  
   Link: https://developers.openai.com/api/docs/guides/embeddings  
   作用：说明 embeddings API 用于把文本转成向量，可用于 search、clustering、recommendations 等任务。

2. OpenAI text-embedding-3-small Model Documentation  
   Link: https://developers.openai.com/api/docs/models/text-embedding-3-small  
   作用：说明 `text-embedding-3-small` 是 embedding model，支持 `v1/embeddings` endpoint。真实调用涉及外部 API 和成本，需要作为 manual external benchmark 管理。

3. MTEB: Massive Text Embedding Benchmark  
   Link: https://arxiv.org/abs/2210.07316  
   作用：说明 embedding model 需要通过多任务 benchmark 评估，不能主观假设某个模型在所有场景都最好。

4. Spring AI EmbeddingModel Documentation  
   Link: https://docs.spring.io/spring-ai/reference/api/embeddings.html  
   作用：说明 `EmbeddingModel` 作为模型抽象是成熟工程方向，支撑本项目 provider abstraction 设计。

这些资料验证了本项目的工程方向：EmbeddingProvider 接入后必须通过 benchmark 评价；真实 API 调用应作为 manual external benchmark，而不是默认 CI；不同 provider 的结果需要结合任务、语料、成本、延迟和稳定性综合判断。

## 20. 本版本在项目中的位置

总结：

```text
V2.4.x 让项目具备 retrieval evaluation 能力。
V2.5 让项目具备真实 embedding provider 接入能力。
V2.5.1 让不同 embedding provider 可以在同一基准下被比较。
```

完整路线：

```text
Document Upload
-> Adaptive Chunking
-> Mock Embedding
-> Vector Search
-> RAG Answer with Citation
-> Retrieval Evaluation
-> Retrieval Metrics Completion
-> Evaluation Dataset Seed
-> Benchmark Runner / Evidence Mapper
-> Real Embedding Provider
-> Provider Benchmark Comparison
-> Local Embedding Worker / External Real Benchmark
-> Vector DB
-> Rerank
-> Generation Evaluation
```

V2.5.1 不证明 Real Embedding 一定更好。它证明的是：这个项目已经从“能接 provider”进一步走到“能用同一套 benchmark 比较 provider”的阶段。
